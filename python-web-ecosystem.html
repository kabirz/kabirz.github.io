01 Django 全栈框架

Django 是 Python Web 领域历史最悠久、生态最完整的全栈框架。它遵循"batteries included"理念，提供了从 ORM 到 Admin 后台再到安全防护的一站式解决方案。

MTV 架构

Django 使用 MTV（Model-Template-View）架构。Model 负责数据层，Template 负责展示层，View 负责业务逻辑。与 MVC 模式相比，Django 的 View 相当于 Controller，Template 相当于 View。

# models.py — 数据模型
from django.db import models

class BlogPost(models.Model):
    title = models.CharField(max_length=200)
    content = models.TextField()
    created_at = models.DateTimeField(auto_now_add=True)
    author = models.ForeignKey('auth.User', on_delete=models.CASCADE)

# views.py — 业务逻辑
from django.views.generic import ListView
from .models import BlogPost

class BlogListView(ListView):
    model = BlogPost
    template_name = 'blog/list.html'
    context_object_name = 'posts'
    paginate_by = 10

# urls.py — URL 路由
from django.urls import path
from . import views

urlpatterns = [
    path('blog/', views.BlogListView.as_view(), name='blog-list'),
]

ORM 与查询

Django ORM 提供了丰富的查询 API，支持链式调用、懒加载、预加载（select_related/prefetch_related）和聚合查询。

# 基本查询
posts = BlogPost.objects.filter(
    author__username='admin',
    created_at__year=2025
).select_related('author').order_by('-created_at')

# 聚合
from django.db.models import Count, Avg
stats = BlogPost.objects.values('author__username').annotate(
    count=Count('id'),
    avg_content_length=Avg('content__length')
)

# 批量更新
from django.db.models import F
BlogPost.objects.filter(published=False).update(
    status='draft',
    version=F('version') + 1
)

Django Admin

Admin 是 Django 的标志性功能。通过简单的配置即可生成功能完整的数据管理后台。

# admin.py
from django.contrib import admin
from .models import BlogPost

@admin.register(BlogPost)
class BlogPostAdmin(admin.ModelAdmin):
    list_display = ('title', 'author', 'created_at', 'is_published')
    list_filter = ('is_published', 'created_at')
    search_fields = ('title', 'content')
    date_hierarchy = 'created_at'
    actions = ['make_published']

    def make_published(self, request, queryset):
        queryset.update(is_published=True)
    make_published.short_description = "标记为已发布"

Django REST Framework

DRF 是 Django 生态中最流行的 API 框架，提供了 Serializer、ViewSet、Router、认证、权限、限流、版本控制等完整功能集。

from rest_framework import serializers, viewsets, routers
from .models import BlogPost

class BlogPostSerializer(serializers.ModelSerializer):
    class Meta:
        model = BlogPost
        fields = '__all__'

class BlogPostViewSet(viewsets.ModelViewSet):
    queryset = BlogPost.objects.all()
    serializer_class = BlogPostSerializer
    permission_classes = [IsAuthenticatedOrReadOnly]

# urls.py
router = routers.DefaultRouter()
router.register('posts', BlogPostViewSet)
urlpatterns = router.urls

Django Channels

Channels 让 Django 支持 WebSocket、HTTP/2、后台任务等异步能力。它将 Django 从 WSGI 扩展到 ASGI，通过 Channel Layer 实现跨进程通信。

# consumers.py
import json
from channels.generic.websocket import AsyncWebsocketConsumer

class ChatConsumer(AsyncWebsocketConsumer):
    async def connect(self):
        self.room_name = self.scope['url_route']['kwargs']['room_name']
        await self.channel_layer.group_add(self.room_name, self.channel_name)
        await self.accept()

    async def receive(self, text_data):
        data = json.loads(text_data)
        await self.channel_layer.group_send(
            self.room_name,
            {'type': 'chat.message', 'message': data['message']}
        )

    async def chat_message(self, event):
        await self.send(text_data=json.dumps({'message': event['message']}))

02 Flask 微框架

Flask 是 Python 最流行的微框架，核心仅包含路由、请求/响应对象和模板渲染。其他功能通过扩展按需集成。它的设计哲学是"简洁而不简单"。

Werkzeug 与 Jinja2

Flask 建立在 Werkzeug（WSGI 工具包）和 Jinja2（模板引擎）之上。Werkzeug 提供了路由、调试器、中间件等底层能力，Jinja2 提供了灵活的模板语法。

from flask import Flask, render_template, request, jsonify

app = Flask(__name__)

@app.route('/')
def index():
    return render_template('index.html', title='首页')

@app.route('/api/data', methods=['POST'])
def create_data():
    data = request.get_json()
    if not data or 'name' not in data:
        return jsonify({'error': '无效数据'}), 400
    return jsonify({'id': 1, **data}), 201

@app.route('/user/')
def user_detail(user_id):
    return f"用户 ID: {user_id}"

Blueprint 模块化

Blueprint 是 Flask 的组织大型应用的方式。它将相关路由、模板和静态文件组织到独立模块中，然后注册到主应用。

# blog/__init__.py
from flask import Blueprint, render_template

blog_bp = Blueprint('blog', __name__, url_prefix='/blog',
                    template_folder='templates',
                    static_folder='static')

@blog_bp.route('/')
def list_posts():
    return render_template('blog/list.html')

@blog_bp.route('/')
def post_detail(post_id):
    return render_template('blog/detail.html', post_id=post_id)

# app.py 注册
from blog import blog_bp
app.register_blueprint(blog_bp)

Flask 插件生态

Flask 的扩展通过 flask-* 命名约定发布。核心扩展包括 Flask-SQLAlchemy、Flask-Login、Flask-Migrate、Flask-Mail、Flask-Caching 等。

from flask_sqlalchemy import SQLAlchemy
from flask_login import LoginManager, UserMixin, login_user
from flask_migrate import Migrate

db = SQLAlchemy()
login_manager = LoginManager()
migrate = Migrate()

def create_app():
    app = Flask(__name__)
    app.config['SQLALCHEMY_DATABASE_URI'] = 'postgresql://localhost/db'
    app.config['SECRET_KEY'] = 'secret-key'

    db.init_app(app)
    login_manager.init_app(app)
    migrate.init_app(app, db)

    return app

class User(UserMixin, db.Model):
    id = db.Column(db.Integer, primary_key=True)
    username = db.Column(db.String(80), unique=True)
    password_hash = db.Column(db.String(128))

SQLAlchemy 集成模式

Flask-SQLAlchemy 是 Flask 和 SQLAlchemy 之间的桥梁。它提供了自动 session 管理、scoped session、模型基类和便捷的查询代理。

class Post(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    title = db.Column(db.String(200), nullable=False)
    content = db.Column(db.Text)
    comments = db.relationship('Comment', backref='post', lazy='dynamic')

class Comment(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    body = db.Column(db.Text, nullable=False)
    post_id = db.Column(db.Integer, db.ForeignKey('post.id'))

# 在请求上下文中自动管理 session
@app.route('/posts')
def posts():
    posts = Post.query.filter(
        Post.published == True
    ).order_by(Post.created_at.desc()).paginate(
        page=request.args.get('page', 1, type=int),
        per_page=20
    )
    return render_template('posts.html', posts=posts)

03 FastAPI 异步框架

FastAPI 是近年崛起最快的 Python Web 框架。它基于 Starlette（ASGI 框架）和 Pydantic（数据验证），开创性地将类型注解用于 API 定义和文档生成。在性能上，FastAPI 接近 Node.js 和 Go 的水平。

Starlette 基础

Starlette 是轻量级的 ASGI 框架，提供了路由、中间件、WebSocket、后台任务、Server-Sent Events 等能力。FastAPI 在 Starlette 之上增加了 OpenAPI 文档生成和数据验证。

from fastapi import FastAPI, HTTPException, BackgroundTasks
from pydantic import BaseModel

app = FastAPI(title="API 服务", version="1.0.0")

class Item(BaseModel):
    name: str
    price: float
    description: str | None = None

# 路径参数、查询参数、请求体——全自动校验和文档化
@app.post("/items/{item_id}")
async def create_item(item_id: int, item: Item, q: str | None = None):
    if item_id <= 0:
        raise HTTPException(status_code=400, detail="无效 ID")
    return {"item_id": item_id, **item.model_dump(), "query": q}

Pydantic 集成

FastAPI 深度集成 Pydantic v2。请求体、路径参数、查询参数、响应模型都使用 Pydantic 模型定义，自动获得验证、序列化和 OpenAPI Schema。

from pydantic import BaseModel, Field, EmailStr
from datetime import datetime
from typing import Literal

class UserCreate(BaseModel):
    username: str = Field(min_length=3, max_length=50)
    email: EmailStr
    role: Literal['admin', 'user'] = 'user'

class UserResponse(BaseModel):
    id: int
    username: str
    email: str
    created_at: datetime

    model_config = {"from_attributes": True}

@app.post("/users", response_model=UserResponse)
async def create_user(user: UserCreate):
    db_user = await db.create_user(user)  # 异步数据库操作
    return db_user

# 自动生成 OpenAPI Schema
# GET /docs 查看 Swagger UI
# GET /redoc 查看 ReDoc

依赖注入系统

FastAPI 的依赖注入系统是最受赞誉的特性之一。依赖可以是函数、类、生成器（用于资源管理），支持嵌套、可缓存、异步。

from fastapi import Depends, FastAPI, Query
from typing import Annotated

async def get_db():
    async with async_session() as session:
        yield session

async def get_current_user(token: str = Query(...)):
    user = await verify_token(token)
    if not user:
        raise HTTPException(status_code=401)
    return user

# 类型别名简化重复的依赖声明
DbDep = Annotated[AsyncSession, Depends(get_db)]
UserDep = Annotated[User, Depends(get_current_user)]

@app.get("/profile")
async def get_profile(
    db: DbDep,
    current_user: UserDep
):
    result = await db.execute(
        select(Profile).where(Profile.user_id == current_user.id)
    )
    return result.scalar_one_or_none()

# 支持不同依赖作用域
# Depends(use_cache=False) 每次重新调用
# yield fixture 支持清理操作

WebSocket 与 Streaming

基于 ASGI，FastAPI 原生支持 WebSocket、SSE 和 StreamingResponse。

from fastapi import WebSocket, WebSocketDisconnect
from fastapi.responses import StreamingResponse
import asyncio

@app.websocket("/ws/chat/{room_id}")
async def websocket_endpoint(websocket: WebSocket, room_id: str):
    await websocket.accept()
    try:
        while True:
            data = await websocket.receive_text()
            await websocket.send_text(f"Echo: {data}")
    except WebSocketDisconnect:
        print(f"{room_id} 断开连接")

async def generate_numbers():
    for i in range(100):
        yield f"data: {i}\n\n"
        await asyncio.sleep(0.5)

@app.get("/stream/numbers")
async def stream_numbers():
    return StreamingResponse(
        generate_numbers(),
        media_type="text/event-stream"
    )

05 中间件与扩展体系对比

三个框架都有中间件机制，但实现方式和扩展生态有所不同。

Django 中间件

Django 中间件采用进程式洋葱圈模型。每个中间件在请求进入和响应离开时都会执行。中间件按 MIDDLEWARE 设置顺序执行。

# 自定义 Django 中间件
class RequestTimingMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        import time
        start = time.time()
        response = self.get_response(request)
        duration = time.time() - start
        response['X-Request-Duration'] = str(duration)
        print(f"{request.path} 耗时 {duration:.3f}s")
        return response

# settings.py
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    # ... 自定义中间件可以放在任何位置
    'myapp.middleware.RequestTimingMiddleware',
]

Flask 中间件模式

Flask 通过 before_request、after_request、teardown_request 钩子实现中间件功能。全局装饰器和 WSGI 中间件是补充机制。

from flask import Flask, g, request
import time

app = Flask(__name__)

@app.before_request
def before_request():
    g.start = time.time()
    g.db = get_db()  # 建立数据库连接

@app.after_request
def after_request(response):
    duration = time.time() - g.start
    response.headers['X-Duration'] = str(duration)
    return response

@app.teardown_request
def teardown_request(exception=None):
    db = g.pop('db', None)
    if db is not None:
        db.close()  # 关闭数据库连接

# WSGI 级中间件
class ProxyFix:
    def __init__(self, app):
        self.app = app

    def __call__(self, environ, start_response):
        environ['wsgi.url_scheme'] = 'https'
        return self.app(environ, start_response)

app.wsgi_app = ProxyFix(app.wsgi_app)

FastAPI 中间件

FastAPI 基于 Starlette 的 ASGI 中间件体系。支持 @app.middleware 装饰器方式和底层 ASGI 中间件方式。

from fastapi import FastAPI, Request
from starlette.middleware.base import BaseHTTPMiddleware
import time

app = FastAPI()

# 方式 1: 装饰器中间件
@app.middleware("http")
async def add_process_time_header(request: Request, call_next):
    start_time = time.perf_counter()
    response = await call_next(request)
    process_time = time.perf_counter() - start_time
    response.headers["X-Process-Time"] = str(process_time)
    return response

# 方式 2: 类中间件
class LoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        print(f"请求: {request.method} {request.url.path}")
        response = await call_next(request)
        print(f"响应: {response.status_code}")
        return response

app.add_middleware(LoggingMiddleware)

# 方式 3: 三方中间件
from fastapi.middleware.cors import CORSMiddleware
from fastapi.middleware.trustedhost import TrustedHostMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://example.com"],
    allow_methods=["*"],
)

07 ASGI vs WSGI 演进

WSGI（PEP 3333）从 Python 2.5 开始就是 Python Web 应用的标准接口。但它有一个根本限制——同步阻塞模型。ASGI（Asynchronous Server Gateway Interface）打破了这一限制。

WSGI 模式

WSGI 定义了服务器和应用的简单调用接口。应用接收 environ 字典和 start_response 回调，返回可迭代的响应体。每个请求对应一个线程或进程，I/O 操作阻塞线程。

# WSGI 应用示例
def simple_app(environ, start_response):
    status = '200 OK'
    headers = [('Content-Type', 'text/plain')]
    start_response(status, headers)
    return [b"Hello, World!"]

# WSGI 服务器（Gunicorn、uWSGI、Waitress）会：
# 1. 接收 HTTP 请求
# 2. 构建 environ 字典
# 3. 在 worker 线程/进程中调用 app
# 4. 收集响应返回给客户端

ASGI 模式

ASGI 将 WSGI 的单次调用改为异步事件驱动模型。应用是一个异步可调用对象，接收 scope（连接信息）、receive（接收事件）和 send（发送事件）。一个连接可以发送和接收多个事件，这对 WebSocket 和 SSE 至关重要。

# ASGI 应用示例
async def simple_asgi_app(scope, receive, send):
    assert scope['type'] == 'http'
    await send({
        'type': 'http.response.start',
        'status': 200,
        'headers': [(b'content-type', b'text/plain')],
    })
    await send({
        'type': 'http.response.body',
        'body': b"Hello, ASGI!",
    })

# 启动方式
# uvicorn main:simple_asgi_app
# daphne main:simple_asgi_app
# hypercorn main:simple_asgi_app

ASGI 服务器对比

混合部署方案

实际部署中，ASGI 和 WSGI 可以共存。使用 Nginx 作为反向代理，将 HTTP 请求转发到 wsgi 服务器，WebSocket 请求转发到 asgi 服务器。或者使用 Gunicorn 的 Uvicorn worker 实现平滑过渡。

# Gunicorn + Uvicorn worker 混合部署
# pip install gunicorn uvicorn

# 启动命令
# gunicorn -w 4 -k uvicorn.workers.UvicornWorker myapp:app

# Nginx 配置示例
# location / {
#     proxy_pass http://127.0.0.1:8000;
#     proxy_http_version 1.1;
#     proxy_set_header Upgrade $http_upgrade;
#     proxy_set_header Connection "upgrade";
# }
提示:
从 WSGI 到 ASGI 的迁移并不需要重写整个应用。Django 4.2+ 和 Flask 3.0+ 都提供了 ASGI 兼容层。可以先只将 WebSocket 部分迁移到 ASGI，再逐步过渡。

08 选型指南

没有最好的框架，只有最合适的框架。以下是针对不同场景的选型建议。

选择 Django 的场景

Django 适合内容管理系统、电商平台、企业内部系统、社交平台等需要完整功能集的大型项目。如果你的团队有 Django 经验，项目周期长，功能需求明确，Django 能让你快速交付。

选择 Flask 的场景

Flask 适合微服务架构、API 网关、原型验证、小到中型 Web 应用、需要灵活定制的项目。Flask 的学习曲线最平缓，适合快速验证想法。它的扩展生态覆盖了几乎所有常见需求。

选择 FastAPI 的场景

FastAPI 适合 RESTful API 服务、实时应用（WebSocket/SSE）、高并发系统、数据科学 API 模型部署。如果你追求性能且希望自动获得类型安全和文档，FastAPI 是最佳选择。

最终建议:
不要过度追求"最佳"框架。选择团队最熟悉、最符合项目当前需求的框架。随着项目演进，可以在不同模块中混合使用多个框架——FastAPI 提供 API，Flask 提供简单页面，Django 提供管理后台，它们可以很好地共存。
