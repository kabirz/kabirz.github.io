01 概述

Linux 内核提供了丰富的锁定机制来处理并发访问。选择正确的锁定机制对性能和正确性至关重要。本文全景梳理所有锁定机制的关系、适用场景和性能特征。

02 机制分类

Linux 锁定机制全景图：

  ┌─────────────────────────────────────────────────────────┐
  │  自旋锁（Spinlock）— 忙等待，不睡眠                      │
  │  ├── raw_spinlock_t  — 真正自旋（PREEMPT_RT 也不让步）    │
  │  ├── spinlock_t      — 普通自旋（RT 下变 rt_mutex）      │
  │  └── bit_spinlock    — 单比特锁，节省内存                │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │  互斥锁（Mutex）— 可睡眠，有等待队列                      │
  │  ├── struct mutex     — 排他锁（默认优先级继承）          │
  │  ├── struct rt_mutex  — RT 互斥锁（优先级继承）           │
  │  └── struct ww_mutex  — 避免死锁的互斥锁（GPU/DRM）      │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │  读写锁 — 读多写少场景                                   │
  │  ├── rwlock_t         — 自旋读写锁（内核态）             │
  │  ├── rw_semaphore     — 睡眠读写锁（内核态）             │
  │  └── percpu-rwsem     — 每 CPU 读写锁（读路径无锁）      │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │  RCU（Read-Copy-Update）— 读无锁，写时复制                │
  │  ├── rcu_read_lock()  — 读端：零开销                      │
  │  ├── rcu_synchronize  — 写端：等待宽限期                   │
  │  └── SLAB_TYPESAFE_BY_RCU — slab 安全回收                 │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │  原子操作 — 无锁的单变量操作                               │
  │  ├── atomic_t / atomic64_t — 整数原子操作                  │
  │  ├── refcount_t           — 引用计数（安全版）             │
  │  └── cmpxchg              — CAS（比较并交换）             │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │  其他机制                                                │
  │  ├── completion       — 一次性同步通知                    │
  │  ├── wait_queue       — 条件等待                          │
  │  ├── per-CPU          — 每 CPU 变量（无锁）               │
  │  └── Local Lock       — per-CPU + 禁用抢占               │
  └─────────────────────────────────────────────────────────┘

03 选择指南

场景推荐机制原因
短临界区 + 不睡眠spinlock最快，忙等待
长临界区 / 可能睡眠mutex睡眠等待，不浪费 CPU
读多写少rwsem读操作并发
读远多于写RCU读零开销
单变量原子操作atomic_t / cmpxchg无锁
引用计数refcount_t防止 use-after-free
一次性通知completion简单高效
条件等待wait_queue灵活的等待机制
per-CPU 数据per-CPU 变量 + local_lock无锁，无竞争
避免死锁（DRM）ww_mutex加权等待避免死锁

04 性能对比

锁定机制性能特征：

  原子操作/CAS      ████████████████████  ~5ns     无锁
  RCU 读端          ████████████████████  ~0ns     零开销
  per-CPU           ████████████████████  ~5ns     无锁
  spinlock          ██████████████        ~100ns   忙等待
  mutex（无竞争）   ████████████          ~200ns   睡眠
  mutex（有竞争）   ████                  ~10μs    上下文切换
  rwsem（读）       ██████████████        ~100ns   读并发
  rwsem（写）       ████████              ~500ms   排他
  RCU（写端）       ██████                ~10ms    宽限期等待
选择原则
① 能用 per-CPU 就不用锁
② 能用 RCU 就不用 rwsem
③ 能用 spinlock 就不用 mutex
④ 能用 atomic 就不用 spinlock
⑤ 优先考虑数据结构设计而非锁定
