01 概述

fork/exec 是 UNIX/Linux 创建新进程的两大系统调用。fork() 创建当前进程的副本（COW），execve() 用新程序替换当前进程的地址空间。两者组合实现了 shell 中 ./program 的完整过程。

02 fork 流程

fork() 完整流程（kernel/fork.c）：

  pid = fork()
        │
        ↓
  kernel_clone()
        │
        ├── 1. copy_process() — 创建子进程描述符
        │       │
        │       ├── dup_task_struct()
        │       │   复制 task_struct + 内核栈
        │       │   子进程的寄存器上下文 = 父进程
        │       │
        │       ├── copy_mm()  ← 关键！COW
        │       │   不复制页表！
        │       │   子进程 mm_struct 指向父进程的 mm
        │       │   所有页表项标记为只读
        │       │
        │       ├── copy_files()  — 复制文件描述符表
        │       ├── copy_sighand() — 复制信号处理
        │       ├── copy_signal()  — 复制信号信息
        │       ├── copy_namespaces() — 复制/创建命名空间
        │       ├── copy_io()     — 复制 IO 上下文
        │       ├── copy_thread() — 设置子进程内核栈
        │       │   子进程 fork 返回值 = 0
        │       │   父进程 fork 返回值 = 子进程 PID
        │       │
        │       └── alloc_pid() — 分配子进程 PID
        │
        ├── 2. wake_up_new_task() — 唤醒子进程
        │       子进程就绪，等待调度
        │
        └── 3. 返回子进程 PID 给父进程

03 写时复制 (COW)

COW 机制（fork 的核心优化）：

  fork 后（无 COW）：
  父进程页面 → [page 1][page 2][page 3]  ← refcount=2（共享）
  子进程页面 →   ↑       ↑       ↑

  父/子进程都指向同一物理页面，页表标记为 只读

  父进程写入 page 2 → 缺页异常
        │
        ↓
  do_wp_page() — mm/memory.c
        │
        ├── 检查 page refcount
        │   refcount > 1 → 有人共享 → 需要 COW
        │
        ├── 分配新页面
        │   alloc_page()
        │
        ├── 复制旧页面内容到新页面
        │   copy_page(new, old)
        │
        ├── 更新父进程页表 → 指向新页面（可写）
        │
        └── 旧页面 refcount--
            refcount=0 → 释放旧页面

  结果：
  父进程 page 2 → [new copy]  ← 独立副本
  子进程 page 2 → [old page]  ← 不变

  优势：
  - fork 极快（只复制页表，不复制数据）
  - 大多数 fork 后立即 exec → 页表也白复制了
  - 70% 的 fork 后不修改大部分页面 → 大量页面保持共享

04 exec 流程

execve() 完整流程（fs/exec.c）：

  execve("/bin/ls", argv, envp)
        │
        ↓
  do_execveat_common()
        │
        ├── 1. 打开可执行文件
        │       kernel_read_file() → 读取 ELF 头
        │
        ├── 2. 验证文件格式
        │       检查 ELF magic number
        │       检查架构（x86_64 / ARM64）
        │       检查权限（setuid/setgid）
        │
        ├── 3. 加载 ELF 段
        │       load_elf_binary()
        │       ├── .text 段 → mmap(PROT_READ|PROT_EXEC)
        │       ├── .data 段 → mmap(PROT_READ|PROT_WRITE)
        │       ├── .bss 段  → mmap(PROT_READ|PROT_WRITE)
        │       ├── 栈      → mmap(向下增长)
        │       └── 堆      → mmap(向上增长)
        │
        ├── 4. 设置新进程参数
        │       argv（命令行参数）
        │       envp（环境变量）
        │       auxv（ELF 辅助向量）
        │
        ├── 5. 设置新进程入口
        │       设置 RIP = ELF entry point
        │       设置 RSP = 新栈顶
        │
        └── 6. 返回用户态 → 从 entry point 开始执行
                新程序"替换"了旧进程
fork + exec 的经典模式
Shell 执行 ls 命令时：
① fork() → 创建子进程（COW，几乎无开销）
② 子进程调用 execve("/bin/ls", ...) → 替换地址空间
③ 父进程（shell）调用 wait() 等待子进程结束
fork 的 COW 优化使得 exec 时不需要复制任何页面——因为 exec 会完全替换地址空间。

05 clone 与线程

fork vs clone vs pthread：

  fork()    → clone(CLONE_VM|CLONE_FS|CLONE_FILES|SIGHAND|SIGCHLD)
             子进程独立（COW），共享文件描述符

  vfork()   → clone(CLONE_VM|CLONE_VFORK|SIGCHLD)
             子进程共享父进程地址空间（不 COW！）
             父进程阻塞直到子进程 exec 或 exit

  pthread_create() → clone(CLONE_VM|CLONE_FS|CLONE_FILES|
                           CLONE_SIGHAND|CLONE_THREAD|...)
             线程共享地址空间、文件、信号处理
             同一进程中的不同线程

  clone 的灵活性：
  通过 flags 精确控制共享哪些资源
  这是 Linux 实现 POSIX 线程的基础
