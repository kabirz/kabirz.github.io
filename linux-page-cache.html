01 概述

Page Cache（页面缓存）是 Linux 内核的文件内容缓存。当进程读写文件时，数据首先在 Page Cache 中缓存，避免每次都要访问磁盘。它是 Linux 文件 I/O 性能的核心——大部分文件读操作都是 Page Cache 命中，根本不涉及磁盘 I/O。

Page Cache 核心价值
  • 极速读取：重复读取直接从内存获取
  • 延迟写入：write() 只写 Page Cache，异步回写磁盘
  • mmap 支撑：mmap 依赖 Page Cache 做页面映射
  • 共享内存：同一文件的 mmap 被多进程共享
  • sendfile 零拷贝：文件→socket 直接走 Page Cache
  • 动态回收：空闲时自动回收，内存紧张时优先回收

03 核心数据结构

// include/linux/fs.h:470 — 地址空间（每个 inode 一个）
struct address_space {
    struct inode          *host;        // 所属的 inode（文件）
    struct xarray          i_pages;     // ★ 页面 XArray（pgoff → page）
    struct rb_root_cached  i_mmap;      // mmap 区域树
    unsigned long          nrpages;     // 缓存的页面总数
    pgoff_t                writeback_index; // 回写起始偏移
    const struct address_space_operations *a_ops; // 操作函数表
    unsigned long          flags;       // AS_DIRTY / AS_EIO 等
};

Page Cache 查找

  Page Cache 使用 XArray（可扩展稀疏数组）管理：

  address_space.i_pages (XArray)
    │
    ├── key = pgoff（文件内页面偏移）
    │
    ├── value = struct page / folio（物理页面）
    │
    └── 支持按 tag 检索：
          PAGECACHE_TAG_DIRTY     → 脏页面
          PAGECACHE_TAG_WRITEBACK → 正在回写的页面
          PAGECACHE_TAG_TOWRITE   → 待写入的页面

  查找：find_get_page(mapping, offset) → O(log n)
  插入：add_to_page_cache_lru() → O(log n)
  删除：delete_from_page_cache() → O(log n)

04 读路径

generic_file_read_iter() — 通用文件读取路径：

  read(fd, buf, count)
        │
        ↓
  generic_file_read_iter() — mm/filemap.c
        │
        ├── 1. 查找 Page Cache
        │       filemap_get_pages()
        │       → find_get_page() 在 XArray 中查找
        │
        ├── 2. Page Cache 命中（常见路径）
        │       页面已在内存中
        │       copy_page_to_iter() → 复制到用户态 buf
        │       无需磁盘 I/O！
        │
        ├── 3. Page Cache 未命中
        │       page_cache_sync_readahead()
        │       → 预读（readahead）策略
        │       → 从磁盘读取多个连续页面
        │       → 加入 Page Cache
        │       → 复制到用户态
        │
        └── 4. 返回读取的字节数
预读（Readahead）
Linux 内核在 Page Cache 未命中时不是只读一个页面，而是预读一批连续页面（默认 32KB~256KB）。这利用了文件访问的空间局部性——读取 offset 100 附近的页面后，很可能马上要读 offset 101、102。

05 写路径与回写

写入路径（延迟写入）：

  write(fd, buf, count)
        │
        ↓
  generic_perform_write()
        │
        ├── 1. 查找/分配 Page Cache 页面
        │       grab_cache_page_write_begin()
        │
        ├── 2. 将数据复制到 Page Cache
        │       copy_page_from_iter_atomic()
        │       页面标记为 DIRTY（脏页）
        │
        ├── 3. 返回 count
        │       write() 立即返回！
        │       数据还在 Page Cache 中，未写磁盘
        │
        ↓
  ──────────────── 异步回写 ────────────────
        │
        ↓
  回写线程（writeback）
        │
        ├── 触发条件：
        │   - dirty_ratio 超限（脏页比例过高）
        │   - dirty_expire_interval 超时（脏页超过 30 秒）
        │   - 内存不足（kswapd 唤醒）
        │   - 用户调用 sync() / fsync()
        │
        ├── 扫描 address_space.i_pages
        │   找到 PAGECACHE_TAG_DIRTY 标记的页面
        │
        └── 执行回写
            a_ops->writepage() / writepages()
            → 提交到 Block Layer → DMA 写入磁盘

脏页管理

参数默认值说明
vm.dirty_ratio20%脏页占系统内存比例上限，超过则阻塞写入
vm.dirty_background_ratio10%触发后台回写的脏页比例
vm.dirty_expire_centisecs3000脏页最大存活时间（30 秒）
vm.dirty_writeback_centisecs500回写线程唤醒间隔（5 秒）

06 调优

# 查看 Page Cache 使用情况
$ free -h
$ cat /proc/meminfo | grep -E "Cached|Buffers|Dirty|Writeback"

# 调整脏页策略（数据库服务器常用）
$ sysctl -w vm.dirty_ratio=40
$ sysctl -w vm.dirty_background_ratio=5
$ sysctl -w vm.dirty_expire_centisecs=500

# 强制同步写入（数据安全优先）
$ sysctl -w vm.dirty_expire_centisecs=100
# 或在应用层使用 O_SYNC / fdatasync()

# 预读调整
$ blockdev --setra 4096 /dev/sda    # 预读 2MB（4096 扇区）

# 直接 I/O（绕过 Page Cache）
$ open(file, O_DIRECT | O_WRONLY)   # 数据库常用
# 注意：O_DIRECT 需要对齐内存地址（512/4096 字节对齐）
