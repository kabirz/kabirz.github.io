01 概述

容器安全不是单一技术，而是一组纵深防御（Defense in Depth）策略。Docker 容器隔离弱于虚拟机，需要多层加固。

容器与虚拟机对比：

容器虚拟机
共享内核是否
隔离机制namespace + cgrouphypervisor + 硬件辅助
启动开销< 1s10s+ (KVM)，1ms (Firecracker)
逃逸难度中等高（需要 hypervisor 漏洞）
资源密度高低

威胁模型：

  • 容器逃逸：攻击者从容器获得宿主权限
  • 横向移动：攻击者从一容器攻击另一容器
  • 供应链攻击：恶意镜像、被篡改的依赖
  • 权限提升：低权限进程获取高权限
  • 拒绝服务：容器耗尽宿主资源

03 守护进程安全

// 1. /var/run/docker.sock 风险
//    任何能访问此 socket 的用户 = root 权限
//    常见错误：把 socket 暴露给容器

docker run -v /var/run/docker.sock:/var/run/docker.sock ...
//   ⚠️ 等同于把 root 给了容器
//   容器可启动任意其他容器
//   可挂载宿主 / 

// 防御：
//   1. 不要将 socket 给容器
//   2. 如必须，限制能用的命令（用 docker.sock proxy）
//   3. 用 rootless 模式

// 2. TLS 远程访问
//    dockerd -H tcp://0.0.0.0:2376 \
//            --tlsverify \
//            --tlscacert=ca.pem \
//            --tlscert=server-cert.pem \
//            --tlskey=server-key.pem
//    
//    客户端：
//    docker --tlsverify --tlscacert=ca.pem \
//           --tlscert=cert.pem --tlskey=key.pem \
//           -H tcp://host:2376 ...

// 3. Rootless Docker（最安全）
//    普通用户运行 dockerd-rootless.sh
//    dockerd 通过 user namespace 运行
//    容器 root 映射到普通用户
//    即使逃逸，也只是普通用户权限
//
//    限制：
//      不能用 privileged
//      不能绑定 < 1024 端口
//      cgroup v2 才完全支持

// 4. daemon.json 加固
{
  "icc": false,                  // 禁止容器间通信（需 publish）
  "userland-proxy": false,        // 禁用 userland-proxy（性能+安全）
  "no-new-privileges": true,      // 禁止 setuid
  "live-restore": true,           // daemon 重启不影响容器
  "userns-remap": "default",      // 启用 user namespace 重映射
  "seccomp-profile": "...",       // 自定义 seccomp
  "default-shm-size": "64M",      // 限制 /dev/shm
  "disable-legacy-registry": true,
  "metrics-addr": "127.0.0.1:9323" // 内部 metrics
}

04 镜像安全

// 1. 镜像来源
//    - 仅信任官方或可信来源
//    - docker pull docker.io/library/nginx:latest
//      ↑ 官方镜像，有 docker 官方签名
//    - 不要从 random 个人仓库拉镜像

// 2. 镜像签名（Docker Content Trust）
//    启用：export DOCKER_CONTENT_TRUST=1
//    之后所有 pull/push 都验证签名
//    
//    签名：docker trust sign myrepo/myimage:latest
//    密钥：~/.docker/trust/
//    
//    内部：Notary v2 / cosign（更现代）

// 3. 漏洞扫描
//    - docker scout（Docker 2023+ 内置）
//    - trivy（开源）
//    - clair（Harbor 集成）
//    - snyk（商业）
//    - grype（开源）

//    例：
//    docker scout cves myimage
//    trivy image myimage:latest

// 4. 最小化基础镜像
//    避免：ubuntu, debian（多包多漏洞）
//    推荐：
//      - alpine（5MB）
//      - distroless（无 shell，最安全）
//      - scratch（空镜像，自带 binary）

// 5. 多阶段构建（减少攻击面）
//    FROM golang:1.21 AS builder
//    COPY . .
//    RUN go build -o app
//    
//    FROM gcr.io/distroless/static
//    COPY --from=builder /app /app
//    ENTRYPOINT ["/app"]
//    
//    最终镜像不含编译器、shell、源码

// 6. 不在 image 存敏感
//    ❌ COPY .env /app/.env
//    ❌ ENV DB_PASSWORD=secret
//    ✅ 运行时通过 secret 注入

// 7. Image 标签策略
//    避免 :latest（不可重现）
//    用 :v1.2.3（版本固定）
//    或用 @sha256:abc...（digest 固定）

// 8. Build-time secret
//    ❌ ARG password=secret  // 留在 history，可见
//    ✅ --mount=type=secret
//       RUN --mount=type=secret,id=npm,target=/root/.npmrc npm install

05 密钥管理

// 1. Docker Secrets（Swarm 模式）
echo "mysecret" | docker secret create db_password -
docker service create --secret db_password mysql
// 容器内：/run/secrets/db_password

// 2. Compose secrets（v2+）
services:
  web:
    secrets:
      - db_password

secrets:
  db_password:
    file: ./secrets/db.txt
//   容器内 /run/secrets/db_password（tmpfs）

// 3. 外部 secret manager
//    - HashiCorp Vault
//    - AWS Secrets Manager / Parameter Store
//    - Azure Key Vault
//    - GCP Secret Manager
//    
//    集成：通过 init container 拉取 secret
//    或通过 CSI driver

// 4. Build-time secret（BuildKit）
docker build --secret id=npm,src=$HOME/.npmrc .
// Dockerfile:
// RUN --mount=type=secret,id=npm,target=/root/.npmrc \
//     npm install
// 优点：
//   - secret 不出现在 image layer
//   - 不在 build cache
//   - 不在 build history

// 5. SSH forwarding（BuildKit）
docker build --ssh default=$SSH_AUTH_SOCK .
// Dockerfile:
// RUN --mount=type=ssh \
//     git clone git@github.com:org/private-repo.git

06 运行时隔离

// 1. 最小化能力
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE ...
// 仅添加必需能力
// 详见 docker-capabilities 文档

// 2. seccomp profile
//    默认 seccomp profile 禁用 60+ 危险 syscall
//    docker run --security-opt seccomp=/path/to/profile.json
//    
//    自定义 profile（精简）：
{
  "defaultAction": "SCMP_ACT_ERRNO",
  "syscalls": [
    {
      "names": ["accept", "bind", "read", "write", ...],
      "action": "SCMP_ACT_ALLOW"
    }
  ]
}

// 3. AppArmor profile
//    docker run --security-opt apparmor=docker-default
//    或自定义：/etc/apparmor.d/docker-<name>
//    
//    profile 限制：
//      文件访问
//      capabilities
//      network
//      mount

// 4. SELinux label
//    docker run --security-opt label=type:container_t
//    RHEL/CentOS/Fedora 默认启用

// 5. user namespace
//    /etc/docker/daemon.json:
//      {"userns-remap": "default"}
//    
//    容器 root → 宿主 UID 100000+
//    即使逃逸也只是普通用户

// 6. read-only rootfs
//    docker run --read-only ...
//    配合 tmpfs：
//    docker run --read-only --tmpfs /tmp --tmpfs /run ...

// 7. no-new-privileges
//    docker run --security-opt no-new-privileges
//    禁止 setuid / setgid 提升

// 8. cgroup 限制（防 DoS）
docker run \
  --memory=512m \
  --memory-swap=512m \
  --cpus=1 \
  --pids-limit=100 \
  --ulimit nofile=1024 \
  ...

完整加固命令：

docker run -d \
  --name app \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  --security-opt no-new-privileges \
  --security-opt seccomp=seccomp-profile.json \
  --security-opt apparmor=docker-default \
  --read-only \
  --tmpfs /tmp \
  --tmpfs /run \
  --memory=512m \
  --cpus=1 \
  --pids-limit=100 \
  --user 1000:1000 \
  --network=internal \
  myimage:latest

07 网络安全

// 1. 网络分段
//    创建独立网络，限制容器间通信
docker network create --internal db_net   # --internal = 不能访问外网
docker network create app_net

docker run -d --network=db_net db
docker run -d --network=app_net \
            --network=db_net app

// 2. iptables / ufw
//    Docker 会修改 iptables
//    可用 ufw-docker 工具管理
//    
//    限制容器出站：
//    iptables -I DOCKER-USER -d 169.254.169.254 -j DROP  # 阻止访问云元数据

// 3. TLS（容器间）
//    - 服务网格（Istio, Linkerd）
//    - 自动 mTLS

// 4. 网络策略（K8s）
//    NetworkPolicy 资源
//    spec:
//      podSelector:
//        matchLabels:
//          app: db
//      ingress:
//        - from:
//            - podSelector:
//                matchLabels:
//                  app: api

// 5. eBPF 安全（Cilium / Calico）
//    - 基于 identity 而非 IP
//    - L7 policy（HTTP method 限制）
//    - 可观测性（流量监控）

// 6. 端口暴露最小化
//    ❌ docker run -p 0.0.0.0:5432:5432 ...  # 全网暴露
//    ✅ docker run -p 127.0.0.1:5432:5432 ...  # 仅本地
//    ✅ 仅容器间通过 expose

09 CIS Benchmark

CIS (Center for Internet Security) Docker Benchmark 是事实标准的容器安全配置规范：

// 主要检查项（部分）：

// 1. 守护进程
//    - 1.x 确保守护进程以最低权限运行
//    - 确保禁用 legacy registry (v1)
//    - 确保 TLS 启用（远程访问）
//    - 确保默认 ulimit 配置

// 2. daemon.json 配置
//    - userns-remap 启用
//    - live-restore 启用
//    - cgroup parent 设置
//    - 默认 seccomp profile
//    - 默认 ulimit

// 3. 镜像构建
//    - 创建非 root 用户
//    - 设定 proper 权限
//    - 最小化镜像

// 4. 容器运行时
//    - privileged=false
//    - --cap-drop=ALL（白名单）
//    - read-only rootfs
//    - set no-new-privileges
//    - limit memory / cpu / pids
//    - 不挂载 docker.sock

// 5. 镜像注册
//    - content trust
//    - 漏洞扫描
//    - RBAC

// 6. 网络
//    - 不暴露 2375/2376 公网
//    - 网络分段

// 自动化检查工具：
//    - docker-bench-security（Docker 官方）
//    - kube-bench（K8s 版本）
//    - trivy config
//    - Clair / Quay

// 运行：
docker run --rm --net host --pid host --userns host --cap-add audit_control \
  -e DOCKER_CONTENT_TRUST=$DOCKER_CONTENT_TRUST \
  -v /var/lib:/var/lib:ro \
  -v /var/run/docker.sock:/var/run/docker.sock:ro \
  -v /usr/lib/systemd:/usr/lib/systemd:ro \
  -v /etc:/etc:ro \
  --label docker_bench_security \
  docker/docker-bench-security

容器安全检查清单:
1. 是否启用 user namespace（userns-remap）？
2. 容器是否以非 root 用户运行？
3. 是否 cap-drop=ALL + 必要的 cap-add？
4. 是否启用 seccomp / AppArmor / SELinux？
5. 镜像是否有签名验证（content trust）？
6. 镜像是否经过漏洞扫描？
7. secret 是否通过 mount 而非 ENV 注入？
8. 是否限制了 memory / cpu / pids？
9. 是否用最小化基础镜像？
10. 是否避免使用 privileged 模式？
11. 是否启用了 no-new-privileges？
12. docker.sock 是否被妥善保护？
如果以上有 3+ 答"否"，需要重新评估容器安全策略。
