01 概述

自旋锁（Spinlock）是 Linux 内核中最快的同步原语，适用于短临界区和中断上下文。与互斥锁（mutex）不同，自旋锁在等待时不会睡眠，而是忙等待（busy-wait）。

核心特性
  • 忙等待：不睡眠，适用于不可睡眠上下文
  • 低延迟：获取/释放仅需几条原子指令
  • SMP 支持：跨 CPU 互斥
  • 中断安全：可在中断上下文中使用
  • 不可嵌套：持有者不能再次获取同一锁
  • 不可睡眠：临界区内不能调用可能睡眠的函数
使用场景
场景原因
中断处理不能睡眠
调度器极短临界区
网络栈高吞吐量要求
驱动程序寄存器访问保护
定时器定时器回调不可睡眠

03 核心 API

spinlock_t API

// 初始化
static DEFINE_SPINLOCK(my_lock);     // 编译时初始化
spin_lock_init(&my_lock);            // 运行时初始化

// 获取锁（可能睡眠版本）
spin_lock(&my_lock);                 // 获取
spin_trylock(&my_lock);              // 尝试获取（非阻塞）
spin_unlock(&my_lock);               // 释放

// 中断安全版本（保存/恢复中断状态）
unsigned long flags;
spin_lock_irqsave(&my_lock, flags); // 禁用中断 + 获取锁
// ... 临界区 ...
spin_unlock_irqrestore(&my_lock, flags); // 恢复中断 + 释放锁

// 底半部安全版本
spin_lock_bh(&my_lock);             // 禁用软中断 + 获取锁
spin_unlock_bh(&my_lock);           // 恢复软中断 + 释放锁

// 检查锁状态
int locked = spin_is_locked(&my_lock);
int contended = spin_is_contended(&my_lock);

IRQ 变体选择

变体禁用中断禁用软中断场景
spin_lock否否进程上下文，无中断
spin_lock_irq是是已知本地中断关闭
spin_lock_irqsave是是保存/恢复中断状态（最安全）
spin_lock_bh否是禁用软中断（如网络栈）
选择原则：如果临界区会被中断打断，必须使用 spin_lock_irqsave()。如果不确定是否中断会获取同一锁，也使用 spin_lock_irqsave()。只有在明确知道中断不会获取锁时才使用 spin_lock()。

04 RT 内核

在 PREEMPT_RT 内核中，大部分 spinlock_t 被转换为可睡眠的互斥锁（rt_mutex），只有 raw_spinlock_t 才真正自旋。

RT 内核下的锁行为

类型普通内核PREEMPT_RT
raw_spinlock_t自旋自旋（硬中断上下文必须）
spinlock_t自旋可睡眠（rt_mutex）
rwlock_t自旋可睡眠（rt_mutex）
local_lock_t禁用抢占禁用抢占（per-CPU）
RT 内核选择：在 PREEMPT_RT 内核中，只有真正需要在硬中断中自旋的锁才使用 raw_spinlock_t。大多数驱动和子系统可以安全使用 spinlock_t，RT 内核会自动处理。新代码建议优先使用 spinlock_t。

05 per-CPU 锁

local_lock_t

// local_lock：保护 per-CPU 数据的轻量级锁
// 在非 RT 内核中仅禁用抢占（无原子操作）
static DEFINE_LOCAL_LOCK(my_local_lock);

local_lock(&my_local_lock);
// ... 访问 per-CPU 数据 ...
val = this_cpu_read(my_counter);
local_unlock(&my_local_lock);

// 中断安全版本
local_lock_irqsave(&my_local_lock, flags);
local_unlock_irqrestore(&my_local_lock, flags);

06 Lockdep

Lockdep（Lock Dependency Validator）是内核的死锁检测器，在运行时分析锁的获取顺序，检测潜在的死锁。

Lockdep 功能

检测能力
  • 锁顺序违规：ABBA 死锁
  • 自旋锁嵌套：同一 CPU 重复获取
  • 中断上下文违规：在持有不可中断锁时进入中断
  • RCU 违规：在 RCU 临界区内睡眠
配置选项
选项说明
CONFIG_LOCKDEP启用 lockdep
CONFIG_PROVE_LOCKING证明锁的正确性
CONFIG_DEBUG_LOCK_ALLOC调试锁分配
CONFIG_DEBUG_SPINLOCK调试自旋锁
Lockdep 输出：当检测到问题时，内核会打印详细的锁依赖图和获取历史。开发时强烈建议开启 CONFIG_PROVE_LOCKING。

07 使用模式

模式一：保护共享数据

static DEFINE_SPINLOCK(data_lock);
static int shared_data;

// 更新（写者）
spin_lock(&data_lock);
shared_data = new_value;
spin_unlock(&data_lock);

// 读取（读者）
spin_lock(&data_lock);
val = shared_data;
spin_unlock(&data_lock);

模式二：中断安全的锁

// 中断处理函数中获取锁
irqreturn_t my_irq_handler(int irq, void *dev_id)
{
    spin_lock(&my_lock);  // 中断中不需要 irqsave（已在中断中）
    // ... 处理中断 ...
    spin_unlock(&my_lock);
    return IRQ_HANDLED;
}

// 进程上下文中保护同一数据
spin_lock_irqsave(&my_lock, flags); // 必须禁用中断！
// ... 访问共享数据 ...
spin_unlock_irqrestore(&my_lock, flags);

模式三：嵌套锁

// 嵌套锁：必须按固定顺序获取
static DEFINE_SPINLOCK(lock_a);
static DEFINE_SPINLOCK(lock_b);

// 正确：始终先获取 lock_a，再获取 lock_b
spin_lock(&lock_a);
spin_lock_nested(&lock_b, SINGLE_DEPTH_NESTING);
// ... 临界区 ...
spin_unlock(&lock_b);
spin_unlock(&lock_a);

// 错误：如果其他地方先获取 lock_b 再 lock_a → 死锁！
常见错误：
• 忘记在持有 spinlock 时禁用中断 → 中断打断临界区 → 死锁
• 临界区内调用可能睡眠的函数（kmalloc(GFP_KERNEL)、mutex_lock）→ 内核警告
• 同一 CPU 重复获取同一自旋锁 → 自死锁
• 持有 spinlock 时间过长 → 其他 CPU 空转，性能下降
