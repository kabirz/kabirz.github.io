01 概述

Linux 内核崩溃是系统最严重的故障。内核提供了多级错误处理机制：Oops（非致命）→ Panic（致命）→ Kdump（转储分析）。理解这些机制对于内核开发、驱动调试和生产环境问题排查至关重要。

02 崩溃类型

Oops（非致命内核错误）：

  内核尝试执行非法操作（如空指针解引用）
        │
        ├── 打印 Oops 信息（寄存器、调用栈、错误地址）
        │
        ├── 发生在用户上下文 → 发送 SIGSEGV，进程被杀死
        └── 发生在中断上下文 → 可能升级为 Panic

Panic（致命内核崩溃）：

  内核检测到无法恢复的错误
        │
        ├── 停止所有正常处理
        ├── 关闭中断
        ├── 打印 Panic 信息
        └── 根据配置：重启 / 挂起 / 等待 Kdump

Kdump（内核崩溃转储）：

  Panic 触发 → 启动 kexec 引导的"捕获内核"
        │
        ├── 捕获内核读取崩溃内存
        ├── 生成 vmcore 转储文件
        └── 重启系统，分析 vmcore

03 Panic 处理流程

  内核检测到致命错误
        │
        ↓
  panic() — kernel/panic.c
        │
        ├── 1. 设置 panic_cpu（防止重入）
        ├── 2. 关闭本地中断
        ├── 3. 打印 Panic 信息
        │       "Kernel panic - not syncing: ..."
        │       函数名、错误描述
        ├── 4. 打印调用栈（dump_stack）
        ├── 5. 通知所有 CPU 停止
        │       smp_send_stop()
        ├── 6. 检查 panic_on_oops
        │       如果 Oops 触发 → 已在 Panic 中
        ├── 7. 执行 panic_notify（通知 Kdump）
        │       kexec 镜像加载，切换到捕获内核
        ├── 8. 最终动作（由配置决定）
        │   ├── 等待 Kdump 完成
        │   ├── 重启（reboot）
        │   └── 挂起（halt）
        └── 9. 无限循环（如果没有配置重启）

04 Kdump 与转储

Kdump 架构：

  正常运行的内核（production kernel）
  ├── 保留一块内存给捕获内核
  ├── 加载 kexec 引导的捕获内核
  │
  └── Panic 发生 →
      │
      ↓ kexec 切换
  捕获内核（capture kernel）
  ├── 独立运行的精简内核
  ├── 读取崩溃时的内存
  ├── 生成 /proc/vmcore 转储
  ├── 保存到磁盘（/var/crash/）
  └── 重启系统

05 配置与调优

# 查看 Panic 配置
$ cat /proc/sys/kernel/panic
0  # 0=挂起, >0=N秒后重启

$ cat /proc/sys/kernel/panic_on_oops
1  # Oops 后是否触发 Panic

# 配置 Kdump
$ sudo yum install kexec-tools crash   # 安装工具
$ sudo systemctl enable kdump
$ sudo systemctl start kdump

# 配置 Panic 后重启
$ echo 1 > /proc/sys/kernel/panic     # 1秒后重启
$ sysctl -w kernel.panic=10            # 10秒后重启

# 查看历史 Oops/Panic
$ dmesg | grep -i "oops\|panic\|bug\|error"
$ journalctl -k | grep -i "oops\|panic"

# 分析 vmcore
$ crash /usr/lib/debug/lib/modules/$(uname -r)/vmlinux /var/crash/*/vmcore
