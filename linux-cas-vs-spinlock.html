01 概述

CAS（Compare-And-Swap/Exchange）和自旋锁（Spinlock）是内核中两种最基础的同步机制。CAS 是无锁原语，通过原子比较和交换实现；自旋锁是有锁原语，通过忙等待实现互斥。理解它们的区别对于编写高效的并发代码至关重要。

CAS（原子比较并交换）
  • 无锁：不持有任何锁
  • 乐观：先尝试，失败则重试
  • 适用：简单计数器、单变量更新
  • 局限：无法保护多步操作
自旋锁
  • 有锁：持有锁才能访问临界区
  • 悲观：等待直到获取锁
  • 适用：多步操作、复杂临界区
  • 局限：持有时间必须短

02 CAS 原理

CAS（Compare-And-Swap）是一条原子硬件指令，它原子地比较内存地址的值与预期值，如果相等则交换为新值。

硬件实现

x86: lock cmpxchg [mem], new
ARM: ldrex / strex (独占访问)
RISC-V: lr.w / sc.w (链接-条件存储)

Linux CAS API

// 基本 CAS — 返回旧值
int old = atomic_cmpxchg(&counter, expected, new);

// 安全 CAS — 推荐使用
int old = expected;
if (atomic_try_cmpxchg(&counter, &old, new)) {
    // 成功：old 仍为预期值，counter 已变为 new
} else {
    // 失败：old 已更新为当前值
}

// 带内存序的变体
atomic_cmpxchg_relaxed(&counter, old, new);   // 无序
atomic_cmpxchg_acquire(&counter, old, new);   // 获取语义
atomic_cmpxchg_release(&counter, old, new);   // 释放语义

// xchg — 无条件交换（无比较）
int old = xchg(&counter, new);  // 原子交换
xchg_relaxed(&counter, new);    // 无序版本

CAS 的语义

if (*addr == expected)
    *addr = new;     // 原子操作
    return true;     // 成功
else
    expected = *addr; // 返回当前值
    return false;     // 失败（需重试）

03 自旋锁原理

自旋锁通过原子操作获取锁，失败时忙等待直到锁被释放。它保证互斥访问临界区。

锁获取流程

spin_lock(&lock)
↓
尝试获取锁
原子操作（test-and-set）
↓
成功
进入临界区
失败
忙等待（自旋）
↓
spin_unlock(&lock)
释放锁，唤醒等待者

自旋锁的代价

方面代价
获取原子指令 + 内存屏障
等待忙等待，消耗 CPU 周期
释放原子指令
上下文禁用抢占（可能禁用中断）
公平性无保证（可能饥饿）

05 性能对比

低竞争场景（单 CPU）

// CAS：几乎无开销
atomic_add(1, &counter);  // 一条指令

// Spinlock：有开销
spin_lock(&lock);       // 禁用抢占 + 原子操作
counter++;               // 修改
spin_unlock(&lock);     // 启用抢占

// 结论：CAS 快 2-5 倍

高竞争场景（多 CPU）

// CAS：自旋循环
do {
    old = atomic_read(&counter);
} while (!atomic_try_cmpxchg(&counter, &old, old + 1));
// 所有 CPU 争抢同一变量 → 频繁失败 → 热点

// Spinlock：有序获取
spin_lock(&lock);
counter++;
spin_unlock(&lock);
// 只有一个 CPU 在临界区，其他等待 → 有序

// 结论：高竞争时 spinlock 更可预测

性能总结

竞争级别CASSpinlock推荐
无竞争~10ns~20nsCAS
低竞争~15ns~25nsCAS
中竞争~50ns（重试）~30ns（等待）视情况
高竞争退化严重有序等待Spinlock
多步操作无法保护天然保护Spinlock

06 使用模式

CAS：无锁计数器

// 无锁引用计数
int old;
do {
    old = atomic_read(&refcount);
    if (old == 0) return -1;  // 已释放
} while (!atomic_try_cmpxchg(&refcount, &old, old + 1));

// 释放
if (atomic_dec_and_test(&refcount)) {
    kfree(obj);
}

CAS：无锁链表插入

// 无锁 LIFO 栈
void push(struct node *n, struct node **top) {
    struct node *old;
    do {
        old = READ_ONCE(*top);
        n->next = old;
    } while (!cmpxchg(&(*top), old, n));
}

Spinlock：保护复杂状态

// 保护多步操作
spin_lock(&dev->lock);
// 1. 读取状态
status = dev->status;
// 2. 检查条件
if (status != READY) {
    spin_unlock(&dev->lock);
    return -EBUSY;
}
// 3. 更新多个字段
dev->status = BUSY;
dev->timeout = jiffies + 100;
dev->error = 0;
spin_unlock(&dev->lock);

// CAS 无法原子地保护这三步

混合使用：CAS + Spinlock

// 快速路径用 CAS，慢速路径用 spinlock
int old;
if (atomic_try_cmpxchg(&counter, &old, old + 1)) {
    return;  // 快速路径成功
}
// CAS 失败，竞争激烈，使用 spinlock
spin_lock(&slow_lock);
counter++;
spin_unlock(&slow_lock);

07 选择决策树

需要同步共享数据？
  │
  ├─ 是 → 是单变量更新吗？
  │       ├─ 是 → CAS / atomic_t （最优选择）
  │       └─ 否 → 需要多步原子操作？
  │               ├─ 是 → spinlock_t （保护临界区）
  │               └─ 否 → 临界区可睡眠？
  │                       ├─ 是 → mutex （可睡眠等待）
  │                       └─ 否 → spinlock_t （不可睡眠上下文）
  └─ 否 → 无需同步

快速决策表

问题答案选择
只更新一个变量？是atomic_cmpxchg()
只做简单加减？是atomic_add()
需要保护多步操作？是spinlock_t
在中断上下文中？是spinlock_t + irqsave
临界区可睡眠？是mutex
读多写少？是rwlock_t / RCU
黄金法则：能用 CAS 解决的就用 CAS（最轻量）；需要保护临界区就用自旋锁；临界区长或可睡眠就用 mutex。不要过度设计——先用最简单的方案，性能不够再优化。
