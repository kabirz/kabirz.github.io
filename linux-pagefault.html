01 概述

缺页异常（Page Fault）是 Linux 虚拟内存管理的核心机制。当进程访问的虚拟地址没有有效的物理页面映射时，CPU 触发缺页异常，内核按需分配物理页面、从磁盘读取数据或执行写时复制。它是 mmap、fork COW、swap 换入、mmap 文件映射的基础。

02 缺页类型

类型原因处理方式
Major Fault页面在 swap 或磁盘上从磁盘读取（慢，~ms）
Minor Fault页面在内存但未映射建立映射（快，~μs）
写时复制 (COW)写只读页面分配新页面 + 复制
权限错误访问无权限的页面发送 SIGSEGV

03 处理流程

缺页异常处理核心流程：

  CPU 访问虚拟地址 → PTE 无效 → 缺页异常
        │
        ↓
  do_page_fault() — 架构特定入口
        │
        ├── 1. 获取错误地址和错误码
        │       fault_address = CR2（x86）
        │       error_code = 读/写、用户/内核、指令/数据
        │
        ├── 2. 查找 VMA
        │       find_vma(mm, fault_address)
        │       ├── 找到 → 检查权限
        │       └── 未找到 → SIGSEGV
        │
        ├── 3. 权限检查
        │       VMA 权限 vs 访问类型
        │       ├── 读只读页面（COW）→ do_wp_page()
        │       ├── 写只读页面（COW）→ do_wp_page()
        │       └── 无权限 → SIGSEGV
        │
        └── 4. handle_mm_fault() — mm/memory.c
                │
                ├── 匿名页面 → do_anonymous_page()
                │   分配零页面，建立映射
                │
                ├── 文件映射 → do_fault()
                │   从磁盘/Page Cache 读取
                │
                ├── 写时复制 → do_wp_page()
                │   分配新页面 + 复制
                │
                └── swap → do_swap_page()
                    从 swap 设备读取

04 TLB 与 TLB Shootdown

TLB Shootdown（TLB 失效广播）：

  CPU 0 修改了某页面的 PTE（如 COW 时分配新页面）
        │
        ├── CPU 0 的 TLB 已更新
        │
        └── 但 CPU 1/2/3 的 TLB 仍缓存旧映射！
            │
            ↓
  发送 IPI（处理器间中断）到其他 CPU
        │
        ├── CPU 1 收到 IPI → 刷新 TLB 中相关条目
        ├── CPU 2 收到 IPI → 刷新 TLB 中相关条目
        └── CPU 3 收到 IPI → 刷新 TLB 中相关条目

  代价：IPI 延迟 ~1-10μs × CPU 数量
  这是 COW 和页面迁移的主要开销来源
