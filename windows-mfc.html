01 概述

MFC（Microsoft Foundation Classes，微软基础类库）是微软提供的面向 Win32/Win64 API 的 C++ 类库封装。它将过程式的 Windows SDK 编程（窗口注册、消息循环、WndProc 回调、GDI 句柄）包装成一套以 CObject 为根的类层次，让开发者用 C++ 面向对象的方式构建桌面应用。

核心特性
  • API 封装：句柄（HWND/HDC/HGDIOBJ）被包装成 C++ 对象
  • 消息映射：用宏表替代 WndProc 的 switch/case
  • Doc/View 架构：数据与表现分离的标准脚手架
  • 运行时类型：自研的 CRuntimeClass 早于标准 C++ RTTI
  • 序列化：CArchive 二进制对象持久化
  • 集合类：CString / CArray / CList / CMap 模板

02 历史与定位

MFC 于 1992 年随 Microsoft C/C++ 7.0 推出，是 Windows 桌面开发的事实标准之一，至今仍随 Visual Studio 分发（VS 2022 内置 MFC 14.4x）。虽然新项目已普遍转向 Qt、WinUI/WPF 或 .NET，但 MFC 在工业控制、金融终端、CAD/EDA、企业遗留系统中存量巨大。

时期版本里程碑
1992MFC 1.0首次发布，封装 Win16/Win32
1993MFC 2.x引入 Doc/View 架构
1995MFC 4.0全面转向 Win32，支持控件
1998MFC 6.0VS 6 经典版本，遗留代码主力
2008+MFC 9.0 ~ 14.x加入 Ribbon、BCGP 商业控件整合
2022MFC 14.4xVS 2022，支持 /std:c++20、ARM64
定位：MFC 是薄封装——它并不隐藏 Win32，而是让你以 OO 方式调用同样的 API。理解 MFC 的前提是理解 Win32 消息驱动模型。

03 类层次架构

MFC 所有类（极少数工具类除外）派生自 CObject，它提供序列化、运行时类型、诊断Dump 三大基础能力。其下最重要的分支是 CCmdTarget——所有能接收命令/消息的对象的基类。

// MFC 核心继承链（简化）
CObject                         // 根：序列化 + RTTI + 诊断
├── CCmdTarget                  // 接收消息/命令的基类
│   ├── CWinThread            // 线程抽象（含消息循环）
│   │   └── CWinApp          // 应用程序对象（全局唯一 theApp）
│   ├── CWnd                  // 所有窗口的基类，封装 HWND
│   │   ├── CFrameWnd        // 框架窗口（SDI 主窗口）
│   │   ├── CMDIFrameWnd     // MDI 主框架
│   │   ├── CMDIChildWnd     // MDI 子框架
│   │   ├── CView            // 视图（数据展示）
│   │   ├── CDialog          // 对话框
│   │   └── CButton / CEdit / ...  // 标准控件包装
│   ├── CDocument             // 文档（数据载体）
│   └── CDocTemplate          // 文档模板（粘合 Doc/View/Frame）
├── CDC                         // 设备上下文，封装 HDC
├── CGdiObject                   // GDI 对象基类
│   ├── CPen / CBrush / CFont
│   └── CBitmap / CPalette / CRgn
├── CStringT / CArray / CList / CMap  // 数据/集合（也派生自 CObject）
└── CArchive / CFile             // 持久化
关键约定：MFC 窗口对象是永久对象（C++ CWnd 对象）与临时对象（HWND）的绑定。FromHandle(HWND) 在找不到永久映射时会创建临时包装，在空闲时由框架回收——不要长期保存临时指针。

04 程序启动与消息泵

MFC 程序的入口不是 WinMain，而是全局应用对象 theApp 的构造。链接器把 MFC 提供的 WinMain 接入 CRT 启动流程，它调用 AfxWinMain，进而驱动应用对象的三个生命周期方法。

// app.h — 声明唯一的应用对象
class CMyApp : public CWinApp {
public:
    virtual BOOL InitInstance();   // 初始化：建窗口、显示
    virtual int  ExitInstance();   // 清理
};

// app.cpp — 实例化全局 theApp（每个进程唯一）
CMyApp theApp;

BOOL CMyApp::InitInstance()
{
    CWinApp::InitInstance();              // 基类：注册公共控件等

    CMainFrame* pFrame = new CMainFrame;
    m_pMainWnd = pFrame;                  // 记住主窗口
    pFrame->Create(NULL, "My MFC App");
    pFrame->ShowWindow(m_nCmdShow);
    pFrame->UpdateWindow();
    return TRUE;                          // 返回 TRUE 才进入消息循环
}

启动到退出的完整流程

// MFC 提供的入口（隐藏在库中，无需手写）
int WINAPI WinMain(...) {
    // 1. theApp 全局对象已在 main 之前构造（CRT 初始化阶段）
    CWinThread* pThread = AfxGetThread();
    CWinApp*     pApp    = AfxGetApp();

    // 2. AfxWinInit：挂接到当前实例、命令行
    // 3. InitApplication：进程级初始化（注册文档模板等）
    // 4. InitInstance：本实例初始化（你重写的那个）
    pApp->InitInstance();

    // 5. Run：消息泵 —— GetMessage/TranslateMessage/DispatchMessage
    return pThread->Run();        // 直到 WM_QUIT 才返回
}

// CWinThread::Run 的核心（简化）
int CWinThread::Run()
{
    for (;;) {
        while (!PeekMessage(&msg, ...)) {  // 队列空
            OnIdle(0);                     // 空闲处理：删临时对象
            if (!PeekMessage(...)) break;
        }
        GetMessage(&msg, ...);
        if (!PreTranslateMessage(&msg)) {  // 预翻译（快捷键、对话框）
            TranslateMessage(&msg);
            DispatchMessage(&msg);        // → WndProc → 消息映射
        }
    }
    return ExitInstance();
}
设计要点：InitInstance 返回 FALSE 会立即终止程序（不进入 Run），常用于许可证检查失败时优雅退出。OnIdle 是低优先级后台任务的钩子，MFC 利用它清理临时句柄映射。

05 消息映射机制

Win32 把所有事件（鼠标、键盘、定时器、命令）以消息（MSG）形式投递到 WndProc。MFC 用消息映射宏把每条消息绑定到一个成员函数，替代原始 switch(LOWORD(wParam)) 的巨型回调。

// MyView.h
class CMyView : public CView {
    DECLARE_DYNCREATE(CMyView)      // 动态创建（Doc/View 必需）
protected:
    // 消息处理函数声明（无 afx_msg 也行，约定写法）
    afx_msg void OnLButtonDown(UINT, CPoint);
    afx_msg void OnPaint();
    afx_msg void OnEditCopy();          // 命令（菜单/按钮 ID）
    afx_msg void OnUpdateEditCopy(CCmdUI*);  // 命令 UI 更新
    DECLARE_MESSAGE_MAP()
};

// MyView.cpp —— 消息映射表
BEGIN_MESSAGE_MAP(CMyView, CView)
    ON_WM_LBUTTONDOWN()                    // 窗口消息
    ON_WM_PAINT()
    ON_COMMAND(ID_EDIT_COPY, &CMyView::OnEditCopy)        // 命令
    ON_UPDATE_COMMAND_UI(ID_EDIT_COPY, &CMyView::OnUpdateEditCopy)
    ON_COMMAND_RANGE(ID_TOOL_FIRST, ID_TOOL_LAST, OnTool) // 范围
    ON_MESSAGE(WM_MY_CUSTOM, OnCustom)            // 自定义/注册消息
END_MESSAGE_MAP()

void CMyView::OnLButtonDown(UINT nFlags, CPoint pt) {
    Invalidate();                          // 触发重绘 → OnPaint
    CView::OnLButtonDown(nFlags, pt);     // 链式调用基类
}

    const AFX_MSGMAP_ENTRY* lpEntries;          // 本类映射表

三对宏必须成对：DECLARE_MESSAGE_MAP（头文件）与 BEGIN_MESSAGE_MAP/END_MESSAGE_MAP（源文件）。漏写任意一个会链接失败（unresolved external "GetThisMessageMap"）。

06 文档/视图架构

Doc/View（Document/View）是 MFC 的旗舰架构，强制把应用拆成三层：文档（数据）、视图（展示）、框架窗口（容器），由文档模板把它们粘合。它实现了数据与界面的分离，天然支持同一份数据的多种视图、文件打开/保存、MRU 列表。

07 运行时类型与动态创建

MFC 在标准 C++ 还没有 RTTI 的年代，就自研了一套运行时类型信息系统（CRuntimeClass）。它支撑了 Doc/View 的动态创建、序列化时的类匹配、安全类型转换。

// 三套宏（按能力递增）
DECLARE_DYNAMIC / IMPLEMENT_DYNAMIC           // 仅类型信息
DECLARE_DYNCREATE / IMPLEMENT_DYNCREATE     // + 无参默认构造（动态创建）
DECLARE_SERIAL / IMPLEMENT_SERIAL          // + 序列化支持

// 使用示例
// .h
class CMyDoc : public CDocument {
    DECLARE_DYNCREATE(CMyDoc)   // Doc/View 必须可动态创建
    ...
};
// .cpp
IMPLEMENT_DYNCREATE(CMyDoc, CDocument)

// 运行时查询
CRuntimeClass* pClass = RUNTIME_CLASS(CMyDoc);
if (pObj->IsKindOf(RUNTIME_CLASS(CMyDoc))) {  // 等价于 dynamic_cast 检查
    CMyDoc* pDoc = DYNAMIC_DOWNCAST(CMyDoc, pObj);
}

// 动态创建：知道类名字符串就能 new（序列化恢复对象的核心）
CObject* pObj = pClass->CreateObject();
前置条件：IMPLEMENT_DYNCREATE 的类必须有无参默认构造函数（框架用它创建对象）。DECLARE_SERIAL 的类还要求 Serialize 实现，且 IMPLEMENT_SERIAL 需要第三参数 schema 版本号。

08 序列化

MFC 的 CArchive 提供类型安全的二进制持久化，配合 CObject::Serialize 实现对象图（含指针、嵌套、集合）的存取。框架在“文件→打开/保存”时自动构造 CArchive 并调用文档的 Serialize。

void CMyDoc::Serialize(CArchive& ar)
{
    CDocument::Serialize(ar);          // 基类先做（MFC 内部状态）
    if (ar.IsStoring()) {
        ar << m_name << m_count;        // CString、int 等内置重载
        m_objects.Serialize(ar);    // 集合类自带 Serialize
    } else {
        ar >> m_name >> m_count;
        m_objects.Serialize(ar);    // 反序列化时按类型动态重建
    }
}

// 派生 CObject 的自定义类：用 DECLARE_SERIAL + 重载 << / >>
class CShape : public CObject {
    DECLARE_SERIAL(CShape)
public:
    CShape() = default;             // 无参构造（动态创建必需）
    virtual void Serialize(CArchive& ar);
};
CArchive& operator<<(CArchive& ar, const CShape& s) { ... }
CArchive& operator>>(CArchive& ar, CShape*& p) {
    p = static_cast<CShape*>(ar.ReadObject(RUNTIME_CLASS(CShape)));
    return ar;                        // 写入时记录类名，读取时动态创建
}
Schema 版本：IMPLEMENT_SERIAL(CShape, CObject, 1) 的第三参数是 schema 版本号。结构变更时应递增并处理旧版本数据，框架会校验避免读到不兼容的格式。

09 对话框与 DDX/DDV

CDialog 封装对话框资源（.rc 中的 DIALOG 模板）。MFC 的两大支柱技术让对话框开发极其简洁：DDX（Dialog Data eXchange，数据交换）与 DDV（Dialog Data Validation，数据校验）。

// MyDlg.h
class CMyDlg : public CDialogEx {
    DECLARE_DYNAMIC(CMyDlg)
public:
    enum { IDD = IDD_MY_DIALOG };       // 关联对话框资源 ID
    CString  m_name;                     // 控件数据成员
    int      m_age;
protected:
    virtual void DoDataExchange(CDataExchange* pDX);  // DDX/DDV 核心
    afx_msg void OnBnClickedOk();
    DECLARE_MESSAGE_MAP()
};

// MyDlg.cpp
void CMyDlg::DoDataExchange(CDataExchange* pDX)
{
    CDialogEx::DoDataExchange(pDX);
    DDX_Text(pDX, IDC_NAME, m_name);             // 控件 ⇄ 成员
    DDX_Text(pDX, IDC_AGE,  m_age);
    DDV_MaxChars(pDX, m_name, 32);                // 字符长度校验
    DDV_MinMaxInt(pDX, m_age, 0, 150);            // 范围校验
    DDX_Control(pDX, IDC_LIST, m_list);          // 绑定控件对象
}

void CMyDlg::OnBnClickedOk() {
    UpdateData(TRUE);     // 控件 → 成员（DDV 失败自动弹错并返回）
    if (m_name.IsEmpty()) { AfxMessageBox("姓名不能为空"); return; }
    EndDialog(IDOK);      // 关闭模态对话框
}

10 GDI 绘图

所有绘制都通过设备上下文（DC），CDC 封装 HDC。CGdiObject 派生类（笔、刷、字体、位图）封装 GDI 句柄。绘制只在 WM_PAINT（OnPaint/OnDraw）中进行——框架已用 CPaintDC 框定了无效区域。

void CMyView::OnDraw(CDC* pDC)
{
    // 1. 创建/选入 GDI 对象（栈对象，自动析构释放句柄）
    CPen pen(PS_SOLID, 2, RGB(255, 0, 0));
    CBrush brush(RGB(0, 200, 0));

    // 2. 选入 DC，保存旧对象
    CGdiObject* pOldPen   = pDC->SelectObject(&pen);
    CGdiObject* pOldBrush = pDC->SelectObject(&brush);

    // 3. 绘图
    pDC->Rectangle(10, 10, 200, 120);
    pDC->MoveTo(10, 10);
    pDC->LineTo(200, 120);

    CFont font;
    font.CreatePointFont(140, "Segoe UI");
    auto* pOldFont = pDC->SelectObject(&font);
    pDC->SetBkMode(TRANSPARENT);
    pDC->TextOut(20, 30, "Hello MFC");

    // 4. 恢复旧对象（必须！否则资源泄漏/绘制异常）
    pDC->SelectObject(pOldPen);
    pDC->SelectObject(pOldBrush);
    pDC->SelectObject(pOldFont);
}

// 自定义窗口的非 OnDraw 绘制：
void CMyWnd::OnPaint() {
    CPaintDC dc(this);     // 构造即 BeginPaint，析构即 EndPaint
    dc.FillSolidRect(rc, RGB(255,255,255));
}
黄金法则：选入新 GDI 对象前保存旧对象，绘制结束恢复旧对象再让新对象析构。GDI 对象在仍被 DC 选中时无法真正释放，会触发“资源泄漏+DeleteObject 失败”。

    EnableTaskbarInteraction(FALSE);
    SetRegistryKey(_T("MyCompany"));    // 配置存注册表而非 INI

    CSingleDocTemplate* pT = new CSingleDocTemplate(
        IDR_MAINFRAME, RUNTIME_CLASS(CMyDoc),
        RUNTIME_CLASS(CMainFrame), RUNTIME_CLASS(CMyView));
    AddDocTemplate(pT);

    CCommandLineInfo cmd; ParseCommandLine(cmd);
    if (!ProcessShellCommand(cmd)) return FALSE;
    m_pMainWnd->ShowWindow(m_nCmdShow);
    m_pMainWnd->UpdateWindow();
    return TRUE;                          // 进入消息循环
}

13 现代开发与对比

MFC 仍受支持但不再演进核心架构。新项目通常选择更现代的技术栈，但 MFC 在维护遗留系统、嵌入原生 Win32 能力、对性能/体积敏感的场景中依然常见。

框架语言渲染定位vs MFC
MFCC++Win32/GDI经典桌面—
QtC++自绘跨平台跨平台、现代 API，但非原生控件
WinFormsC#Win32/GDI托管桌面.NET 生态、生产力高
WPFC#/XAMLDirectX富客户端矢量、数据绑定、现代 UI
WinUI 3 / WinAppSDKC++/C#DirectX现代 Windows微软主推的下一代原生
何时仍用 MFC
  • 维护数百万行遗留代码
  • 深度调用 Win32 / COM / 驱动
  • 工业、医疗、金融领域存量系统
  • 需要原生控件外观与零运行时依赖
现代化技巧
  • 启用 /std:c++17/20，混用 STL/智能指针
  • 用 CString ↔ std::wstring 互转
  • 替换裸指针为 std::unique_ptr 管理 GDI 对象
  • VS 向导生成 Ribbon / 现代主题
陷阱：MFC 是单线程公寓（STA）模型——所有 UI 操作必须在主线程。后台线程要更新 UI 必须通过 PostMessage（自定义消息）切回主线程，直接跨线程操作窗口对象会导致断言失败或死锁。
