01 概述

本文以 UDP 数据包为例，介绍数据包如何从应用程序一步步传递到网卡并最终发送出去。整个过程分为五个阶段：Socket 层 → UDP 层 → IP 层 → Netdevice 子系统 → Device Driver。

与收包流程的对应关系
收包路径：NIC → 驱动 → NAPI → IP → UDP → Socket → 应用
发包路径：应用 → Socket → UDP → IP → Netdevice → 驱动 → NIC
发包路径是收包路径的逆过程，但不完全对称。

函数说明
socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP)创建 socket 结构体，初始化 UDP 相关的操作函数
sendto(sock, ...)应用层调用该函数开始发送数据包
inet_sendmsg检查 socket 是否绑定了源端口，未绑定则调用 inet_autobind
inet_autobind调用 socket 的 get_port 获取一个可用端口（UDP 模式）

03 UDP 层

            |
            ↓
     +--------------+
     | udp_sendmsg  |   UDP 发送入口
     +------+-------+
            |
            |  1. 路由查找
            ↓
     +------------------------+
     | ip_route_output_flow   |   根据目的 IP 查找路由（源 IP + 出接口）
     +------+-----------------+
            |
            |  2. 构造 skb
            ↓
     +------------------+
     |   ip_make_skb    |   分配 skb + IP 包头 + 分片（如果需要）
     +------+-----------+
            |
            |  3. 填充 UDP 头
            ↓
     +------------------------+
     | udp_send_skb(skb, fl4) |   填充 UDP 包头 + checksum
     +------+-----------------+
            |
            ↓
       +----------+
       | IP Layer |
       +----------+
函数说明
udp_sendmsgUDP 模块发送入口，控制整个发送流程
ip_route_output_flow根据路由表和目的 IP 查找发送设备和源 IP。如果 socket 已绑定源 IP 但路由不可达，数据包被丢弃，sendto 返回错误
ip_make_skb构造 skb，分配 IP 包头并设置源 IP。调用 __ip_append_data 进行分片。检查 send buffer 是否用光（满了返回 ENOBUFS）
udp_send_skb填充 UDP 包头 + checksum，然后调用 IP 层发送函数
路由查找的关键作用
ip_route_output_flow 同时完成了两件重要的事：
1. 找到数据包应该从哪个网卡发出去
2. 如果 socket 没有绑定源 IP，自动选择一个最合适的源 IP
如果 socket 已绑定源 IP 但从该 IP 对应的网卡无法到达目的地址，数据包会被丢弃。

函数说明
ip_send_skbIP 发送入口
__ip_local_out_sk设置 IP 报文长度和 checksum，调用 NF_INET_LOCAL_OUT 钩子
NF_INET_LOCAL_OUTNetfilter OUTPUT 链，iptables 可在此处理数据包

ip_finish_output2查找下一跳的 ARP 表项（neigh），未找到则创建空结构体
neigh_resolve_output发送 ARP 请求获取 MAC 地址

Netfilter 在发送路径中的位置
LOCAL_OUT → POST_ROUTING
iptables 的 OUTPUT 链对应 LOCAL_OUT；POST_ROUTING 对应 SNAT/MASQUERADE。

06 Device Driver

ndo_start_xmit 绑定到具体网卡驱动。不同网卡驱动处理方式不同，但大致流程：

  • 1. 将 skb 放入网卡自己的发送队列（TX ring queue）
  • 2. 通知网卡发送数据包
  • 3. 网卡发送完成后发送中断给 CPU
  • 4. 收到中断后进行 skb 清理（释放内存）
流控反馈：当网卡发送队列满时，驱动返回 NETDEV_TX_BUSY 给上层。队列有空闲时，驱动通过 NET_TX_SOFTIRQ 通知上层重试发送。

07 关键参数

参数说明
SO_SNDBUFUDP 的发送缓冲区上限。skb 占用内存超过此值 → 返回 ENOBUFS。调大无意义，因为没有真正的 send buffer
txqueuelenqdisc 队列长度。仅部分 qdisc 类型使用（如默认的 pfifo_fast）
HW TX ring网卡硬件发送队列，大小通过 ethtool 配置。满时驱动返回 NETDEV_TX_BUSY
net.core.wmem_max系统级发送缓冲区上限
ENOBUFS 错误
对于 UDP，SO_SNDBUF 不是一个真正的发送缓冲区，只是 skb 内存占用的上限。只要不出现 ENOBUFS 错误，调大 SO_SNDBUF 没有意义。

08 收包 vs 发包对比

收包（RX）                                    发包（TX）

   NIC 网卡                                      Application
     ↓  DMA                                      ↓
   内存（驱动缓冲区）                              socket()
     ↓  IRQ                                      ↓
   CPU → NIC Driver                              inet_sendmsg
     ↓  softIRQ                                  ↓
   ksoftirqd → net_rx_action                     udp_sendmsg
     ↓                                           ↓
   napi_gro_receive                              ip_route + ip_make_skb
     ↓                                           ↓
   __netif_receive_skb_core                      udp_send_skb
     ↓                                           ↓
   AF_PACKET（tcpdump）                           ip_send_skb
     ↓                                           ↓
   ip_rcv → PRE_ROUTING                          LOCAL_OUT → POST_ROUTING
     ↓                                           ↓
   ip_local_deliver                              ip_finish_output
     ↓                                           ↓
   udp_rcv                                       dst_neigh_output（ARP）
     ↓                                           ↓
   socket recv_queue                             dev_queue_xmit
     ↓                                           ↓
   Application recvfrom()                        ndo_start_xmit → NIC
收包关键点
  • NAPI 轮询模式批量处理
  • GRO 合并小包减少协议栈处理
  • RPS 将负载分散到多个 CPU
  • 多级 Netfilter 钩子（4 个）
  • AF_PACKET 在协议栈之前
发包关键点
  • 路由查找决定出接口和源 IP
  • ARP 表查找获取目标 MAC
  • Traffic Control (qdisc) 排队调度
  • NET_TX_SOFTIRQ 重试机制
  • AF_PACKET 在 dev_hard_start_xmit
tcpdump 抓包位置差异
收包：在 __netif_receive_skb_core 中（协议栈之前）
发包：在 dev_hard_start_xmit 中（TC 之后、驱动之前）
这意味着 tcpdump 看到的发包是经过路由和 qdisc 处理后的包。
