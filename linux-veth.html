01 概述

veth（Virtual Ethernet）是 Linux 内核中最常用的虚拟网络设备之一。它的特点是成对出现，一端连接内核协议栈，另一端与配对设备互连，形成虚拟网线。它是 Docker、Kubernetes 等容器网络方案的基石。

核心特性
  • 成对出现：创建时同时产生 veth0 和 veth1
  • 虚拟网线：一端连协议栈，另一端互连
  • 零拷贝：数据直接在两个设备间传递
  • 可加入 bridge：与网桥配合实现复杂网络拓扑
  • 容器网络：Docker/K8s 的核心组件
典型场景
场景说明
Docker 容器容器通过 veth 连接 bridge
Kubernetes PodPod 网络隔离
网络命名空间隔离不同网络环境
测试环境本地网络拓扑模拟

03 工作原理

数据传输流程

协议栈发送数据到 veth0
        │
        ▼
  veth0 xmit()
        │
        ▼
  数据包直接传递到 veth1
  （无需经过外部网络）
        │
        ▼
  veth1 收到数据包
        │
        ▼
  协议栈接收处理

关键行为：

  • veth0 和 veth1 是一对设备，彼此直接连接
  • 数据从一端发送后，直接传递到另一端，不经过任何外部网络
  • 两端都连接到各自的协议栈，可以独立配置 IP
  • 没有数据包丢失，因为是内核内部直接传递

源码核心路径

// drivers/net/veth.c — 发送路径
static netdev_tx_t veth_xmit(struct sk_buff *skb,
                               struct net_device *dev)
{
    struct veth_priv *priv = netdev_priv(dev);
    struct net_device *peer = priv->peer;

    // 检查配对设备是否存在
    if (!priv->peer_added) {
        dev_kfree_skb(skb);
        return NETDEV_TX_OK;
    }

    // 数据包传递到配对设备
    veth_xmit_rcu(skb, dev, peer);

    return NETDEV_TX_OK;
}

04 实验演示

场景一：veth1 无 IP，ping 失败

# 创建 veth 对
$ sudo ip link add veth0 type veth peer name veth1
$ sudo ip addr add 192.168.2.11/24 dev veth0
$ sudo ip link set veth0 up
$ sudo ip link set veth1 up

# veth1 没有 IP，ping 失败
$ ping -c 1 -I veth0 192.168.2.1
From 192.168.2.11 icmp_seq=1 Destination Host Unreachable

原因分析：veth0 发送 ARP 请求，通过 veth1 传递到 veth1 的协议栈。但 veth1 没有 192.168.2.1 这个 IP，协议栈丢弃了 ARP 请求，所以没有 ARP 应答。

场景二：veth1 有 IP，ping 本地成功

# 给 veth1 配置 IP
$ sudo ip addr add 192.168.2.1/24 dev veth1

# 指定走 veth0 ping veth1
$ ping -c 4 192.168.2.1 -I veth0
64 bytes from 192.168.2.1: icmp_seq=1 ttl=64 time=0.032 ms  # 成功

数据包流程：

1. ping 进程构造 ICMP echo request，通过 socket 发给协议栈
                          │
                          ▼
2. 协议栈检查路由表，确定走 veth0
                          │
                          ▼
3. 数据包交给 veth0
                          │
                          ▼
4. veth0 → veth1（虚拟网线传递）
                          │
                          ▼
5. veth1 收到包，交给 veth1 的协议栈
                          │
                          ▼
6. 协议栈发现本地有 192.168.2.1，构造应答包
                          │
                          ▼
7. 应答包走 lo（本地回环），因为目的和源 IP 相同
                          │
                          ▼
8. ping 进程收到应答

场景三：ping 外部地址失败

# ping 网段外的地址
$ ping -c 1 -I veth0 192.168.2.2
From 192.168.2.11 icmp_seq=1 Destination Host Unreachable

# ping 公网地址
$ ping -c 1 -I veth0 baidu.com
From 192.168.2.11 icmp_seq=1 Destination Host Unreachable

原因：数据包到达 veth1 后，veth1 的协议栈发现目的地址不是本地，但 veth1 只连着 veth0，没有连接外部网络的路径。veth1 没有默认路由，所以无法转发数据包。

容器 vs 虚拟机网络

特性Docker（veth）虚拟机（tun/tap）
连接方式veth → bridgetun/tap → bridge
数据路径Container → veth → br0 → Host 协议栈 → NAT → eth0VM → tun/tap → br0 → eth0（不经 Host）
性能中（经过 Host 协议栈）高（零拷贝）
安全容器在内网VM 完全隔离
MAC 地址需要转换VM 独立 MAC
无线兼容好需要 MAC 转换
