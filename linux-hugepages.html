01 概述

Huge Pages（大页）是 Linux 支持的大于 4KB 的内存页面。标准页面为 4KB，大页可以是 2MB（PMD 级）或 1GB（PUD 级）。大页减少了 TLB（Translation Lookaside Buffer）的条目数量，降低 TLB miss 率，显著提升内存密集型应用（如数据库、HPC、KVM）的性能。

标准 4KB 页面 vs 2MB 大页：

  TLB（翻译后备缓冲器）容量固定，比如 64 条目

  4KB 页面：
  64 条目 × 4KB = 256KB 覆盖范围
  1GB 内存需要 262144 条目 → TLB miss 频繁

  2MB 大页：
  64 条目 × 2MB = 128MB 覆盖范围
  1GB 内存只需 512 条目 → TLB miss 极少

  性能提升：数据库查询加速 5-15%

03 TLB 与性能

TLB miss 的代价：

  CPU 访问虚拟地址 → TLB 命中 → 1-2 周期
  CPU 访问虚拟地址 → TLB 未命中 → 需要遍历页表
                                        │
                                        ├── 4 级页表遍历 → 24+ 次内存访问
                                        ├── 每次内存访问 ~100ns
                                        └── 总计 ~400ns（vs 1-2ns TLB 命中）

  大页减少 TLB miss：
  每个大页 TLB 条目覆盖更大范围
  相同条目数 → 覆盖更多内存 → miss 率更低

04 使用流程

静态大页（HugeTLBFS）：

  # 1. 预留大页（启动时或运行时）
  echo 1024 > /proc/sys/vm/nr_hugepages   # 预留 1024 × 2MB = 2GB

  # 2. 挂载 HugeTLBFS
  mount -t hugetlbfs nodev /mnt/huge

  # 3. 应用使用
  mmap(NULL, 2*1024*1024, PROT_READ|PROT_WRITE,
       MAP_SHARED|MAP_HUGETLB, fd, 0);
  # 或使用 libhugetlbfs

透明大页（THP）— 自动管理：

  # 查看 THP 状态
  $ cat /sys/kernel/mm/transparent_hugepage/enabled
  [always] madvise never

  # 内核自动合并/拆分
  # 2MB 连续 4KB 页面 → 自动合并为大页
  # 碎片化时 → 自动拆分为 4KB 页面
  # 无需应用修改

05 调优

transparent_hugepage/defrag碎片整理策略
什么时候不用 THP？
Redis：THP 可能导致 fork() 时 COW 开销增加（2MB 页面的 COW 比 4KB 更贵）。
数据库：MySQL/PostgreSQL 建议 madvise 模式或关闭 THP。
HPC：计算密集型应用通常推荐开启 THP。
