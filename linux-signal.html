01 概述

Signal（信号）是 Linux 内核提供的进程间异步通知机制。当某个事件发生时（如用户按下 Ctrl+C、子进程退出、段错误），内核向目标进程发送一个信号。进程可以注册信号处理函数来响应信号，也可以选择忽略或使用默认行为。Signal 是 Linux 最古老的 IPC 机制，核心实现 kernel/signal.c 共 5050 行。

Signal 核心特性
  • 异步通知：不阻塞，立即处理
  • 不可排队：标准信号（1-31）不排队，合并
  • 实时信号：32-64 可排队，不合并
  • 三种处理：自定义 handler / 忽略 / 默认
  • 特权检查：非特权进程不能向其他进程发信号
典型场景
信号触发默认行为
SIGINTCtrl+C终止进程
SIGKILLkill -9强制终止
SIGSEGV段错误终止+core dump
SIGCHLD子进程退出忽略（需 wait 回收）
SIGTERMkill pid优雅终止

03 核心数据结构

// include/linux/signal_types.h — 信号核心结构

// 信号队列节点
struct sigqueue {
    struct list_head     list;    // 链表节点
    int                  flags;   // SIGQUEUE_PREALLOC 等
    kernel_siginfo_t     info;    // ★ 信号信息（发送者、信号号等）
    struct ucounts       *ucounts;// 用户命名空间计数
};

// 待处理信号集
struct sigpending {
    struct list_head list;  // 排队的 sigqueue 链表
    sigset_t        signal; // ★ 位图：哪些信号待处理
};

// 信号处理动作
struct sigaction {
    __sighandler_t  sa_handler; // 信号处理函数指针
    unsigned long   sa_flags;   // SA_RESTART / SA_SIGINFO 等
    sigset_t        sa_mask;    // 处理信号时屏蔽的信号集
};

04 发送与投递流程

发送信号（kill / raise / 内核触发）：

  用户态：kill(pid, SIGTERM)
    或
  内核：do_siginfo()（如 SIGSEGV、SIGCHLD）
        │
        ↓
  send_signal_locked() — kernel/signal.c
        │
        ├── 1. 权限检查
        │       CAP_KILL → 可发给任意进程
        │       同一用户 → 只能发给同用户的进程
        │       目标进程的 uid/egid 匹配
        │
        ├── 2. 标准信号合并检查
        │       如果是标准信号（1-31）且 pending 位图已有该信号
        │       → 不重复入队（标准信号不排队）
        │
        ├── 3. 分配 sigqueue 节点
        │       从 slab 缓存分配，填充 info（sender pid/signal no）
        │
        ├── 4. 加入目标进程的 pending 队列
        │       list_add_tail(&sq->list, &pending->list)
        │       set_bit(sig, &pending->signal)
        │
        └── 5. 唤醒目标进程
                signal_wake_up_state(t, ...)
                → 设置 TIF_SIGPENDING 标志
                → 如果目标在睡眠：try_to_wake_up()
                → 如果目标在用户态：下次返回内核时处理

投递信号（目标进程被唤醒后）：

  进程从用户态返回内核态（syscall / 中断 / 调度返回）
        │
        ↓
  检查 TIF_SIGPENDING 标志
        │
        ├── 未设置 → 正常返回用户态
        │
        └── 已设置 → do_signal() → get_signal()
                │
                ├── 1. 获取待处理信号
                │       从 shared_pending + pending 位图取最高优先级
                │
                ├── 2. 检查信号是否被 blocked
                │       blocked 位图中该位为 1 → 跳过
                │
                ├── 3. 根据 sa_handler 执行：
                │       SIG_IGN → 忽略
                │       SIG_DFL → 执行默认行为
                │       用户函数 → 构造信号栈帧，跳转到 handler
                │
                └── 4. handler 返回后 → 信号系统调用重启动
                        （如果设置了 SA_RESTART）

05 信号处理

// 注册信号处理函数
struct sigaction sa;
sa.sa_handler = my_handler;
sigemptyset(&sa.sa_mask);
sa.sa_flags = SA_RESTART; // 被信号中断的系统调用自动重启

sigaction(SIGTERM, &sa, NULL);

// 信号处理函数
void my_handler(int sig) {
    // 只能使用 async-signal-safe 函数
    // 不能使用 malloc、printf 等
    // 只能设置全局 volatile sig_atomic_t 标志
}

信号处理的安全限制

信号处理函数中只能调用 async-signal-safe 函数
在信号处理函数中，进程可能处于任意状态（如 malloc 内部锁被持有），调用非安全函数可能导致死锁或数据损坏。

安全的函数：_exit、write、read、close、kill、sigprocmask、sigaddset
不安全的函数：malloc、printf、free、getenv、localtime、任何 mutex 操作

SA_RESTART 的影响

标志行为
无 SA_RESTART被信号中断的系统调用返回 -1, errno=EINTR
有 SA_RESTART内核自动重启被中断的系统调用

06 实时信号

标准信号（1-31）：

  发送 SIGTERM → pending 已有 SIGTERM → 不入队（合并）
  进程只能知道收到过 SIGTERM，不能知道几次

实时信号（32-64）：

  发送 SIGRTMIN+1 → 入队 → 排列到 pending 队列
  再发 SIGRTMIN+1 → 再入队 → 队列中有 2 个
  进程可以精确知道收到几次，按顺序处理

  sigwaitinfo() / sigtimedwait() 可以获取 siginfo_t
  包含发送者 PID、信号值、排队时间等信息

// 发送实时信号（可携带数据）
union sigval val;
val.sival_int = 42;
sigqueue(pid, SIGRTMIN, val);

// 接收实时信号（获取详细信息）
siginfo_t info;
sigwaitinfo(&set, &info);
// info.si_signo = 信号号
// info.si_value.sival_int = 42
// info.si_pid = 发送者 PID
