01 Net/HTTP 包架构

net/http 是 Go 标准库中最重要也最复杂的包之一。它提供了一个完整的 HTTP 客户端和服务端实现，涵盖 HTTP/1.1 和 HTTP/2 协议。整个包的架构围绕几个核心类型展开。

核心类型一览

类型职责关键方法/字段
http.ServerHTTP 服务端，监听端口、管理连接ListenAndServe, Shutdown, Handler
http.Handler请求处理器接口ServeHTTP(ResponseWriter, *Request)
http.Request封装 HTTP 请求Method, URL, Header, Body, Context
http.ResponseWriter构建 HTTP 响应Write, WriteHeader, Header
http.ClientHTTP 客户端，发送请求Do, Get, Post, Transport
http.Transport底层连接池和传输RoundTrip, MaxIdleConns

请求处理流程

// net/http 服务端请求处理的核心流程
//
// 1. server.ListenAndServe()  → net.Listen("tcp", addr)
// 2. server.Serve(l)          → 接受连接，启动 goroutine 处理
// 3. conn.serve()             → 读取请求、创建 http.Request
// 4. server.Handler.ServeHTTP → 路由到具体 handler
// 5. handler 写入 ResponseWriter

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("/hello", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "text/plain")
        w.Write([]byte("Hello, World!"))
    })
    
    server := &http.Server{
        Addr:    ":8080",
        Handler: mux,
    }
    
    log.Fatal(server.ListenAndServe())
}

Server 配置详解

server := &http.Server{
    Addr:         ":8080",
    Handler:      myHandler,
    ReadTimeout:  5 * time.Second,    // 读取请求超时
    WriteTimeout: 10 * time.Second,   // 写入响应超时
    IdleTimeout:  120 * time.Second,   // keep-alive 空闲超时
    
    // 请求体大小限制
    MaxHeaderBytes: 1 << 20,  // 1MB
    
    // 优雅关闭的回调
    ConnContext: func(ctx context.Context, c net.Conn) context.Context {
        return context.WithValue(ctx, "conn", c)
    },
    
    // 自定义错误处理
    ErrorLog: log.New(os.Stderr, "HTTP: ", log.LstdFlags),
}
关键点：http.Server 的 Shutdown 方法提供了优雅关闭能力，它会停止接受新连接，等待现有请求完成后再退出。

02 ServeMux 路由

Go 1.22 增强的 ServeMux

Go 1.22 对 http.ServeMux 进行了重大升级，新增了方法匹配、通配符路由和路径优先级规则。这使得标准库路由在没有第三方库的情况下也能满足大多数场景。

func main() {
    mux := http.NewServeMux()
    
    // Go 1.22 新增: 方法匹配
    mux.HandleFunc("GET /api/users", listUsers)
    mux.HandleFunc("POST /api/users", createUser)
    
    // 通配符参数
    mux.HandleFunc("GET /api/users/{id}", getUser)
    mux.HandleFunc("PUT /api/users/{id}", updateUser)
    mux.HandleFunc("DELETE /api/users/{id}", deleteUser)
    
    // 通配符后缀匹配
    mux.HandleFunc("GET /files/{path...}", serveFile)
    
    log.Fatal(http.ListenAndServe(":8080", mux))
}

func getUser(w http.ResponseWriter, r *http.Request) {
    // 获取通配符参数
    id := r.PathValue("id")
    w.Write([]byte("User ID: " + id))
}

模式匹配优先级

ServeMux 的模式匹配遵循明确的优先级规则：

  1. 精确路径（如 /api/users）优先于通配符路径
  2. 通配符路径（如 /api/users/{id}）优先于后缀通配符
  3. 后缀通配符（如 /files/{path...}）优先级最低
  4. 如果多个模式同等匹配，则方法匹配的（"GET /api/users"）优先于无方法匹配的（"/api/users"）

Handler 注册机制

// 内部实现要点
//
// ServeMux 内部维护一个 pattern → handler 的映射表
// 并按优先级排序。匹配时按优先级遍历，返回第一个匹配项。
//
// patterns 的注册格式:
//   "/path"             - 无方法限制
//   "GET /path"         - 仅 GET
//   "POST /path"        - 仅 POST
//   "PUT /path/{param}" - 带通配符参数
//   "GET /{param...}"   - 后缀通配符

// 注册 handler 的内部流程:
// 1. 解析 pattern 字符串（方法 + 路径模板）
// 2. 提取通配符名称（{name}、{name...}）
// 3. 计算优先级分数
// 4. 插入到排序后的 patterns 列表
// 5. 检查是否有冲突（相同 pattern 重复注册）

DefaultServeMux 与全局注册

func init() {
    // 不推荐生产使用 - 全局状态难以管理
    http.HandleFunc("/health", healthHandler)
}

func main() {
    // 如果 Handler 为 nil，使用 DefaultServeMux
    log.Fatal(http.ListenAndServe(":8080", nil)) // 使用 DefaultServeMux
}

// 推荐做法：始终使用显式的 ServeMux 实例
func main() {
    mux := http.NewServeMux()
    mux.Handle("/api", &apiHandler{})
    log.Fatal(http.ListenAndServe(":8080", mux))
}
注意：Go 1.22 的 ServeMux 变更了某些模式的行为。如果你的代码升级自旧版本，请测试路由匹配是否正确，特别是斜杠结尾和通配符相关模式。

03 中间件模式

HandlerFunc 链

Go 的 HTTP 中间件模式通过嵌套 http.Handler 实现。每个中间件接收一个 Handler，返回一个新的 Handler，在其中添加额外逻辑后调用下一个 Handler。

type Middleware func(http.Handler) http.Handler

// 日志中间件
func Logging() Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            start := time.Now()
            
            // 包装 ResponseWriter 以捕获状态码
            sw := &statusWriter{ResponseWriter: w, status: http.StatusOK}
            
            next.ServeHTTP(sw, r)
            
            log.Printf("%s %s %d %v",
                r.Method, r.URL.Path, sw.status, time.Since(start))
        })
    }
}

// 应用中间件链
func Apply(handler http.Handler, middlewares ...Middleware) http.Handler {
    for i := len(middlewares) - 1; i >= 0; i-- {
        handler = middlewares[i](handler)
    }
    return handler
}

上下文传递

// 认证中间件 - 将用户信息注入 context
type contextKey string

const UserKey contextKey = "user"

func Auth(validator func(string) (string, error)) Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            token := r.Header.Get("Authorization")
            if token == "" {
                http.Error(w, "unauthorized", http.StatusUnauthorized)
                return
            }
            
            userID, err := validator(token)
            if err != nil {
                http.Error(w, "invalid token", http.StatusUnauthorized)
                return
            }
            
            // 将用户信息注入 context
            ctx := context.WithValue(r.Context(), UserKey, userID)
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}

// 获取 context 中的用户信息
func UserFromContext(ctx context.Context) (string, bool) {
    user, ok := ctx.Value(UserKey).(string)
    return user, ok
}

恢复中间件（Recovery）

func Recovery() Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            defer func() {
                if err := recover(); err != nil {
                    log.Printf("panic recovered: %v\n%s", err, debug.Stack())
                    http.Error(w, "Internal Server Error", http.StatusInternalServerError)
                }
            }()
            next.ServeHTTP(w, r)
        })
    }
}

// 限速中间件
func RateLimit(limit int, per time.Duration) Middleware {
    limiter := rate.NewLimiter(rate.Limit(limit)/rate.Limit(per.Seconds()), limit)
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            if !limiter.Allow() {
                http.Error(w, "too many requests", http.StatusTooManyRequests)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}

// 使用链
mux := http.NewServeMux()
mux.HandleFunc("/api", apiHandler)

handler := Apply(mux, Logging(), Recovery(), RateLimit(100, time.Minute))
http.ListenAndServe(":8080", handler)

中间件排序

中间件顺序：Recovery 应放在最外层（最先 apply），Requests 进来时最后执行，这样在 panic 时 Recovery 能捕获。日志中间件通常在外层记录完整请求。认证中间件在业务 handler 之前执行。

04 HTTP/1.1 连接管理

连接复用与 Keep-Alive

HTTP/1.1 默认使用持久连接（Connection: keep-alive）。在 net/http 中，服务端在同一个 TCP 连接上串行处理多个请求，直到客户端关闭连接或超时。

// 服务端 keep-alive 配置
server := &http.Server{
    Addr:        ":8080",
    Handler:     mux,
    IdleTimeout: 60 * time.Second,  // 空闲连接超时
}

// 客户端连接池配置
client := &http.Client{
    Transport: &http.Transport{
        MaxIdleConns:       100,               // 全局最大空闲连接
        MaxIdleConnsPerHost: 10,               // 每主机最大空闲连接
        IdleConnTimeout:     90 * time.Second,  // 空闲连接超时
        DisableKeepAlives:   false,
    },
}

超时配置详解

超时类型覆盖阶段默认值建议
ReadTimeout读取整个请求（含 Body）0（无限制）5-30s
ReadHeaderTimeout读取请求头0（无限制）5-10s
WriteTimeout写入完整响应0（无限制）10-60s
IdleTimeoutkeep-alive 连接的空闲等待0（使用 ReadTimeout）30-120s

连接状态监控

// ConnState 回调 - 监控连接生命周期
server := &http.Server{
    Addr: ":8080",
    ConnState: func(conn net.Conn, state http.ConnState) {
        log.Printf("conn %s -> %s", conn.RemoteAddr(), state)
        // state 取值:
        // StateNew - 新连接
        // StateActive - 正在处理请求
        // StateIdle - 空闲（keep-alive）
        // StateHijacked - 被劫持（WebSocket）
        // StateClosed - 已关闭
    },
}

// 关闭连接处理器
func closeHandler() http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Connection", "close")  // 关闭此连接
        w.Write([]byte("closing"))
    })
}
性能关键：合理设置 IdleTimeout 可以防止连接泄漏。设置 WriteTimeout 可以防止慢客户端占用连接资源。在生产环境中，永远不要使用默认的 0 超时。

05 HTTP/2 实现

HTTP/2 协商机制

Go 的 net/http 通过 golang.org/x/net/http2 包支持 HTTP/2。在 TLS 连接上，通过 ALPN（Application-Layer Protocol Negotiation）协商为 h2 协议。在非加密连接上，可以通过 h2c（HTTP/2 Clear Text）显式启用。

// HTTP/2 over TLS（自动协商）
func startHTTP2Server() {
    server := &http.Server{
        Addr:    ":443",
        Handler: mux,
    }
    
    // 只要使用 TLS，且证书正确配置，HTTP/2 将自动启用
    log.Fatal(server.ListenAndServeTLS("cert.pem", "key.pem"))
}

// HTTP/2 h2c（非加密 - 需手动配置）
import "golang.org/x/net/http2/h2c"

func startH2CServer() {
    handler := h2c.NewHandler(mux, &http2.Server{})
    server := &http.Server{
        Addr:    ":8080",
        Handler: handler,
    }
    log.Fatal(server.ListenAndServe())
}

HTTP/2 流与帧

HTTP/2 将 HTTP 语义映射到二进制帧（Frame）和流（Stream）上：

  • 帧：HTTP/2 的最小通信单元，包括 HEADERS、DATA、SETTINGS、GOAWAY 等类型
  • 流：在连接上独立的双向字节序列，每个请求对应一个流
  • 多路复用：多个流可以在同一个 TCP 连接上交错传输
// HTTP/2 Server Push（已废弃，Go 1.20+ 不推荐）
func pushHandler(w http.ResponseWriter, r *http.Request) {
    if pusher, ok := w.(http.Pusher); ok {
        err := pusher.Push("/style.css", &http.PushOptions{
            Header: http.Header{
                "Accept-Encoding": r.Header["Accept-Encoding"],
            },
        })
        if err != nil {
            log.Printf("push failed: %v", err)
        }
    }
    w.Write([]byte("<html>..."))
}

// HTTP/2 帧类型
// HEADERS     - 请求/响应头
// DATA        - 请求/响应体
// PRIORITY    - 流优先级
// RST_STREAM  - 终止流
// SETTINGS    - 连接参数协商
// PUSH_PROMISE - 服务端推送（已废弃）
// PING        - 健康检查/RTT 测量
// GOAWAY      - 连接关闭通知
// WINDOW_UPDATE - 流量控制
// CONTINUATION - 延续头块

HTTP/2 流量控制

HTTP/2 提供了基于 stream 和连接的流量控制机制。Go 的 net/http 自动管理 WINDOW_UPDATE 帧的发送，开发者无需手动处理。但在高吞吐场景中，了解流量控制窗口大小对性能有影响。

// 配置 HTTP/2 server 参数
server := &http.Server{
    Addr:    ":443",
    Handler: mux,
}

// 通过 http2.Server 配置 HTTP/2 特定参数
err := http2.ConfigureServer(server, &http2.Server{
    MaxConcurrentStreams: 1000,        // 最大并发流数
    MaxReadFrameSize:     1 << 20,    // 最大帧大小 1MB
    IdleTimeout:          60 * time.Second,
})

log.Fatal(server.ListenAndServeTLS("cert.pem", "key.pem"))

06 Client 实现

Client 类型与配置

// http.Client 是发送 HTTP 请求的核心类型
// 它是一个"零配置即可用"的类型，但生产环境需要仔细配置

type Client struct {
    Transport RoundTripper    // 传输层（默认为 http.DefaultTransport）
    CheckRedirect func(req *Request, via []*Request) error  // 重定向策略
    Jar            CookieJar   // Cookie 管理
    Timeout        time.Duration  // 全局超时
}

// 生产级 Client 配置
client := &http.Client{
    Timeout: 30 * time.Second,  // 包含 DNS 解析到响应读取
    Transport: &http.Transport{
        Proxy: http.ProxyFromEnvironment,
        DialContext: (&net.Dialer{
            Timeout:   5 * time.Second,   // 连接超时
            KeepAlive: 30 * time.Second,   // TCP keep-alive
        }).DialContext,
        TLSHandshakeTimeout:   10 * time.Second,
        ResponseHeaderTimeout: 10 * time.Second,
        ExpectContinueTimeout: 1 * time.Second,
        MaxIdleConns:          100,
        MaxIdleConnsPerHost:   20,
        IdleConnTimeout:       90 * time.Second,
    },
}

Transport 连接池

http.Transport 实现了连接池功能。它对每个目标主机维护一个空闲连接列表，在发送请求时复用空闲连接，避免每次请求都建立新的 TCP 连接。

// Transport 的内部结构（简化）
type Transport struct {
    // 连接池: map[connectMethodKey][]*persistConn
    idleConn     map[connectMethodKey][]*persistConn
    idleConnWait map[connectMethodKey]waitReqBucket
    
    // 正在建立的连接
    dialsInProgress map[connectMethodKey]*dialReq
    
    // 最大连接数控制
    MaxConnsPerHost     int
    MaxIdleConns        int
    MaxIdleConnsPerHost int
}

// 连接池统计
stats := client.Transport.(*http.Transport).CloseIdleConnections()
// 可通过该方法关闭所有空闲连接
transport.CloseIdleConnections()

// 自定义 Dial 函数 - 连接池绕过
transport := &http.Transport{
    DialContext: func(ctx context.Context, network, addr string) (net.Conn, error) {
        // 自定义连接逻辑（如 Unix domain socket）
        return net.DialTimeout("unix", "/tmp/app.sock", 5*time.Second)
    },
}

重定向处理

client := &http.Client{
    CheckRedirect: func(req *http.Request, via []*http.Request) error {
        if len(via) >= 5 {  // 最多允许 5 次重定向
            return fmt.Errorf("too many redirects")
        }
        // 记录重定向历史
        log.Printf("redirect %d: %s", len(via), req.URL)
        return nil  // nil 表示允许重定向
    },
}

// 不允许任何重定向
clientNoRedirect := &http.Client{
    CheckRedirect: func(req *http.Request, via []*http.Request) error {
        return http.ErrUseLastResponse
    },
}

并发请求与连接复用

// 共享 Transport 的 Client 可以并发安全地复用连接
transport := &http.Transport{
    MaxConnsPerHost:    10,
    MaxIdleConnsPerHost: 5,
    IdleConnTimeout:    90 * time.Second,
}

client := &http.Client{Transport: transport}

// 并发请求 - 共享连接池
var wg sync.WaitGroup
for i := 0; i < 20; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        resp, err := client.Get("https://example.com")
        if err == nil {
            resp.Body.Close()
        }
    }()
}
wg.Wait()

// 注意：resp.Body 必须被读取并关闭，否则连接无法复用
常见陷阱：每次调用 http.Get() 或 http.Post() 都会创建一个新的 http.Client（使用 DefaultTransport），这意味着不会复用连接。始终共享一个 Client 实例或显式指定 Transport。

07 请求上下文

Context 的超时控制

在 HTTP 服务端，每个请求都关联一个 context。当客户端断开连接或请求超时时，context 会被取消，通知所有依赖该 context 的 goroutine 停止工作。

func slowHandler(w http.ResponseWriter, r *http.Request) {
    ctx := r.Context()
    
    // 选择操作：如果 context 取消则立即返回
    select {
    case <-time.After(5 * time.Second):
        w.Write([]byte("done"))
    case <-ctx.Done():
        // 客户端断开或超时
        log.Printf("request cancelled: %v", ctx.Err())
        return
    }
}

// 客户端发起带超时的请求
ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()

req, _ := http.NewRequestWithContext(ctx, "GET", "https://slow.service", nil)
resp, err := client.Do(req)
if err != nil {
    log.Printf("request failed: %v", err)
}

http.TimeoutHandler

标准库提供了便捷的超时包装器：

// 为 handler 设置超时
handler := http.TimeoutHandler(
    http.HandlerFunc(slowHandler),
    2*time.Second,
    `{"error":"timeout"}`,
)

mux := http.NewServeMux()
mux.Handle("/api", handler)

// TimeoutHandler 内部实现：
// 1. 创建 context.WithTimeout(context.Background(), timeout)
// 2. 用新的 context 替换 r.Context()
// 3. 在单独的 goroutine 中执行 handler
// 4. 超时时返回 503 Service Unavailable

完整的超时链路

一个 HTTP 请求的完整超时链路包含多个层级：

// 层级 1: Server 级别的 ReadTimeout/WriteTimeout
// 层级 2: http.TimeoutHandler 包装
// 层级 3: Handler 内部使用 context.WithTimeout
// 层级 4: 业务代码中的 select + ctx.Done()

// 最佳实践：多层级超时覆盖
func robustHandler(w http.ResponseWriter, r *http.Request) {
    // 在 handler 内部继续加超时
    ctx, cancel := context.WithTimeout(r.Context(), 2*time.Second)
    defer cancel()
    
    // 向下游调用传递 context
    req, _ := http.NewRequestWithContext(ctx, "GET", backendURL, nil)
    resp, err := client.Do(req)
    if err != nil {
        if errors.Is(err, context.DeadlineExceeded) {
            http.Error(w, "upstream timeout", http.StatusGatewayTimeout)
        } else {
            http.Error(w, "upstream error", http.StatusBadGateway)
        }
        return
    }
    defer resp.Body.Close()
    io.Copy(w, resp.Body)
}
context 最佳实践：始终将 r.Context() 传递给下游依赖。不要在请求处理完成后继续使用 context。使用 context cancellation 来优雅释放资源。
