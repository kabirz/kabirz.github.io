01 概述

传统 UNIX 权限模型只有 root（UID=0）和普通用户。root 拥有所有特权——修改系统时间、绑定低端口、加载内核模块、访问所有文件。Linux Capabilities 将 root 的超级权限拆分为 41 个独立能力，允许精细控制进程权限，是 Docker、容器、Android 安全的基础。

02 Linux Capabilities

五种能力集合

┌──────────────────────────────────────────────────────────┐
│                   struct cred 中的五种能力集              │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  cap_permitted  ── 进程允许的全部能力                     │
│       │                                                  │
│       ├── cap_effective  ── 实际生效的能力                │
│       │   （系统调用检查这个集合）                          │
│       │                                                  │
│       ├── cap_inheritable ── exec 时可继承的能力          │
│       │   （文件 capability 允许 + 此集合 → 继承）        │
│       │                                                  │
│       ├── cap_bset ── 能力边界集（上限）                  │
│       │   （fork 的子进程能力不能超过此集）                 │
│       │                                                  │
│       └── cap_ambient ── ambient 能力                     │
│           （非特权 exec 时保留的能力）                     │
│                                                          │
└──────────────────────────────────────────────────────────┘

常用 Capabilities

能力值允许的操作
CAP_NET_RAW13原始套接字（ping、抓包）
CAP_NET_BIND_SERVICE10绑定 1024 以下端口
CAP_NET_ADMIN12网络管理（路由、iptables、网卡配置）
CAP_SYS_ADMIN21大部分系统管理操作（mount、namespace 等）
CAP_SYS_PTRACE19ptrace 调试其他进程

CAP_KILL5向任意进程发送信号
CAP_DAC_OVERRIDE1绕过文件权限检查

03 核心数据结构

// include/linux/cred.h:113 — 进程凭证
struct cred {
    kuid_t           uid;         // 真实 UID
    kuid_t           euid;        // 有效 UID
    kuid_t           suid;        // 保存的 UID
    kgid_t           gid;         // 真实 GID
    kgid_t           egid;        // 有效 GID
    kuid_t           fsuid;       // 文件系统 UID
    kgid_t           fsgid;       // 文件系统 GID
    kernel_cap_t     cap_inheritable; // 可继承能力
    kernel_cap_t     cap_permitted;   // 允许的能力
    kernel_cap_t     cap_effective;   // ★ 有效能力（检查用这个）
    kernel_cap_t     cap_bset;        // 能力边界集
    kernel_cap_t     cap_ambient;     // ambient 能力
    struct user_namespace *user_ns; // 能力的命名空间上下文
};

04 检查流程

系统调用中的能力检查：

  进程调用 bind(fd, addr, 80)  // 绑定端口 80（需要特权）
        │
        ↓
  内核执行权限检查
        │
        ├── 1. 检查 euid == 0？
        │       是 → 检查 cap_effective
        │       否 → 检查文件权限 + ambient
        │
        ├── 2. cap有效能包含 CAP_NET_BIND_SERVICE？
        │       是 → 允许绑定端口 80
        │       否 → 返回 EACCES（权限拒绝）
        │
        └── 3. 执行系统调用

capability_capable() — 核心检查函数

  capable(CAP_NET_ADMIN)
        │
        ↓
  security_capable(current->cred, CAP_NET_ADMIN, CAP_OPT_NONE)
        │
        ├── 检查 cap_effective 对应位
        │   bit 12 = 1 → 允许
        │   bit 12 = 0 → 拒绝
        │
        └── 返回 0（允许）或 -EPERM（拒绝）

05 应用场景

# 查看进程的 capabilities
$ cat /proc/self/status | grep Cap
CapPrm: 0000003fffffffff  # permitted（64位掩码）
CapEff: 0000003fffffffff  # effective
CapBnd: 0000003fffffffff  # bounding set

# 转换为可读格式
$ capsh --decode=0000003fffffffff
0x0000003fffffffff=cap_chown,cap_dac_override,...

# Docker 默认授予的能力
$ docker run --rm alpine cat /proc/self/status | grep Cap
CapPrm: 00000000a80425fb
# 仅 14 个能力（对比完整 41 个）

# 添加额外能力
$ docker run --cap-add=NET_RAW --rm alpine ping 8.8.8.8

# 移除能力（最小权限）
$ docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE nginx

Docker 默认能力

能力说明容器需要？
CAP_CHOWN修改文件所有者是
CAP_DAC_OVERRIDE绕过文件权限是
CAP_FSETID设置 setuid/setgid是
CAP_FOWNER绕过文件所有权检查是
CAP_NET_BIND_SERVICE绑定低端口是
CAP_SYS_CHROOTchroot 系统调用是
CAP_KILL发送信号是
CAP_SETGID修改 GID是
CAP_SETUID修改 UID是
CAP_SYS_ADMIN大部分管理操作否（危险）
CAP_SYS_MODULE加载内核模块否（危险）
CAP_SYS_RAWIO原始 I/O否（危险）
