01 概述

WPF（Windows Presentation Foundation，曾经代号 Avalon）是微软在 .NET Framework 3.0（2006）推出的新一代桌面 UI 框架，彻底取代了基于 User32/GDI 的 WinForms 范式。它以 XAML 声明式标记描述界面、用 DirectX 9（milcore）保留模式渲染矢量图形、并通过依赖属性系统提供原生数据绑定与变更通知，是 .NET 桌面富客户端开发的里程碑。

核心特性
  • 矢量与保留模式：基于 DirectX，控件缩放无锯齿，UI 维护一棵视觉树
  • 声明式 XAML：界面与逻辑分离，工具（Blend/VS 设计器）友好
  • 数据绑定：双向/单向/一次，DataContext 继承，INotifyPropertyChanged
  • 依赖属性：值的优先级系统、变更回调、动画与样式共同驱动
  • 路由事件：冒泡/隧道机制，控件组合复用
  • 外观与逻辑分离：ControlTemplate / DataTemplate 完全重写外观
适用场景
场景选择理由
企业级 LOB 应用MVVM + 强绑定，工程化好
富媒体/动画界面Storyboard、3D、Effect
跨分辨率自适应矢量、Viewbox、流式布局
设计-开发协作XAML 让设计师直接产出
主题/皮肤系统ResourceDictionary 动态切换

WPF vs WinForms

维度WinFormsWPF
渲染底层User32 + GDI/GDI+（即时模式）DirectX（保留模式，milcore）
界面描述C# 代码 / 设计器拖拽XAML 声明式标记
布局绝对坐标为主（Anchor/Dock）流式布局面板（Grid/StackPanel）
数据绑定简陋（BindingSource）一等公民，DataContext 链式继承
外观定制OnPaint 自绘 / OwnerDrawControlTemplate / Style 完全替换
图形位图为主，GDI 绘制矢量 Shape/Geometry，硬件加速
设计哲学：WPF 把界面看作一棵对象树 + 一组绑定关系，而不是一串句柄和绘制命令。这种“声明式 + 数据驱动 + 模板化”的范式深刻影响了后来的 Silverlight、WinRT XAML、WinUI 与跨平台 Xamarin/MAUI。

02 架构与渲染管线

WPF 的核心由托管层（managed）与非托管层（milcore）组成。milcore.dll 用 C++ 编写，封装 DirectX 9，负责合成、纹理、动画时间线；托管程序集在其之上提供 Dispatcher、Visual、输入、布局、控件、数据绑定等高层 API。

// WPF 托管程序集（按抽象层级从下到上）
PresentationCore          // Visual / Dispatcher / Input / Geometry（底层）
PresentationFramework     // Window / Control / Panel / Style / Binding（高层）
System.Xaml               // XAML 解析器、标记扩展、类型映射
WindowsBase               // DependencyProperty / DispatcherObject / 集合

// 非托管层
milcore.dll               // 直接调用 DirectX 9，合成与渲染
wpfgfx / d3d9             // 图形设备封装、纹理、着色器

Dispatcher：UI 线程调度器

所有 UI 对象派生自 DispatcherObject，归属于创建它的 Dispatcher（即 UI 线程）。跨线程访问 UI 会抛 InvalidOperationException。后台线程通过 Dispatcher.Invoke/BeginInvoke 切回。

// 后台线程更新 UI
private async void OnLoad(object s, RoutedEventArgs e)
{
    string data = await Task.Run(() => DownloadHeavy());
    // 此时回到 UI 线程，await 的 SynchronizationContext 自动切回
    ResultText.Text = data;

    // 手动切线程的等价写法
    await Application.Current.Dispatcher.InvokeAsync(() => {
        ResultText.Text = "done";
    });
}

可视化树（Visual Tree）与逻辑树（Logical Tree）

WPF 维护两棵树。逻辑树对应 XAML 中直接书写的元素层次，用于属性继承、资源查找、事件路由；可视化树是逻辑树的渲染展开，把每个控件的 ControlTemplate 内部细节也展开为 Visual 节点，是 milcore 真正绘制的对象图。

// XAML（逻辑树视角）
<Window x:Name="Root">
  <Button Content="OK" />
</Window>

// 逻辑树（简化）：两个节点
Window
  └── Button (Content="OK")

// 可视化树（Button 的默认 ControlTemplate 展开）
Window
  └── Border                          // ChromeBorder
      └── AdornerDecorator
          └── ContentPresenter            // Window 的内容承载
              └── Button
                  └── ButtonChrome           // 默认主题模板
                      └── ContentPresenter
                          └── TextBlock      // "OK" 文本
逻辑树用途
  • 资源查找（StaticResource 向上冒泡）
  • 属性继承（DataContext、FontFamily）
  • 路由事件的真实传播路径
  • LogicalTreeHelper API
可视化树用途
  • milcore 实际渲染的节点
  • 命中测试（HitTest）
  • 变换、剪裁、透明度作用域
  • VisualTreeHelper API
线程模型：WPF 中除 Freezable 派生类（如 Pen、Geometry、BitmapImage）冻结后可跨线程共享外，所有 UI 对象都线程亲和。后台线程构造的 BitmapImage 必须 Freeze() 后才能传给 UI 线程，否则仍是线程亲和。

渲染管线（保留模式）

WinForms 是即时模式：每次 OnPaint 都由应用主动重绘。WPF 是保留模式：你修改 Visual 树上的属性，milcore 在合成线程（Composition Thread）和 UI 线程配合下，以 60fps 持续重新合成画面，应用代码无需处理 WM_PAINT。

// UI 线程：执行托管代码、布局、动画时间线
// 合成线程：从 Visual 树合成最终帧、提交 DirectX 纹理
// 渲染线程：Direct3D 提交 GPU

// 数据流：
应用代码改 DP 属性
   → 触发 PropertyChanged / InvalidateMeasure / InvalidateVisual
   → UI 线程下一拍执行 Measure/Arrange/Render
   → 生成 DrawingContext 指令序列，提交到合成线程
   → 合成线程把多个 Visual 节点合成最终场景
   → DirectX 9 / GPU 渲染到屏幕

03 XAML 标记语言

XAML（eXtensible Application Markup Language）是一种声明式 XML 方言，用于描述对象图。XAML 元素映射到 .NET 类型，属性映射到 .NET 属性或附加属性，每个 XAML 文件都有对应的 code-behind 分部类（partial class）承载逻辑。

基本结构与命名空间

<!-- MainWindow.xaml -->
<Window x:Class="MyApp.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="Demo" Height="300" Width="400">
    <Grid>
        <Button Content="Click Me"
                Click="OnButtonClick"
                Margin="10" />
    </Grid>
</Window>

两个核心命名空间：

  • xmlns（默认）：WPF 控件命名空间，映射到 System.Windows.Controls 等多个 CLR 命名空间的合集。
  • xmlns:x：XAML 语言命名空间，提供 x:Class、x:Key、x:Name、x:Static、x:Type、x:Null 等语言级关键字。

属性元素与内容属性

复杂属性用属性元素语法（Type.Property）；元素的内联子节点由 ContentPropertyAttribute 决定（如 Window.Content、Panel.Children、Border.Child），无需显式写出。

<!-- 属性元素语法：给 Button.Background 设渐变画刷 -->
<Button Content="Gradient">
    <Button.Background>
        <LinearGradientBrush StartPoint="0,0" EndPoint="1,1">
            <GradientStop Color="Yellow" Offset="0" />
            <GradientStop Color="Red" Offset="1" />
        </LinearGradientBrush>
    </Button.Background>
</Button>

<!-- ContentProperty：Grid 的内联 Children 省略 -->
<Grid>
    <TextBlock Text="Hello" />   <!-- 等价于 Grid.Children -->
</Grid>

标记扩展（Markup Extensions）

用大括号 {...} 在属性字符串中触发运行时求值。WPF 内置常用的标记扩展：

扩展用途示例
{Binding}声明数据绑定Text="{Binding Name}"
{StaticResource}编译期查找资源Background="{StaticResource Accent}"
{DynamicResource}运行时查找，随主题变化Background="{DynamicResource {x:Static SystemColors.WindowBrushKey}}"
{x:Static}引用静态属性/字段Text="{x:Static local:Constants.AppName}"
{x:Type}引用类型（用于 Style.TargetType）DataType="{x:Type vm:Item}"
{x:Null}null 值Foreground="{x:Null}"
{RelativeSource}相对绑定源{Binding ..., RelativeSource={RelativeSource FindAncestor, AncestorType=Window}}

code-behind 与 x:Class

// MainWindow.xaml.cs —— partial 类与 XAML 编译产物合并
namespace MyApp;

public partial class MainWindow : Window
{
    public MainWindow() {
        InitializeComponent();   // 由 XAML 编译器生成的 g.i.cs 提供
    }

    private void OnButtonClick(object sender, RoutedEventArgs e) {
        MessageBox.Show("Hi!");
    }
}
BAML：XAML 在编译时被解析并压缩为 BAML（Binary Application Markup Language），嵌入到程序集资源中。运行时由 Application.LoadComponent 反序列化、构造对象图。这就是为什么改 XAML 后可视化设计器能即时反映——它走的是同一条 BAML 加载路径。

04 控件模型

WPF 控件层次的核心是“内容模型”：大多数控件的 Content 或 Items 属性可以接受任意对象，由 DataTemplate 决定如何呈现。这让 WPF 控件比 WinForms“控件 = 外观 + 数据”的耦合松得多。

DispatcherObject
└── Visual
    └── UIElement                  // 布局、输入、路由事件
        └── FrameworkElement      // Width/Height/Style/DataContext/Binding
            ├── Control            // 控件基类（Template/Foreground）
            │   ├── ContentControl     // 单一内容（Button/Label/Window）
            │   │   ├── ButtonBase
            │   │   │   ├── Button / ToggleButton / RepeatButton
            │   │   │   └── CheckBox / RadioButton
            │   │   ├── HeaderedContentControl  // GroupBox / Expander / HeaderedContent
            │   │   └── UserControl              // 自定义组合控件
            │   ├── ItemsControl       // 列表基类（无选择）
            │   │   ├── Selector          // + 选择
            │   │   │   ├── ListBox / ComboBox / ListView
            │   │   │   ├── TabControl
            │   │   │   └── TreeView       // 间接派生
            │   │   ├── Menu / ContextMenu
            │   │   ├── StatusBar
            │   │   └── DataGrid          // 表格

            │   ├── RangeBase          // ProgressBar / Slider / ScrollBar

ContentControl：内容模型

ContentControl 的 Content 属性类型是 object——可以是字符串、UIElement、或任意业务对象。如果是 UIElement，直接渲染；否则通过 ContentTemplate（DataTemplate）转换。

<!-- 内容是字符串 -->
<Button Content="Save" />

<!-- 内容是 UIElement（任意复杂） -->
<Button>
    <StackPanel Orientation="Horizontal">
        <Image Source="save.png" Width="16" />
        <TextBlock Text="Save" Margin="4,0,0,0" />
    </StackPanel>
</Button>

<!-- 内容是业务对象（由 DataTemplate 渲染） -->
<ContentControl Content="{Binding SelectedUser}" />

ItemsControl：列表模型

ItemsControl 的 ItemsSource 接收 IEnumerable，对每个数据项用 ItemTemplate（DataTemplate）生成 UI 容器，由 ItemsPanel 决定容器布局。

<ItemsControl ItemsSource="{Binding Users}">
    <ItemsControl.ItemsPanel>
        <ItemsPanelTemplate>
            <StackPanel Orientation="Vertical" />
        </ItemsPanelTemplate>
    </ItemsControl.ItemsPanel>
    <ItemsControl.ItemTemplate>
        <DataTemplate>
            <Border Padding="8">
                <TextBlock Text="{Binding Name}" />
            </Border>
        </DataTemplate>
    </ItemsControl.ItemTemplate>
</ItemsControl>

ControlTemplate vs DataTemplate

ControlTemplate

定义控件本身的视觉外观。把 Button 变成圆形、把 ProgressBar 变成进度条样式。通过 TemplateBinding 把模板内的属性桥接到控件本身的属性。

DataTemplate

定义数据对象如何呈现为 UI。一个 User 业务对象绑定到 ContentControl 时，由 DataTemplate 决定显示成卡片、行还是图标。是 MVVM 数据驱动的关键。

关键区别：ControlTemplate 描述“控件长什么样”（与数据无关），DataTemplate 描述“数据长什么样”。这两者解耦让 WPF 的视觉定制能力远超 WinForms。

05 布局系统

WPF 不使用绝对坐标，而是嵌套面板。每个 Panel 派生类实现自己的布局算法，两阶段：先 Measure（询问每个子元素想要多大），再 Arrange（按父面板规则放置子元素）。这让 UI 自然适应不同分辨率、字体大小、DPI。

核心面板对比

面板排列方式典型用途
StackPanel单行/单列堆叠简单表单、按钮组
WrapPanel流式自动换行标签云、图标墙
DockPanel停靠（Top/Bottom/Left/Right/Fill）主窗口结构（菜单顶、状态栏底、内容填充）
Grid二维行列网格（最强大）表单、复杂界面
UniformGrid等大格子网格棋盘、按键
Canvas绝对坐标图表、拖拽画板
VirtualizingStackPanel栈式 + 视口虚拟化ListBox/ListView 默认 ItemsPanel

Grid：最强大的面板

Grid 用 <RowDefinition> / <ColumnDefinition> 声明行列，Width 支持三种单位：

  • Auto：按内容自动撑开
  • *（比例）：占用剩余空间，2* 是 * 的两倍
  • 固定数值（如 100，单位为 device-independent pixel = 1/96 英寸）
<Grid>
    <Grid.ColumnDefinitions>
        <ColumnDefinition Width="Auto" />      <!-- 标签列，按内容 -->
        <ColumnDefinition Width="*"  />      <!-- 输入框列，填充 -->
    </Grid.ColumnDefinitions>
    <Grid.RowDefinitions>
        <RowDefinition Height="Auto" />
        <RowDefinition Height="Auto" />
        <RowDefinition Height="*"    />   <!-- 占满剩余 -->
    </Grid.RowDefinitions>

    <TextBlock Grid.Row="0" Grid.Column="0" Text="Name:" />
    <TextBox   Grid.Row="0" Grid.Column="1" Margin="4" />
    <TextBlock Grid.Row="1" Grid.Column="0" Text="Email:" />
    <TextBox   Grid.Row="1" Grid.Column="1" Margin="4" />
    <Button Grid.Row="2" Grid.Column="1"
             Content="OK" HorizontalAlignment="Right" />
</Grid>

这里的 Grid.Row / Grid.Column 是附加属性——由 Grid 定义，但附加到子元素身上（见 06 章）。

两阶段布局

// UIElement 提供的虚拟方法（简化）
protected abstract Size MeasureCore(Size availableSize);
protected abstract Rect ArrangeCore(Rect finalRect);

// Panel 子类重写 MeasureOverride / ArrangeOverride
protected override Size MeasureOverride(Size available) {
    Size desired = new(0, 0);
    foreach (UIElement child in InternalChildren) {
        child.Measure(available);          // 问子元素想要多大
        desired = Union(desired, child.DesiredSize);
    }
    return desired;
}

protected override Size ArrangeOverride(Size finalSize) {
    double y = 0;
    foreach (UIElement child in InternalChildren) {
        double h = child.DesiredSize.Height;
        child.Arrange(new Rect(0, y, finalSize.Width, h));  // 真正放置
        y += h;
    }
    return finalSize;
}

虚拟化（Virtualization）

VirtualizingStackPanel（ListBox/ListView/DataGrid 的默认 ItemsPanel）只为可见范围内的项生成容器，滚动时回收复用。结合 VirtualizingPanel.VirtualizationMode="Recycling"，可流畅处理 10 万级数据。

性能陷阱：把 ItemsControl 直接放在 ScrollViewer 内会破坏虚拟化——ScrollViewer 给 ItemsControl “无限大”的可用尺寸，导致所有项都被实例化。需要虚拟化时务必用 ListBox 等内置 VirtualizingStackPanel 的控件，且不要外层包 ScrollViewer。

06 依赖属性系统

依赖属性（DependencyProperty, DP）是 WPF 的灵魂。普通 CLR 属性只是字段包装，而 DP 是一个集中注册的、带优先级表的属性系统，原生支持样式、动画、绑定、继承、变更通知。WPF 中几乎所有 FrameworkElement 的属性都是 DP。

注册与包装

public class FancyButton : Button
{
    // 1. 注册：名称、类型、宿主类、默认元数据（默认值 + 变更回调）
    public static readonly DependencyProperty CornerRadiusProperty =
        DependencyProperty.Register(
            nameof(CornerRadius),         // 属性名（必须唯一）
            typeof(CornerRadius),    // 属性类型
            typeof(FancyButton),    // 宿主类型
            new PropertyMetadata(
                default(CornerRadius),    // 默认值
                OnCornerRadiusChanged));     // 变更回调（静态）

    // 2. CLR 包装：调用 GetValue/SetValue
    public CornerRadius CornerRadius {
        get => (CornerRadius)GetValue(CornerRadiusProperty);
        set => SetValue(CornerRadiusProperty, value);
    }

    // 3. 变更回调（实例 → 静态桥接）
    private static void OnCornerRadiusChanged(
        DependencyObject d, DependencyPropertyChangedEventArgs e) {
        var btn = (FancyButton)d;
        btn.InvalidateVisual();     // 触发重绘
    }
}

值优先级

DP 的“当前值”不是单一来源，而是从多个源按优先级取最高者。完整的优先级从高到低：

优先级来源说明
1（最高）CoerceValue 回调强制修正
2动画Active Animation
3本地值（Local Value）XAML/代码直接赋值或绑定
4Style 中的 TriggerStyle.Triggers / DataTrigger
5模板触发器ControlTemplate.Triggers
6Style Setter显式样式
7Theme Style主题默认样式
8继承（Inheritance）从父元素继承（DataContext/FontFamily）
9（最低）默认值PropertyMetadata 的 default
陷阱：一旦给属性赋了本地值（如 btn.Width = 100），它会压住同优先级及以下的 Style/Trigger/动画。要让动画或 Trigger 重新生效，必须调用 btn.ClearValue(WidthProperty) 清除本地值。这是 WPF 调试中“为什么样式没生效”的常见根因。

附加属性（Attached Property）

附加属性是一种特殊的 DP，由一个类型注册，但附加到另一个类型的实例上。最典型的是 Grid.Row / Canvas.Left / DockPanel.Dock——父面板把坐标信息“贴”在子元素身上。

// Canvas 注册的附加属性：Canvas.Left / Canvas.Top
public static readonly DependencyProperty LeftProperty =
    DependencyProperty.RegisterAttached(
        "Left", typeof(double), typeof(Canvas),
        new FrameworkPropertyMetadata(0.0,
            FrameworkPropertyOptions.AffectsParentArrange));

// 提供静态 Get/Set 访问器（XAML 编译器查找的就是这两个）
public static void SetLeft(UIElement e, double v) => e.SetValue(LeftProperty, v);
public static double GetLeft(UIElement e) => (double)e.GetValue(LeftProperty);

// XAML：Canvas.Left 写在子元素上，但由 Canvas 读取
<!-- <Ellipse Canvas.Left="50" Canvas.Top="20" /> -->

只读依赖属性

像 IsMouseOver、IsKeyboardFocusWithin 这类由框架维护的属性用 RegisterReadOnly 注册，返回 DependencyPropertyKey，对外只暴露 DependencyProperty。

DP 是 WPF 的基石：Style、Trigger、Animation、Binding、Property Value Inheritance、Validation——六大特性全部构建在 DP 之上。没有 DP 就没有 WPF 的声明式编程模型。理解 DP 优先级表，是排查“为什么样式/动画/绑定没生效”的第一步。

07 数据绑定

数据绑定是 WPF 的一等公民。一条 Binding 把源属性（source）与目标属性（target，必须是 DP）连起来，自动同步。绑定的源可以是任意对象，只要实现 INotifyPropertyChanged，变更就能自动传播。

Binding 表达式与 DataContext

DataContext 是 FrameworkElement 的可继承属性——子元素不显式指定 Source 时，会沿逻辑树向上找到最近的非 null DataContext。这是 MVVM 的核心机制：在 Window 上设一次 ViewModel，所有子控件都能绑定。

<!-- DataContext 继承：根窗口设一次 -->
<Window.DataContext>
    <local:MainViewModel />
</Window.DataContext>

<Grid>
    <!-- Path 默认就是 DataContext 上的属性 -->
    <TextBlock Text="{Binding Path=Title}" />
    <!-- Path= 可省略 -->
    <TextBlock Text="{Binding Title}" />
    <!-- {Binding .}：绑定 DataContext 本身 -->
    <ContentControl Content="{Binding .}" />
</Grid>

INotifyPropertyChanged 与 ObservableCollection

public class UserViewModel : INotifyPropertyChanged
{
    private string _name = "";
    public string Name {
        get => _name;
        set {
            if (_name != value) {
                _name = value;
                OnPropertyChanged();     // [CallerMemberName] 自动填 "Name"
            }
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string? p = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(p));
}

// 集合：ObservableCollection 在增删时触发 CollectionChanged
public ObservableCollection<UserViewModel> Users { get; } = new();

IValueConverter

源类型与目标类型不一致时（如 bool → Visibility），用 IValueConverter 转换。内置 BooleanToVisibilityConverter 最常用。

public class InverseBooleanConverter : IValueConverter {
    public object Convert(object v, Type t, object p, CultureInfo c)
        => !(bool)v;
    public object ConvertBack(object v, Type t, object p, CultureInfo c)
        => !(bool)v;
}

<!-- 在资源中实例化 -->
<!-- <local:InverseBooleanConverter x:Key="InvBool" /> -->
<!-- <CheckBox IsChecked="{Binding IsEnabled, Converter={StaticResource InvBool}}" /> -->
常见坑：INotifyPropertyChanged 触发时，属性名必须大小写完全一致，否则绑定不更新。用 [CallerMemberName] 比手写字符串更安全。此外，替换整个集合对象（Users = new ...）不会触发 ObservableCollection 的事件——必须在原集合上 Add/Remove，或让持有集合的属性也通知变更。

08 路由事件与命令

WPF 的路由事件（Routed Event）不是普通 CLR 事件，它会在元素树上传播。这让外层容器可以监听内层任意控件的事件——例如把 StackPanel 上所有 Button 的 Click 统一在一个处理器里。

三种路由策略

策略传播方向前缀约定典型事件
Tunnel（隧道）根 → 源PreviewPreviewMouseDown、PreviewKeyDown
Bubble（冒泡）源 → 根（多数）Click、MouseDown、TextChanged
Direct（直接）不传播—ButtonBase.Click（实为 Bubble，但语义直接）

隧道事件总是先于冒泡事件触发——这让父元素有机会预览并拦截。在 Preview* 中设 e.Handled = true 会让随后的对应 Bubble 事件也被取消。

// 路由事件的定义（注册到 EventManager）
public static readonly RoutedEvent TapEvent =
    EventManager.RegisterRoutedEvent(
        "Tap", RoutingStrategy.Bubble,
        typeof(RoutedEventHandler), typeof(MyControl));

public event RoutedEventHandler Tap {
    add => AddHandler(TapEvent, value);
    remove => RemoveHandler(TapEvent, value);
}

// 触发
protected virtual void OnTap() {
    RoutedEventArgs e = new(TapEvent, this);
    RaiseEvent(e);     // 沿可视化树冒泡
}

// 任意祖先都能监听后代元素的事件
// XAML: <StackPanel Button.Click="Stack_Click">
//      <Button /><Button /></StackPanel>
private void Stack_Click(object s, RoutedEventArgs e) {
    Button? btn = e.OriginalSource as Button;   // 实际起源
    Button? src = e.Source as Button;          // 逻辑源（可能被修正）
}

RoutedEventArgs 关键成员

  • Handled：设为 true 后续节点不再收到该事件（同样阻止对应 Preview/Bubble 配对）。
  • Source：逻辑树中触发事件的元素，可能被事件处理器显式修改。
  • OriginalSource：可视化树中真正的起源（例如 Button 内的 TextBlock）。
  • RoutedEvent：标识是哪个路由事件，可用来在多事件复用同一处理器时区分。

命令（Commands）

事件直接耦合 View 与 code-behind；命令则把“用户意图”抽象为可绑定对象，是 MVVM 解耦的关键。WPF 的命令系统由三部分组成：

  • RoutedCommand：WPF 内置命令实现，本身不含逻辑，只是个标识符（如 ApplicationCommands.Copy）。
  • CommandBinding：把命令绑定到具体的 Executed / CanExecute 处理器，通常挂在 Window/UserControl 上。
  • InputBinding：把快捷键/鼠标手势绑定到命令。
<!-- 内置命令绑定 -->
<Window.CommandBindings>
    <CommandBinding Command="ApplicationCommands.Open"
                   Executed="OnOpen" CanExecute="OnCanOpen" />
</Window.CommandBindings>

<Window.InputBindings>
    <KeyBinding Command="ApplicationCommands.Open"
                Gesture="Ctrl+O" />
</Window.InputBindings>

<Button Command="ApplicationCommands.Open" Content="Open" />

// code-behind
private void OnOpen(object s, ExecutedRoutedEventArgs e) { /* 打开 */ }
private void OnCanOpen(object s, CanExecuteRoutedEventArgs e) {
    e.CanExecute = HasDocument();
}

ICommand：MVVM 友好的命令

RoutedCommand 依赖 CommandBinding（通常在 code-behind），MVVM 中更常用 ICommand 接口，由 ViewModel 直接实现，无需 CommandBinding。

public interface ICommand {
    event EventHandler? CanExecuteChanged;
    bool CanExecute(object? parameter);
    void Execute(object? parameter);
}

// 经典 RelayCommand 实现（详见 10 章 MVVM）
public class RelayCommand : ICommand {
    private readonly Action<object?> _execute;
    private readonly Func<object?, bool> _canExecute;
    public RelayCommand(Action<object?> exec, Func<object?, bool>? can = null) {
        _execute = exec; _canExecute = can ?? (_ => true);
    }
    public bool CanExecute(object? p) => _canExecute(p);
    public void Execute(object? p) => _execute(p);
    public event EventHandler? CanExecuteChanged {
        add => CommandManager.RequerySuggested += value;
        remove => CommandManager.RequerySuggested -= value;
    }
}

09 样式与模板

WPF 把视觉外观完全从控件逻辑中剥离。Style 负责统一属性，ControlTemplate 负责重塑控件本身的外观，DataTemplate 负责重塑数据的呈现，三者结合可实现 WinForms 不可企及的视觉定制力。

Style 与 Setter

<!-- 在 Window.Resources 中定义命名样式 -->
<Window.Resources>
    <Style x:Key="PrimaryButton" TargetType="Button">
        <Setter Property="Background" Value="#0a6cff" />
        <Setter Property="Foreground" Value="White" />
        <Setter Property="FontWeight" Value="Bold" />
        <Setter Property="Padding" Value="16,8" />
        <Style.Triggers>
            <!-- 鼠标悬停 -->
            <Trigger Property="IsMouseOver" Value="True">
                <Setter Property="Background" Value="#0050c8" />
            </Trigger>

            <DataTrigger Binding="{Binding IsEnabled}" Value="False">
                <Setter Property="Opacity" Value="0.5" />
            </DataTrigger>

    </Style>
</Window.Resources>

<Button Style="{StaticResource PrimaryButton}" Content="Save" />

<!-- 无 x:Key 的 Style 自动应用到所有 TargetType（隐式样式） -->
<Style TargetType="TextBlock">      <!-- 所有 TextBlock 生效 -->
    <Setter Property="Foreground" Value="#333" />
</Style>

ControlTemplate：彻底重塑控件外观

<!-- 把 Button 变成圆角按钮 -->
<Style TargetType="Button">

资源与 ResourceDictionary

资源（画刷、样式、模板、转换器）存放在 <FrameworkElement.Resources> 或独立 ResourceDictionary（XAML 文件）中，用 x:Key 索引。StaticResource 在加载时一次性查找，性能好但替换资源不会更新引用；DynamicResource 运行时查找，能响应主题切换与语言切换。

StaticResource vs DynamicResource：StaticResource 要求在使用之前定义（XAML 是顺序解析），跨文件引用必须先 Merge；DynamicResource 可以前向引用、能动态切换主题，但每次访问都查字典、性能略低。主题相关的系统颜色（SystemColors.WindowBrushKey）必须用 DynamicResource 才能跟随系统暗黑模式切换。

10 MVVM 模式

MVVM（Model-View-ViewModel）是 WPF 的事实标准架构，由 John Gossman 于 2005 年提出，是 Martin Fowler 的 Presentation Model 在 WPF 上的具体化。核心思想：ViewModel 用纯 C# 表达 UI 状态与逻辑，View 用纯 XAML 绑定 ViewModel，二者完全解耦，ViewModel 可单元测试。

三层职责

层技术职责不应做的事
Model纯 C# / DTO业务实体、数据持久化不引用 UI、不实现 INPC
ViewModelC# + INPC + ICommandUI 状态、命令逻辑、转换不引用 View 类型
ViewXAML + 少量 code-behind布局、绑定、视觉不含业务逻辑

完整 ViewModel 示例

public class UserEditorViewModel : ObservableObject   // INPC 基类
{
    private string _name = "";
    private int _age;
    private User? _selected;

    public string Name {
        get => _name;
        set => SetProperty(ref _name, value);   // 触发 INPC
    }
    public int Age {
        get => _age;
        set => SetProperty(ref _age, value);
    }
    public User? Selected {
        get => _selected;
        set => SetProperty(ref _selected, value);
    }

    public ObservableCollection<User> Users { get; } = new();

    public ICommand AddCommand { get; }
    public ICommand RemoveCommand { get; }

    public UserEditorViewModel() {
        AddCommand    = new RelayCommand(_ => Add(), _ => !string.IsNullOrWhiteSpace(Name));
        RemoveCommand = new RelayCommand(_ => Remove(), _ => Selected is not null);
    }

    private void Add() {
        Users.Add(new User { Name = Name, Age = Age });
        Name = ""; Age = 0;
    }
    private void Remove() {
        if (Selected is not null) Users.Remove(Selected);
    }
}

CommunityToolkit.Mvvm 与源生成器

微软的 CommunityToolkit.Mvvm（前身是 MvvmLight / Prism 的部分能力）用 C# 源生成器（Source Generator）在编译期自动生成 INPC 与 ICommand 的样板，VM 代码变得极简：

public partial class UserEditorViewModel : ObservableObject   // partial 必须
{
    // [ObservableProperty] 自动生成 public string Name { get; set; } + INPC
    [ObservableProperty]
    private string _name = "";

    [ObservableProperty]
    private int _age;

    // [RelayCommand] 自动生成 ICommand AddCommand + 私有 Add 方法
    [RelayCommand(CanExecute = nameof(CanAdd))]
    private void Add() { /* ... */ }
    private bool CanAdd() => !string.IsNullOrWhiteSpace(Name);

    public ObservableCollection<User> Users { get; } = new();
}
MVVM 是 WPF 的最佳实践：它让 UI 与业务逻辑彻底解耦——ViewModel 不引用任何 UI 类型（无 Window、MessageBox），可独立测试，可在 WPF/UWP/WinUI/MAUI 间复用。View 通过 XAML 数据绑定驱动，状态变更由 INPC 自动传播。Designer 在设计时还能注入“假数据”预览界面。这套范式已成为 XAML 生态的通用架构。
何时违反 MVVM：纯 UI 行为（弹窗位置、Focus 管理、动画触发）可以放 code-behind，强行塞进 ViewModel 反而让 VM 依赖 View 抽象。MessageBox 这种“UI 服务”应抽象成 IDialogService 接口注入 VM，保持可测试性。

11 绘图与动画

WPF 的图形系统基于 DirectX，所有绘图都是矢量的，缩放无锯齿。Shape 是高层的可交互图形元素，Geometry 是纯几何描述，Drawing/Visual 是轻量的低开销绘图。动画则构建在依赖属性之上——任何 DP 都能被 Timeline 驱动。

Shape 与 Geometry

<!-- Shape：可直接放进面板的图形元素 -->
<Canvas>
    <Ellipse Width="100" Height="100" Fill="Orange" />
    <Rectangle Canvas.Left="120" Width="80" Height="40"
               Stroke="Black" StrokeThickness="2" />
    <Path Stroke="Blue" StrokeThickness="2">
        <Path.Data>
            <!-- Stream Geometry：M=Move, L=Line, Z=Close -->
            <PathGeometry Figures="M 10,10 L 100,50 L 10,90 Z" />
        </Path.Data>
    </Path>
</Canvas>

Drawing / Visual：轻量绘图

Shape 每个都是完整 FrameworkElement，参与布局与事件，开销大。Drawing 是纯绘图指令，DrawingVisual 是最轻量的 Visual——直接通过 DrawingContext 绘制，适合做图表、地图、粒子系统这类“上千个图形”的场景。

public class FastCanvas : FrameworkElement
{
    private readonly DrawingVisual _visual = new();

    public FastCanvas() {
        using (DrawingContext dc = _visual.RenderOpen()) {
            dc.DrawRectangle(Brushes.White, null, new Rect(0, 0, 1000, 1000));
            for (int i = 0; i < 10000; i++) {
                dc.DrawEllipse(Brushes.OrangeRed, null,
                    new Point(Rand(), Rand()), 2, 2);
            }
        }
        AddVisualChild(_visual);
        AddLogicalChild(_visual);
    }
    protected override int VisualChildrenCount => 1;
    protected override Visual GetVisualChild(int i) => _visual;
}

动画（Storyboard + Timeline）

WPF 动画的本质是“随时间改变依赖属性的值”。由于 DP 优先级中动画高于本地值，动画运行时本地值会被压住。Storyboard 是一组并行/串行 Timeline 的容器，通常由 EventTrigger 触发。

<Button Width="100" Height="40" Content="Pulse">
    <Button.Triggers>
        <EventTrigger RoutedEvent="Button.Click">
            <BeginStoryboard>
                <Storyboard>
                    <!-- DoubleAnimation：动画 double 类型的 DP（如 Width） -->
                    <DoubleAnimation Storyboard.TargetProperty="Width"
                                     To="200" Duration="0:0:0.4"
                                     AutoReverse="True" RepeatBehavior="3x" />
                    <!-- 用缓动函数让动画更自然 -->
                    <DoubleAnimation Storyboard.TargetProperty="Opacity"
                                     From="1" To="0.3" Duration="0:0:0.3">
                        <DoubleAnimation.EasingFunction>
                            <CubicEase EasingMode="EaseOut" />
                        </DoubleAnimation.EasingFunction>
                    </DoubleAnimation>
                </Storyboard>
            </BeginStoryboard>
        </EventTrigger>
    </Button.Triggers>
</Button>
内置动画类型
  • DoubleAnimation：double（Width/Opacity/Canvas.Left）
  • ColorAnimation：Color（SolidColorBrush.Color）
  • PointAnimation：Point（Path 几何节点）
  • ThicknessAnimation：Margin/Padding
  • ObjectAnimationUsingKeyFrames：离散对象（如图标切换）
EasingFunction 缓动
  • Linear：匀速
  • CubicEase / QuadraticEase：经典缓动
  • ElasticEase / BounceEase：弹性/弹跳
  • BackEase：回弹
  • CircleEase / SineEase：三角函数曲线
动画的硬件加速：WPF 中只要动画影响的是变换（Transform）、不透明度（Opacity）、剪裁（Clip），合成线程就能直接在 GPU 上插值，不占用 UI 线程。把 Button 包在 <RenderTransform><ScaleTransform/></RenderTransform> 里、动画 Scale，比动画 Width 高效得多。

12 完整 MVVM 示例

下面是一个完整的用户管理 CRUD 界面，体现 MVVM + 数据绑定 + 命令的完整协作：输入框双向绑定、列表绑定 ObservableCollection、按钮绑定 ICommand、状态控制 CanExecute。

Model

// Models/User.cs
namespace UserMgr.Models;

public class User
{
    public Guid Id { get; set; } = Guid.NewGuid();
    public string Name { get; set; } = "";
    public int Age { get; set; }
}

ViewModel

// ViewModels/UserEditorViewModel.cs
using CommunityToolkit.Mvvm.Input;
using CommunityToolkit.Mvvm.ComponentModel;
using UserMgr.Models;

namespace UserMgr.ViewModels;

public partial class UserEditorViewModel : ObservableObject
{
    [ObservableProperty]
    private string _name = "";

    [ObservableProperty]
    [NotifyPropertyChangedFor(nameof(CanAdd)))]   // Name 变化时刷新 CanAdd
    private int _age;

    [ObservableProperty]
    [NotifyCanExecuteChangedFor(nameof(RemoveCommand))]
    private User? _selected;

    public ObservableCollection<User> Users { get; } = new();

    public bool CanAdd => !string.IsNullOrWhiteSpace(Name) && Age > 0 && Age < 150;

    [RelayCommand(CanExecute = nameof(CanAdd))]
    private void Add() {
        Users.Add(new User { Name = Name, Age = Age });
        Name = ""; Age = 0;
    }

    [RelayCommand(CanExecute = nameof(CanRemove))]
    private void Remove() {
        if (Selected is not null) Users.Remove(Selected);
    }
    private bool CanRemove() => Selected is not null;
}

View（XAML）

13 对比 WinForms/WinUI

WPF 已是成熟稳定的框架，仍在维护但不再有大特性。微软当前主推的 Windows 原生 UI 框架是 WinUI 3 / Windows App SDK（以及跨平台的 .NET MAUI）。对开发者而言，三者各有定位：

框架渲染语言XAML数据绑定定位维护状态
WinFormsGDI+（即时）C# / VB无（设计器拖拽）弱（BindingSource）经典桌面、维护遗留稳定维护
WPFDirectX 9（保留）C# + XAML有（WPF XAML）强（DP + Binding）富客户端、企业 LOB稳定，无新特性
WinUI 3DirectX 12（WinRT）C# + C++ + XAML有（WinUI XAML）强（{x:Bind} 编译期）现代 Windows 原生微软主推
.NET MAUI各平台原生C# + XAML有（MAUI XAML）强跨平台（Win/Mac/iOS/Android）演进中
WPF vs WinForms
  • 渲染：WPF DirectX 矢量；WinForms GDI 位图
  • 布局：WPF 流式面板；WinForms 绝对坐标为主
  • 绑定：WPF 一等公民；WinForms 弱化（BindingSource）
  • XAML：WPF 声明式；WinForms 代码 + 设计器
  • 学习曲线：WPF 陡（DP/路由事件/MVVM）；WinForms 平
  • 性能：WinForms 轻量；WPF 启动慢、内存高
WPF vs WinUI 3
  • 架构：WPF 托管；WinUI 3 是 WinRT 投影（C++/C#）
  • 绑定：WPF {Binding} 反射；WinUI {x:Bind} 编译期
  • 主题：WPF 自定义；WinUI 内置 Fluent/Mica
  • 控件：WinUI 更现代（ColorPicker、TreeView 重设计）
  • 兼容：WPF 跨 Windows 版本好；WinUI 仅 Win10 1809+
  • 未来：WPF 维护态；WinUI 是微软下一代主推
选型建议：维护遗留系统、深度调用 Win32、企业内部 LOB 应用，WPF 仍是稳妥选择；全新面向消费者的现代 Windows 应用、需要 Fluent 设计，选 WinUI 3；跨平台需求选 .NET MAUI；极轻量工具类程序可继续用 WinForms。不要在 WPF 与 WinUI 之间犹豫太久——两者的 MVVM 范式与 XAML 概念高度一致，技能可迁移。
WPF 的局限：基于 .NET Framework / .NET Core 3+，仅 Windows；默认主题（Aero/Luna）相比 Fluent 显得过时（需第三方库如 WPF-UI、MaterialDesign 补救）；启动慢、内存占用高于 WinForms；3D 能力弱（基于 DirectX 9 的 Viewport3D，不再演进）。
