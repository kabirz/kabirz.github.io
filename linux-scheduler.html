01 概述

调度器是 Linux 内核最核心的子系统之一，负责决定哪个进程在哪个 CPU 上运行多长时间。Linux 采用模块化的调度类（sched_class）架构，支持多种调度策略共存。

调度器的核心职责
  • 选择进程：从就绪队列中选择最优任务执行
  • 时间分配：公平或按优先级分配 CPU 时间
  • 负载均衡：在多核间平衡工作负载
  • 实时性保证：确保实时任务的延迟约束
  • 能效管理：在性能和功耗间平衡
设计目标
  • 交互响应：桌面/移动设备的快速响应
  • 吞吐量：服务器的高吞吐处理
  • 公平性：多用户/容器环境的资源公平
  • 实时性：工业/音频的确定性延迟
  • 可扩展性：从单核到数千核的线性扩展

05 CFS 调度器

CFS（Completely Fair Scheduler）是 Linux 的默认调度器，用于 SCHED_NORMAL、SCHED_BATCH、SCHED_IDLE 策略。它基于虚拟运行时间（vruntime）实现 O(log n) 的公平调度。

核心原理

  • 虚拟运行时间：每个任务维护 vruntime，代表它已获得的"公平"CPU 时间
  • 权重机制：nice 值通过权重映射影响 vruntime 增长速度
  • 红黑树：按 vruntime 排序，最左节点（vruntime 最小）最先被调度
  • 时间片：基于 cfs_bandwidth 或 nr_running 动态计算

vruntime 计算

// 虚拟运行时间更新
delta_exec = now - prev->se.exec_start;     // 实际执行时间
vruntime += delta_exec * (NICE_0_LOAD / weight);  // 加权计算

// nice 值越小（优先级越高），权重越大，vruntime 增长越慢
// 因此高优先级任务获得更多 CPU 时间

// 权重表（sched_pelt.c）
// nice -20: weight = 88761    (最重)
// nice   0: weight = 1024     (基准)
// nice +19: weight = 15       (最轻)

CFS 带宽控制

cfs_bandwidth：通过 cpu.cfs_quota_us 和 cpu.cfs_period_us 限制 cgroup 的 CPU 使用量。当配额用尽时，cfs_rq 被 throttled，直到下一个周期开始。

06 RT 调度器

RT（Real-Time）调度器用于 SCHED_FIFO 和 SCHED_RR 策略，提供固定优先级的实时调度。

核心特性

SCHED_FIFO
  • 先进先出，不会被同优先级任务抢占
  • 高优先级任务总是抢占低优先级
  • 无时间片限制，一直运行直到阻塞或让出
  • 优先级 1-99（数值越大优先级越高）
SCHED_RR
  • 轮转调度，同优先级任务共享时间片
  • 时间片用完后放到同优先级队列尾部
  • 默认时间片 = 100ms（可配置）
  • 高优先级任务仍然抢占低优先级

优先级数组

// RT 使用位图 + 链表实现 O(1) 调度
struct rt_rq {
    struct rt_prio_array active;  // 100 个优先级的位图 + 链表
    unsigned int rt_nr_running;
    unsigned int rr_nr_running;
    // ...
};

struct rt_prio_array {
    DECLARE_BITMAP(bitmap, MAX_RT_PRIO+1);  // 100 位位图
    struct list_head queue[MAX_RT_PRIO];    // 100 个链表
};
带宽控制：sysctl_sched_rt_runtime（默认 950000us）和 sysctl_sched_rt_period（默认 1000000us）限制 RT 任务最多使用 95% 的 CPU 时间，防止饥饿 CFS 任务。

07 Deadline 调度器

Deadline 调度器实现 EDF（Earliest Deadline First）算法，提供严格的时间保证。每个任务有三个参数：运行时间（runtime）、周期（period）、截止时间（deadline）。

参数说明

// sched_setattr() 设置
struct sched_attr {
    u32 size;
    u32 sched_policy;
    u64 sched_flags;
    s32 sched_nice;
    u32 sched_priority;
    // Deadline 特定参数:
    u64 sched_runtime;    // 每次执行的最大时间
    u64 sched_deadline;   // 相对于周期的截止时间
    u64 sched_period;     // 重复周期
};

调度逻辑

EDF: 选择截止时间最早的任务
↓
runtime ≤ deadline ≤ period
保证 CPU 利用率 ≤ 100%
↓
任务按 (deadline - runtime) 排序
可调度性检查：使用 chrt -d 验证。Deadline 任务的总 CPU 利用率不能超过 100%，否则会拒绝设置。

08 sched_ext

sched_ext（SCHED_EXT，调度类 ext_sched_class）是 Linux 7.x 引入的 BPF 可编程调度器框架，允许通过 BPF 程序自定义调度策略，而无需修改内核。

核心特性

设计目标
  • 可编程：BPF 程序实现调度逻辑
  • 安全：BPF 验证器确保安全
  • 可热替换：运行时加载/卸载调度器
  • 兼容性：回退到 CFS
  • 可观测：内置跟踪和统计
典型用例
  • 游戏/VR 的低延迟调度
  • 容器的自定义资源分配
  • 研究新型调度算法
  • 特定工作负载的优化
  • 桌面响应性增强

09 负载均衡

负载均衡器在多核系统间迁移任务，确保各 CPU 的工作负载均衡。

调度域层级

root_domain
  └── sched_domain (SMT)        ← 同一物理核的超线程
        └── sched_domain (MC)    ← 同一物理核的多个核心
              └── sched_domain (DIE)  ← 同一 die 的核心
                    └── sched_domain (NUMA) ← 跨 NUMA 节点

均衡策略

触发函数说明
定时器load_balance()定期检查并迁移任务
唤醒select_task_rq_fair()唤醒时选择最空闲的 CPU
forkselect_task_rq_fair()新进程放置到最优 CPU
execsched_exec()exec 时迁移到最空闲 CPU
主动均衡active_balance()强制迁移特定任务

NOHZ（无滴答）

NO_HZ_FULL：当 CPU 上只有一个可运行任务时，停止定时器滴答（tickless），减少中断开销。这对计算密集型和延迟敏感型工作负载非常有益。

10 抢占模型

Linux 支持多种抢占模型，通过 CONFIG_PREEMPT 选项控制。

模型CONFIG说明
无抢占PREEMPT_NONE仅在返回用户态时检查，适合服务器
自愿抢占PREEMPT_VOLUNTARY在显式检查点（cond_resched）时可抢占
完全抢占PREEMPT内核几乎任何地方都可抢占
RT 抢占PREEMPT_RT实时补丁，大多数自旋锁变为可睡眠锁
