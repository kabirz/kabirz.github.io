01 概述

containerd 是从 Docker 中独立出来的容器运行时核心，2017 年捐赠给 CNCF，2019 年成为毕业项目。它是 Kubernetes、Docker、Podman 的底层基础。

定位：

层项目职责
编排Kubernetes, Docker Swarm多机调度
CLI / APIdocker CLI, crictl, ctr用户接口
Container Runtime (High-level)containerd, CRI-O镜像、快照、生命周期
Container Runtime (Low-level)runc, crun, kataOCI 标准执行

核心特性：

  • 最小化：只做容器运行时该做的事，不做编排、CLI、构建
  • 插件化：snapshotter、runtime、image store 都是插件
  • gRPC API：完全可程序化
  • 稳定接口：API v1 向后兼容
  • CRI 兼容：原生支持 Kubernetes

典型使用者：

  • Docker：作为底层运行时
  • Kubernetes：通过 CRI plugin
  • Containerd 直接使用：IoT、嵌入式、边缘
  • 云厂商：AWS Firecracker、GCP GKE、Azure AKS 都基于 containerd

03 Content Store

Content Store 是内容寻址的 blob 存储，所有镜像数据（manifest、config、layer）都存这里。

// containerd/core/content/content.go
type Store interface {
    Reader(ctx context.Context, dgst digest.Digest) (ReaderAt, error)
    Info(ctx context.Context, dgst digest.Digest) (Info, error)
    Update(ctx context.Context, info Info, fieldpaths ...string) (Info, error)
    Walk(ctx context.Context, fn WalkFunc, filters ...string) error
    Delete(ctx context.Context, dgst digest.Digest) error
    ListStatuses(ctx context.Context, filters ...string) ([]Status, error)
    Status(ctx context.Context, ref string) (Status, error)
    Abort(ctx context.Context, ref string) error
    Writer(ctx context.Context, opts ...WriterOpt) (Writer, error)
}

// 每个 blob 用 digest 标识：
type Digest = digest.Digest
// 例: "sha256:9759..."

// 文件存储路径：
// /var/lib/containerd/io.containerd.content.v1.content/blobs/sha256/ab/abc123.../data

// Info 元数据（存 BoltDB）：
type Info struct {
    Digest    digest.Digest
    Size      int64
    CreatedAt time.Time
    UpdatedAt time.Time
    Labels    map[string]string
}

// Writer 流式写入：
type Writer interface {
    io.WriteCloser
    Digest() digest.Digest
    Commit(ctx context.Context, size int64, expected digest.Digest, opts ...Opt) error
    Status(ctx context.Context) (Status, error)
    Truncate(size int64) error
}

// Status（下载进度）：
type Status struct {
    Ref       string
    Offset    int64
    Total     int64
    Expected  digest.Digest
    StartedAt time.Time
    UpdatedAt time.Time
}

04 Snapshotter

Snapshotter 把不可变的 image layer 转化为可写的容器 rootfs。

// containerd/core/snapshots/snapshotter.go
type Snapshotter interface {
    Stat(ctx context.Context, key string) (Info, error)
    Update(ctx context.Context, info Info, fieldpaths ...string) (Info, error)
    Usage(ctx context.Context, key string) (Usage, error)
    Mounts(ctx context.Context, key string) ([]mount.Mount, error)
    Prepare(ctx context.Context, key, parent string, opts ...Opt) ([]mount.Mount, error)
    View(ctx context.Context, key, parent string, opts ...Opt) ([]mount.Mount, error)
    Commit(ctx context.Context, name, key string, opts ...Opt) error
    Remove(ctx context.Context, key string) error
    Walk(ctx context.Context, fn WalkFunc, filters ...string) error
    Close() error
}

// 三种 snapshot 状态：
//   View     - 临时只读视图（用于查询）
//   Active   - 临时可写（容器在跑）
//   Committed- 已提交（变 image layer）

// 工作流：
//   1. image pull 时：每解压一层 → Commit 一个 snapshot
//      snapshotter.Commit(name="layer-N", key="work-N")
//      
//   2. 启动容器时：Prepare 一个 Active snapshot
//      snapshotter.Prepare(key="container-X", parent="layer-N")
//      → 返回 mount options
//      
//   3. 容器运行中：mount options 挂载到 rootfs
//      
//   4. 容器 commit 时：Commit Active snapshot → 新的 image layer

// mount.Mount：
type Mount struct {
    Type    string         // "overlay" / "bind" / ...
    Source  string         // 设备名或路径
    Target  string
    Options []string       // mount 选项
}

// 例：overlayfs mount
Mount{
    Type:    "overlay",
    Source:  "overlay",
    Options: []string{
        "lowerdir=/var/lib/containerd/.../snapshots/1/fs",
        "upperdir=/var/lib/containerd/.../snapshots/2/fs",
        "workdir=/var/lib/containerd/.../snapshots/2/work",
    },
}

多种 snapshotter 实现：

Snapshotter特点
overlayfs默认，Linux OverlayFS
native完整 copy（无 union）
btrfsBtrfs subvolume
zfsZFS filesystem
stargz延迟加载（lazy pull）
nydus镜像加速
fuse-overlayfs用户态 overlayfs（rootless）
devmapperdevice-mapper thin pool

07 插件系统

containerd 的所有组件都是插件，通过统一的 plugin 框架注册和发现：

// containerd/pkg/plugin/plugin.go
type Plugin interface {
    Init() error
    Instance() interface{}
    Meta() Registration
}

type Registration struct {
    Type     Type       // 插件类型
    ID       string     // 插件 ID（同 type 内唯一）
    Config   interface{}
    Requires []Type     // 依赖的其他插件类型
    InitFn   func(*InitContext) (interface{}, error)
}

type Type string
const (
    RuntimePlugin       Type = "io.containerd.runtime.v1"
    SnapshotPlugin      Type = "io.containerd.snapshotter.v1"
    TaskPlugin          Type = "io.containerd.task.v1"
    ContentPlugin       Type = "io.containerd.content.v1"
    DiffPlugin          Type = "io.containerd.differ.v1"
    MetadataPlugin      Type = "io.containerd.metadata.v1"
    ImagePlugin         Type = "io.containerd.image.v1"
    ContainerPlugin     Type = "io.containerd.container.v1"
    LeasePlugin         Type = "io.containerd.lease.v1"
    EventPlugin         Type = "io.containerd.event.v1"
    RuntimePluginV2     Type = "io.containerd.runtime.v2"
    MonitorPlugin       Type = "io.containerd.monitor.v1"
    TransferPlugin      Type = "io.containerd.transfer.v1"
    SandboxStorePlugin  Type = "io.containerd.sandbox.store.v1"
    SandboxControllerPlugin Type = "io.containerd.sandbox.controller.v1"
    GCPlugin            Type = "io.containerd.gc.v1"
    CRIGRPCPlugin       Type = "io.containerd.cri.v1"

// 注册插件（init.go）：
func init() {
    plugin.Register(&plugin.Registration{
        Type: RuntimePluginV2,
        ID:   "runc",
        Requires: []plugin.Type{
            plugin.MetadataPlugin,
            plugin.LeasePlugin,
            plugin.EventPlugin,
            plugin.TaskPlugin,
        },
        InitFn: func(ic *plugin.InitContext) (interface{}, error) {
            return runc.New(ic)
        },
    })
}

// 加载流程：
// 1. cmd/containerd/main.go
// 2. server.New(ctx, config)
// 3. plugin.Load(config.Plugins)
//    - 拓扑排序依赖
//    - 按序调用 InitFn
//    - 每个 plugin 返回实例
// 4. 注册到 service

// containerd/pkg/cri/server/
//   - PodSandbox  = containerd Sandbox + 多个 Container
//   - 通过 sandbox API 管理 pod 沙箱

// CRI 概念到 containerd 概念的映射：
//   CRI PodSandbox   →   containerd Sandbox (sandbox API v2)
//   CRI Container    →   containerd Container + Task
//   CRI Image        →   containerd Image
//   CRI Image Pull   →   containerd Image Pull

Kubelet 与 containerd 的交互：

// Kubelet 通过 CRI gRPC 调用 containerd
// Socket: /run/containerd/containerd.sock

// 启动 Pod：
// 1. Kubelet → RunPodSandbox(pod_sandbox_config)
//    containerd:
//    - 创建 sandbox container（pause 容器）
//    - 配置网络（CNI）
//    - 持有 sandbox namespace
//
// 2. Kubelet → CreateContainer(pod_sandbox_id, container_config)
//    containerd:
//    - 解析 image
//    - 创建 containerd container（共享 sandbox ns）
//
// 3. Kubelet → StartContainer(container_id)
//    containerd:
//    - 启动 task
//
// 4. 停止：
//    StopPodSandbox → 停止所有容器，保留沙箱
//    RemovePodSandbox → 删除沙箱和容器

// pause 容器（pause image）：
//   一个永远 sleep 的极简进程
//   持有 pod 的所有 namespace（PID、NET、IPC、UTS）
//   其他容器共享 pause 的 namespace
//   pause 死亡 → 整个 pod 死亡
