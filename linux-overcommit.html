01 概述

Overcommit（内存超分配）是 Linux 内核允许进程申请的虚拟内存超过物理内存总量的策略。大多数程序 malloc 了大量内存但只使用一小部分，overcommit 利用这一特性提高内存利用率。但如果超分配过多，可能导致 OOM Killer 杀死进程。

02 三种模式

模式vm.overcommit_memory行为
启发式（默认）0内核估计是否合理，过度分配时拒绝
总是允许1总是允许 overcommit（Redis 推荐）
严格限制2可用 RAM + swap × overcommit_ratio% 才允许
overcommit_memory = 0（默认，启发式）：

  进程 mmap(1GB)
        │
        ↓
  内核检查 overcommit 可用量
        │
        ├── 可用量 > 0 → 允许，虚拟内存增加
        │
        └── 可用量 ≤ 0 → 检查申请是否"合理"
            │
            ├── 合理（单次大量分配）→ 允许
            └── 不合理（疯狂小分配）→ 拒绝，返回 ENOMEM

overcommit_memory = 1（总是允许）：

  任何 mmap/brk 请求都允许
  适合：Redis、数据库等"我知道自己在做什么"的应用
  风险：进程可能分配远超物理内存的虚拟内存

overcommit_memory = 2（严格限制）：

  可用内存 = SwapTotal × overcommit_ratio/100 + CommitLimit
  每次分配都检查：已提交 + 新请求 ≤ CommitLimit
  超过 → 立即拒绝 ENOMEM
  适合：对内存稳定性要求极高的场景

03 overcommit 检查流程

__vm_enough_memory() — overcommit 检查入口：

  进程申请内存（mmap / brk / malloc）
        │
        ↓
  __vm_enough_memory()
        │
        ├── overcommit_memory == 1 → 总是返回 0（允许）
        │
        ├── overcommit_memory == 2 →
        │       计算 allowed = SwapTotal × ratio/100
        │       如果 committed + pages > allowed → 拒绝
        │
        └── overcommit_memory == 0（默认）→
                计算 available = free + buffers + cached - commit_limit
                │
                ├── available > 0 → 允许
                │
                └── available ≤ 0 → 估算"合理性"
                    │
                    ├── 申请量 < 总内存 / 8 → 允许（小分配通常安全）
                    ├── 申请量 < 总内存 / 4 → 可能允许
                    └── 申请量 > 总内存 / 4 → 拒绝

04 OOM Killer

OOM Killer 触发条件：

  overcommit 允许了过多分配
        │
        ↓
  进程实际访问页面 → 缺页异常
        │
        ↓
  物理内存不足，无法分配页面
        │
        ├── 已有 swap → 换出页面（kswapd）
        │
        └── 无 swap 或 swap 也满
            │
            ↓
  out_of_memory() → OOM Killer
        │
        ├── 计算每个进程的 oom_score
        │       oom_score = RSS + swap + page_table + oom_score_adj
        │
        ├── 分数最高的进程被选中
        │
        ├── 发送 SIGKILL
        │
        └── 进程退出 → 释放内存 → 恢复

  oom_score_adj（用户可调）：
    -1000 = 永不 OOM（如 oom_score_adj=-1000）
       0 = 默认（大多数进程）
    1000 = 最先被 OOM 杀死

# 查看进程 OOM 分数
$ cat /proc/self/oom_score
0

# 设置 OOM 优先级
$ echo -1000 > /proc/self/oom_score_adj  # 保护自己
$ echo 1000 > /proc/self/oom_score_adj   # 优先被杀

# 查看 OOM 日志
$ dmesg | grep -i "out of memory"

05 调优

参数默认值说明

vm.overcommit_ratio50overcommit_memory=2 时的比率
vm.oom_kill_allocating_task01=杀死申请者而非最高分进程
vm.panic_on_oom01=OOM 时内核 panic
vm.min_free_kbytes~3%RAM保留的最低空闲内存
Redis 推荐配置
Redis 建议设置 vm.overcommit_memory=1，因为 Redis fork() 时需要大量内存（COW），启发式模式可能误判拒绝。
