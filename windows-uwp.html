01 概述

UWP（Universal Windows Platform，通用 Windows 平台）是微软在 Windows 10（2015） 推出的统一应用模型。它基于 Windows Runtime（WinRT）这一原生 COM 基底的 API 层，提供一套跨设备（PC、手机、Xbox、HoloLens、IoT）的 API 表面，通过 Microsoft Store + MSIX 分发，运行在 AppContainer 沙箱中，实现安全隔离与可信安装。应用以 C# / VB / C++ / JavaScript 编写，UI 由 XAML（Windows.UI.Xaml）声明与驱动。

核心特性
  • 一次开发，多端运行：同一程序包跑在 PC/手机/Xbox/HoloLens
  • WinRT API 投影：原生 COM，多语言自然调用
  • 沙箱隔离：AppContainer + Capability 声明权限
  • MSIX 打包：清单驱动、证书签名、干净卸载
  • Store 分发：认证、自动更新、企业私域分发
  • 自适应 UI：VisualStateManager + AdaptiveTrigger 断点
  • 异步优先：WinRT 所有 I/O 暴露为 IAsyncOperation
适用场景
  • 面向 Microsoft Store 的消费类应用
  • Xbox / HoloLens / Surface Hub 设备应用
  • 需要安全沙箱、低权限的企业工具
  • 深度集成 Toast/磁贴/Live Tile 通知的应用
  • 跨 Windows 设备家族复用代码的 LOB 应用
为何被 WinUI 3 取代：UWP 的 XAML 栈随操作系统版本发布，API 表面受目标版本约束，开发者无法在旧系统上使用新控件；AppContainer 沙箱也限制了 Win32 互操作。微软自 2020 年起把 XAML 剥离成 WinUI 3（Microsoft.UI.Xaml），随应用而非系统分发，新项目应优先选用 WinUI 3 / Windows App SDK。

02 架构与语言投影

UWP 的底层是 Windows Runtime（WinRT）——一组以 原生 COM（IUnknown、注册表元数据、.winmd）为基底的二进制 API，所有 Windows 设备家族共享。各语言通过语言投影（Language Projection）把 WinRT 投射为本地惯用法：C# 看到 .NET 风格类型，C++/WinRT 看到 Modern C++ 类，JavaScript 看到Promise。上层 Windows.UI.Xaml 提供 XAML 框架，应用运行在 AppContainer 进程内。

┌──────────────────────────────────────────────────────────┐
│  你的应用（C# / C++/WinRT / JavaScript / VB）              │
│        XAML（Windows.UI.Xaml）+ 业务逻辑                    │
├──────────────────────────────────────────────────────────┤
│  语言投影：C#/.NET · C++/WinRT · JavaScript · Rust(社区)    │
├──────────────────────────────────────────────────────────┤
│  Windows Runtime（WinRT，COM 基底 + .winmd 元数据）         │
│   Windows.UI · Windows.Storage · Windows.Devices · ...     │
├──────────────────────────────────────────────────────────┤
│  Win32 / NT 内核 / 驱动 / HAL                               │
│  进程隔离：AppContainer 沙箱（低完整性 + Capability）         │
└──────────────────────────────────────────────────────────┘
语言投影形态特点
C# / VB.NET托管（.NET Native / .NET）最常用，await 自动投影 WinRT 异步；XAML 设计器支持最好
C++/WinRTModern C++（头文件库）微软推荐，零开销投影；投影代码由 cppwinrt.exe 生成
C++/CXC++ 语言扩展（^ 句柄）已废弃，VS 2022 起不再推荐，应迁移到 C++/WinRT
JavaScriptHTML/CSS/JS（已弱化）UWP 早期支持，新项目不应再选；已转向 PWA + WebView
.NET（C# 6+）.NET Native 编译发布时 AOT 编译为本地代码，启动快、无 JIT
WinRT = COM + 元数据：每个 WinRT 类型都在 .winmd（ECMA-335 格式）中声明，投影层据此生成本地语言的绑定。无论用哪种语言调用，最终都是 COM vtable 调用，没有跨语言封送的运行时开销。

03 应用模型与生命周期

UWP 应用以 Windows.UI.Xaml.Application 为入口，三层结构 Window → Frame → Page 承载 UI。生命周期状态机由系统驱动：用户切走或锁屏时进入 Suspended，内存压力下被 Terminated。开发者需在 Suspending 事件中持久化会话状态，在 OnLaunched 中检测上次是否被终止并恢复。

三层 UI 结构
  • Window：顶级窗口，每应用可多窗口
  • Frame：导航容器，维护页面栈
  • Page：单个 UI 视图，XAML + code-behind
  • Frame.Navigate(typeof(DetailPage), param) 入栈
  • Frame.GoBack() 出栈
生命周期状态机
状态含义
NotRunning未启动
Running前台可见，正常执行
Suspended用户切走，冻结但驻留内存
Terminated系统回收内存，需重启重建
ClosedByUser用户主动关闭（不算 Terminate）
// App.xaml.cs — 应用入口与启动导航
sealed partial class App : Application
{
    public App() {
        this.InitializeComponent();
        Suspending          += OnSuspending;
        Resuming           += OnResuming;
        UnhandledException += OnUnhandled;
    }

    // 启动或激活时调用 —— 创建窗口、导航首页
    protected override async void OnLaunched(LaunchActivatedEventArgs e) {
        Frame rootFrame = Window.Current.Content as Frame;
        if (rootFrame == null) {
            rootFrame = new Frame();
            // 恢复上次会话的导航状态
            rootFrame.NavigationFailed += OnNavigationFailed;
            if (e.PreviousExecutionState == ApplicationExecutionState.Terminated) {
                // 读取 SuspensionManager 保存的导航状态
                await SuspensionManager.RestoreAsync();
            }
            Window.Current.Content = rootFrame;
        }
        if (rootFrame.Content == null)
            rootFrame.Navigate(typeof(MainPage), e.Arguments);
        Window.Current.Activate();
    }

    // 挂起：保存状态（约 5 秒内必须完成，否则被强制终止）
    private async void OnSuspending(object s, SuspendingEventArgs e) {
        SuspendingDeferral deferral = e.SuspendingOperation.GetDeferral();
        await SuspensionManager.SaveAsync();      // 持久化 Frame 栈
        await ApplicationData.Current.LocalSettings.SaveAsync();
        deferral.Complete();
    }

    private void OnResuming(object s, object e) {
        // 从 Suspended 恢复，无需重建（进程仍在）
    }
}
应用数据存储
  • LocalSettings：本地键值，仅本机
  • RoamingSettings：跨设备漫游（同账号）
  • LocalFolder / RoamingFolder / TemporaryFolder
  • ApplicationDataContainer：分层设置容器
  • 漫游数据有 100KB 上限，超出停止同步
SuspensionManager
  • 微软模板提供的辅助类
  • 序列化 Frame 的页面栈与参数
  • 挂起时 SaveAsync，恢复时 RestoreAsync
  • 保存到 LocalSettings["_SuspensionManager"]
  • 配合 NavigationCacheMode 重建页面状态
挂起时限：系统发出 Suspending 后给应用 5 秒 完成保存（GetDeferral 延长窗口）。超时会被强制 Terminate，用户下次启动看到的是冷启动而非恢复。耗时操作应分块或转后台任务。

04 XAML 控件与布局

UWP 用 XAML 声明 UI，根命名空间是 http://schemas.microsoft.com/winfx/2006/xaml/presentation（即 Windows.UI.Xaml），与 WPF 的 System.Windows 同源但不完全兼容——UWP 是 WPF XAML 的精简分支。控件随系统版本演进，配合 VisualStateManager + AdaptiveTrigger 实现跨屏宽自适应。

样式语法基本一致更丰富的 Triggers
编译绑定x:Bind仅 {Binding}
常用控件
  • 基础：Button / TextBox / TextBlock / ToggleSwitch / Slider
  • 容器：Grid / StackPanel / RelativePanel / Border
  • 集合：ListView / GridView / ItemsRepeater
  • 导航：Frame / NavigationView / Pivot
  • 命令：CommandBar / AppBarButton / SplitView
  • 现代：AutoSuggestBox / MapControl / InkCanvas

自适应布局：AdaptiveTrigger + VisualStateManager

NavigationView 是 UWP 的现代导航控件（汉堡菜单 + 顶部/左侧模式），自带自适应行为：展开 / 紧凑 / 最小化三态随窗口宽度自动切换，PaneDisplayMode 可手动覆盖。新项目应优先用它而非手工拼 SplitView。

05 数据绑定

UWP 同时支持两套绑定：{Binding}（WPF 经典反射绑定，运行时解析 DataContext）与 {x:Bind}（编译期绑定，强类型、高性能，UWP 引入并随 WinUI 演进）。x:Bind 把绑定路径映射到 code-behind 的字段/属性/方法，编译时类型检查，性能比 {Binding} 高一个数量级。

维度{Binding}（经典）{x:Bind}（编译绑定）
解析运行时反射编译期生成代码
类型安全无（出错运行时才暴露）有，编译失败即暴露
性能基线约 10x（无反射）
数据源DataContextpage/usercontrol 自身（this）或其成员
函数绑定需 IValueConverter原生支持 {x:Bind FormatName(Item)}
事件绑定不支持{x:Bind OnClick} 直接绑方法
Mode 默认OneWayOneTime（需显式声明 OneWay/TwoWay）
// ViewModel.cs —— INotifyPropertyChanged + ObservableCollection
public class MainViewModel : INotifyPropertyChanged
{
    private TodoItem _selected;
    public TodoItem Selected {
        get => _selected;
        set { if (_selected != value) { _selected = value; OnPropertyChanged(); } }
    }

    public ObservableCollection<TodoItem> Items { get; } = new();

    public string FormatCount(int n) => $"{n} 项";   // x:Bind 函数绑定可直接调用

    public event PropertyChangedEventHandler PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string p = null)
        => PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(p));
}

// MainPage.xaml.cs —— x:Bind 默认绑定到 this 上的成员
public sealed partial class MainPage : Page
{
    public MainViewModel ViewModel { get; } = new MainViewModel();

    public MainPage() { InitializeComponent(); Loaded += OnLoaded; }

    private async void OnLoaded(object s, RoutedEventArgs e) {
        await ViewModel.LoadAsync();
        Bindings.UpdateTarget();   // x:Bind 默认 OneTime，需手动刷新或用 Mode=OneWay
    }

    // x:Bind 事件绑定：直接指向方法（签名须匹配委托）
    private void OnAddClick(object sender, RoutedEventArgs e) {
        ViewModel.Items.Add(new TodoItem());
    }
}

Binding Mode
  • OneTime：初始化一次（x:Bind 默认）
  • OneWay：源 → 目标（{Binding} 默认）
  • TwoWay：双向（表单控件常见）
UpdateSourceTrigger
  • Default：失去焦点时回写
  • PropertyChanged：每次输入即回写
  • Explicit：手动 Bindings.UpdateSource
  • 仅 TwoWay 生效
ObservableCollection<T> 实现 INotifyCollectionChanged，Add/Remove/Clear 自动触发 UI 刷新；但批量改动会逐条通知，应考虑 ObservableVector<T> 或临时 Suspension 模式减少重绘。

06 异步编程

WinRT 所有可能阻塞的 API 都以异步接口暴露：IAsyncOperation<TResult>（有返回值）、IAsyncAction（无返回值）、IAsyncOperationWithProgress<T,P>（带进度）。C# 投影把它们映射为 Task 兼容形式，可用 await 直接消费；C++/WinRT 用 co_await 协程。

WinRT 接口C# 投影语义
IAsyncActionTask无返回值的异步操作
IAsyncOperation<T>Task<T>有返回值的异步操作
IAsyncActionWithProgress<P>Task + 进度回调带进度的 Action
IAsyncOperationWithProgress<T,P>Task<T> + 进度回调带进度的 Operation
// 文件读取：StorageFile.OpenAsync → IAsyncOperation<IRandomAccessStream>
private async void OnOpenFileClick(object sender, RoutedEventArgs e)
{
    FileOpenPicker picker = new FileOpenPicker();
    picker.FileTypeFilter.Add(".txt");

    StorageFile file = await picker.PickSingleFileAsync();   // IAsyncOperation<StorageFile>
    if (file == null) return;

    CancellationTokenSource cts = new CancellationTokenSource();
    try {
        IRandomAccessStream stream = await file.OpenAsync(FileAccessMode.Read)
                                                  .AsTask(cts.Token);

        using (DataReader reader = new DataReader(stream.GetInputStreamAt(0))) {
            uint len = (uint)stream.Size;
            await reader.LoadAsync(len);              // 加载到缓冲
            StatusText.Text = reader.ReadString(len);   // 切回 UI 线程更新（默认同上下文）
        }
    }
    catch (OperationCanceledException) {
        StatusText.Text = "已取消";
    }
    catch (Exception ex) {
        StatusText.Text = $"失败: {ex.Message}";
    }
}

// 自定义异步方法返回 IAsyncOperation（C# 直接返回 Task 即可）
public async Task<string> FetchAsync(CancellationToken ct) {
    HttpClient client = new HttpClient();
    HttpResponseMessage resp = await client.GetAsync("https://api.example.com", ct);
    resp.EnsureSuccessStatusCode();
    return await resp.Content.ReadAsStringAsync();
}
UI 线程上下文
  • UWP：CoreDispatcher（Window.Current.Dispatcher.RunAsync）
  • WinUI 3：DispatcherQueue（API 名变化）
  • await 默认捕获 SynchronizationContext，回到原线程
  • 后台线程更新 UI 必须切回，否则异常
取消与超时
  • CancellationTokenSource 触发取消
  • .AsTask(token) 桥接 WinRT → .NET
  • cts.CancelAfter(TimeSpan.FromSeconds(5)) 超时
  • OperationCanceledException 非错误，不应上报
async void 的陷阱：事件处理器只能用 async void，其他场景一律 async Task。async void 中的异常无法被 await 捕获，会直接到 Application.UnhandledException。务必在事件处理器内 try/catch。

07 沙箱与权限

UWP 应用运行在 AppContainer 进程内，默认以低完整性令牌运行，所有敏感能力（Capability）必须在 Package.appxmanifest 中静态声明——用户安装时一次性授权，运行时不再弹窗（少数隐私类除外）。文件系统、网络、设备访问都受此边界约束。

Capability用途类别
internetClient出站 HTTP/Socket（默认必需）常规
privateNetworkClientServer访问局域网（家庭/企业网）常规
internetClientServer入站连接（服务端）常规
removableStorage访问 U 盘/SD 卡（需文件类型声明）常规
webcam / microphone摄像头 / 麦克风隐私
locationGPS 定位隐私
contacts / appointments通讯录 / 日历隐私
enterpriseAuthentication域账号 SSO受限
broadFileSystemAccess任意路径文件访问受限（需用户设置开启）
runFullTrust以完整 Win32 权限运行（桌面桥）受限
<!-- Package.appxmanifest —— 声明能力（静态授权） -->
<Package ...>
  <Capabilities>
    <!-- 常规能力 -->
    <Capability Name="internetClient" />
    <Capability Name="privateNetworkClientServer" />

    <!-- 设备能力（硬件接口） -->
    <DeviceCapability Name="webcam" />
    <DeviceCapability Name="microphone" />

    <!-- 受限能力（需特殊账号 / 用户设置开启） -->
    <rescap:Capability xmlns:rescap="..."
                          Name="broadFileSystemAccess" />
  </Capabilities>
</Package>
文件访问边界
  • 自由访问：ApplicationData.LocalFolder/RoamingFolder（应用数据目录）
  • Picker 协商：FileOpenPicker / FolderPicker 用户选
  • FutureAccessList：Picker 选后保留令牌（最多 1000 项）
  • 库访问：KnownFolders.PicturesLibrary 等需 Capability
  • 全盘访问：仅 broadFileSystemAccess（用户设置开启）
网络与隐私
  • 网络隔离：默认仅出站，无监听
  • 隐私 Capability：摄像头/位置/麦克风/通讯录
  • 安装时静态声明，运行时首次使用弹系统授权
  • 用户可在设置中随时撤回
  • 访问前应 await Permissions.RequestAsync()
沙箱是 UWP 的本质限制：不能 CreateProcess 启动任意 Win32 程序（需 runFullTrust + 桌面桥扩展），不能写注册表 HKLM，不能注入系统进程。这正是 UWP 被 Win32 用户诟病的原因——也是 WinUI 3 + WinAppSDK 取代它的关键动机之一。

08 后台任务与通知

应用在前台时可直接执行；用户切走进入 Suspended 后，所有代码冻结。要做定时同步、消息推送、Tile 更新，必须注册后台任务（BackgroundTask）。系统在触发条件满足时另起进程执行 IBackgroundTask.Run，资源严格配额（CPU/网络/电量）。

后台任务组成
  • 触发器：TimeTrigger / SystemTrigger / PushTrigger / MaintenanceTrigger
  • 条件：InternetAvailable / UserNotPresent / BatteryNotLow
  • 入口：实现 IBackgroundTask 的类（独立项目）
  • 注册：BackgroundTaskBuilder.Register
  • 清单：声明 <Extension Category="windows.backgroundTasks">
通知类型
类型用途
Toast弹出横幅通知
Tile开始菜单磁贴
Badge磁贴角标数字
RawWNS 推送原始数据
// 后台任务入口（运行在独立进程，无 UI）
public sealed class SyncTask : IBackgroundTask
{
    public async void Run(IBackgroundTaskInstance taskInstance) {
        BackgroundTaskDeferral deferral = taskInstance.GetDeferral();
        try {
            taskInstance.Canceled += (s, e) => { /* 取消清理 */ };
            await CloudSync.PullAsync();
            ShowToast("同步完成");
        }
        finally { deferral.Complete(); }
    }
}

// 前台注册后台任务
public static async void RegisterSyncTask() {
    var status = await BackgroundExecutionManager.RequestAccessAsync();
    if (status == BackgroundAccessStatus.DeniedByUser) return;

    BackgroundTaskBuilder builder = new BackgroundTaskBuilder {
        Name = "SyncTask",
        TaskEntryPoint = typeof(SyncTask).FullName
    };
    builder.SetTrigger(new TimeTrigger(15, false));  // 每 15 分钟（最小间隔）
    builder.AddCondition(new SystemCondition(SystemConditionType.InternetAvailable));
    BackgroundTaskRegistration reg = builder.Register();
}

单进程后台任务（Windows 10 1709+）允许后台入口直接写在主程序内，省去独立项目，BackgroundTaskBuilder.TaskEntryPoint 留空即可。触发时以同进程内组件运行，启动开销更低。

09 打包与分发

UWP 应用以 MSIX / AppX 包格式分发，清单驱动：Package.appxmanifest 声明应用身份、能力、可视化元素、入口点、扩展等。包必须用证书签名才能安装（自签名测试 / Store 证书 / 企业代码签名）。通过 Microsoft Store 提交，经认证流程（自动检测恶意代码、API 合规、年龄分级、定价）后上架。

清单节点作用
Identity包名 Name + 版本 Version + 发布者
Properties显示名、Logo、描述、能力声明
Dependencies目标设备家族 + 最低 SDK 版本
Applications可执行入口、可视化元素（Logo/磁贴/闪屏）
Capabilities权限声明（internet/webcam/broadFileSystemAccess 等）
Extensions文件关联、协议激活、后台任务、共享目标等
VisualElements磁贴模板、闪屏、前景/背景色

<Package
  xmlns="http://schemas.microsoft.com/appx/manifest/foundation/windows10"
  xmlns:mp="http://schemas.microsoft.com/appx/2014/phone/manifest"
  IgnorableNamespaces="uap mp">

  <Identity Name="Contoso.MyApp"
            Publisher="CN=Contoso"
            Version="1.0.0.0" />

    <Resource Language="en-us"/>
  </Resources>

  <Applications>
    <Application Id="App" Executable="$targetnametoken$.exe" EntryPoint="App.App">
      <uap:VisualElements

打包流程
  • VS 右键项目 → 打包 → 生成 .msix/.appx
  • Release 包需用有效证书签名
  • 自签名测试：VS 生成本地证书，开发机 sideload
  • Store 上传：.msixupload 包含所有架构
  • 应用包捆绑（.msixbundle）合并 x86/x64/ARM64
Store 提交流程
  • Partner Center 创建应用，预留名称
  • 上传包（自动认证扫描）
  • 填写年龄分级、隐私政策、定价
  • 认证：技术合规 + 内容合规（约 1-3 天）
  • 版本管理：每次上传需递增 Version
  • 可选 Flight 灰度 / 私域分发
版本号约定：Version 格式为 主.次.修订.构建。Store 拒绝递减版本，且每个上传包必须严格大于已发布版本。框架依赖包（Framework）与应用包版本独立管理。

10 对比 WinUI 3

UWP 的核心痛点在于 XAML 框架绑定操作系统——新控件要等 Windows 大版本，老系统用户用不上；同时 AppContainer 沙箱让 Win32 互操作困难，开发者大量需求 runFullTrust 绕过。WinUI 3（Windows App SDK 的一部分）把 XAML 栈剥离系统，作为 NuGet 包随应用分发，解决版本绑定，并默认以标准 Win32 进程运行（无沙箱限制）。

维度UWPWinUI 3 / WinAppSDK
XAML 命名空间Windows.UI.XamlMicrosoft.UI.Xaml
XAML 分发随系统随应用（NuGet）
目标版本受最低系统约束1809+ 全可用最新控件
进程模型AppContainer 沙箱标准 Win32 进程（可选拓宽）
Win32 互操作受限，需 runFullTrust原生支持，任意 Win32 API
分发仅 Store / MSIXMSIX / 解压安装 / 任意
调度器CoreDispatcherDispatcherQueue
未来路线仅维护，不再演进微软主推
// UWP 命名空间
using Windows.UI.Xaml;
using Windows.UI.Xaml.Controls;
using Windows.UI.Core;              // CoreDispatcher

// WinUI 3 命名空间（迁移：全局替换 Windows.UI → Microsoft.UI）
using Microsoft.UI.Xaml;
using Microsoft.UI.Xaml.Controls;
using Microsoft.UI.Dispatching;     // DispatcherQueue

// Dispatcher 调用切换
// UWP:
await Window.Current.Dispatcher.RunAsync(CoreDispatcherPriority.Normal, () => { });

// WinUI 3:
m_window.DispatcherQueue.TryEnqueue(DispatcherQueuePriority.Normal, () => { });
迁移要点
  • 项目类型：Windows 应用 (C#) WinAppSDK 模板
  • 全局替换 Windows.UI → Microsoft.UI
  • Window.Current → m_window 实例
  • CoreDispatcher → DispatcherQueue
  • ApplicationView 部分能力由 AppWindow 取代
  • 沙箱取消后，broadFileSystemAccess 不再需要
新项目建议
  • 新项目首选 WinUI 3（WinAppSDK 1.4+）
  • 已有 UWP 可继续维护（仍受支持）
  • 增量迁移：先迁 XAML 命名空间
  • WinUI 3 桌面应用 = Win32 + XAML
  • WPF/WinForms 可嵌入 WinUI 3 控件
  • 长期看 UWP 进入维护模式
新项目不要再选 UWP：微软已明确不再为 UWP 添加新功能，新 API 直接进入 WinAppSDK。UWP 仍可维护与提交 Store，但任何长期项目都应从 WinUI 3 / Windows App SDK 起步，避免未来再迁移。仅当目标是 Xbox 紧密集成或 HoloLens 旧设备时才考虑 UWP。
