01 概述

sched_ext（SCHED_EXT）是 Linux 内核中的一个可扩展调度类。它允许开发者通过 BPF 程序实现自定义调度策略，而无需编译或重启内核。

传统上，调度器是内核中最"神圣"的核心模块——修改调度策略需要改内核代码、重新编译、部署新内核。sched_ext 将调度策略变成了可动态加载的 BPF 程序。

传统调度器
  • 调度策略写死在内核中
  • 修改 → 编译内核 → 重启
  • 安全 → 内核社区 Code Review
  • 每个人只能用同一种策略
  • 实验性调度器很难合并
sched_ext
  • 调度策略 = BPF 程序
  • 修改 → bpftool 加载 → 立即生效
  • 安全 → BPF Verifier
  • 不同场景可加载不同调度器
  • 写一个实验调度器 = 写一个 .bpf.c 文件

这是"机制而非策略"（mechanism not policy）哲学的极致体现。

sched_ext 实现了一个新的 struct sched_class ext_sched_class。它与其他调度类（CFS/EEVDF、RT、Deadline）在同一框架下工作，通过优先级决定谁先被选择。

当 sched_ext 加载 BPF 调度器后：

  • 所有 SCHED_EXT 策略的任务由 BPF 程序调度
  • BPF 回调决定"哪个任务该在哪个 CPU 上运行多久"
  • 内核提供 scx_bpf_* 帮助函数供 BPF 使用

03 struct sched_ext_ops

这是 BPF 调度器的核心接口——开发者实现以下回调来定义完整的调度策略：

struct sched_ext_ops {
    // 选择 CPU：为任务选择最合适的运行 CPU
    s32 (*select_cpu)(struct task_struct *p,
                      s32 prev_cpu, u64 wake_flags);

    // 入队：任务变为可运行时调用
    void (*enqueue)(struct task_struct *p, u64 enq_flags);

    // 出队：任务不再可运行时调用
    void (*dequeue)(struct task_struct *p, u64 deq_flags);

    // 派发：CPU 需要选一个任务运行时调用
    // 通过 scx_bpf_dispatch() 将任务送入指定 DSQ
    void (*dispatch)(s32 cpu, struct task_struct *prev);

    // 时钟 tick：处理时间片耗尽
    void (*tick)(struct task_struct *p);

    // 运行开始：任务开始执行 CPU 时调用
    void (*running)(struct task_struct *p);

    // 运行停止：任务停止执行时调用
    void (*stopping)(struct task_struct *p,
                     bool runnable);

    // 初始化/退出：task 生命周期钩子
    s32 (*init_task)(struct task_struct *p,
                     struct scx_init_task_args *args);
    void (*exit_task)(struct task_struct *p);

    // CPU hotplug
    s32 (*cpu_online)(s32 cpu);
    s32 (*cpu_offline)(s32 cpu);

    // cgroup
    void (*cgroup_move)(struct task_struct *p,
                        struct cgroup *from, struct cgroup *to);

    // 超线程协同调度
    void (*core_sched)(s32 cpu, struct task_struct *p);

    // 更新空闲状态
    void (*update_idle)(s32 cpu, bool idle);

    // 标志
    u64 flags;  // SCX_OPS_* 标志位
};

6 个核心回调（必须实现或设为空）：select_cpu、enqueue、dequeue、dispatch、tick、running。

04 调度队列 (DSQ)

DSQ = Dispatch Queue，是 sched_ext 的核心调度概念。任务通过 DSQ 在 CPU 之间流动。

DSQ 类型名称说明
本地 DSQ SCX_DSQ_LOCAL 每个 CPU 私有的本地队列。任务只在该 CPU 上执行。
全局 DSQ SCX_DSQ_GLOBAL 系统级 FIFO 队列。任何空闲 CPU 都能消费。
自定义 DSQ 其他 ID BPF 调度器创建的专用队列。可以实现分群或优先级。 
// BPF 可调用以下帮助函数

// 将任务派发到指定 DSQ
void scx_bpf_dispatch(struct task_struct *p,
    u64 dsq_id, u64 slice, u64 flags);
// dsq_id: SCX_DSQ_LOCAL / SCX_DSQ_GLOBAL / 自定义 DSQ ID
// slice: 时间片长度 (ns)
// flags: SCX_ENQ_*

// 从其他 DSQ 消费任务
bool scx_bpf_consume(u64 dsq_id);
// 消费成功 → 任务在当前 CPU 上执行

// 从指定 DSQ 中取一个任务
struct task_struct *scx_bpf_dispatch_from_dsq(
    struct scx_dispatch_q *dsq, u32 dsq_id);

// 唤醒一个 CPU 来消费队列
s32 scx_bpf_kick_cpu(s32 cpu, u64 flags);

典型派发逻辑：

// dispatch 回调中：CPU 需要选一个任务
void BPF_STRUCT_OPS(my_dispatch, s32 cpu, struct task_struct *prev)
{
    // 优先消费全局队列
    if (scx_bpf_consume(SCX_DSQ_GLOBAL))
        return;

    // 或者把当前任务放回本地队列
    if (prev && prev->scx.dsq == SCX_DSQ_LOCAL)
        scx_bpf_dispatch(prev, SCX_DSQ_LOCAL, 0, 0);
}

05 调度生命周期

┌─────────────────────────────────────────┐
│ 任务唤醒                                  │
│  → select_cpu()  选择目标 CPU             │
│  → enqueue()     加入调度队列               │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ CPU 空闲 / tick 触发                      │
│  → dispatch()   从 DSQ 选出下一个任务      │
│  → 上下文切换到选定任务                     │
│  → running()    任务开始运行               │
└────────────────┬────────────────────────┘
                 │
         ┌───────┴───────┐
         ▼               ▼
┌─────────────────┐ ┌─────────────────┐
│ tick 时间片到     │ │ 任务主动睡眠/阻塞 │
│  → tick()       │ │  → dequeue()    │
│  → 再次 dispatch│ │  → stopping()    │
└─────────────────┘ └─────────────────┘
         │               │
         └─── 回到 dispatch ──┘

每一步都有对应的 BPF 回调介入：

阶段回调BPF 可做的事
唤醒选核select_cpu选择最空闲或最合适的 CPU
加入队列enqueue按策略排入 DSQ（如 FIFO/Priority/VRuntime）
选任务dispatch从 DSQ 中筛选最适合的任务
开始运行running统计运行时间，或设置运行环境
时间片tick检查是否该抢占了
出队dequeue任务不再可运行，清理状态

06 BPF 集成

sched_ext 使用 BPF struct_ops 机制对接 BPF 程序：

// 在 BPF C 代码中定义一个调度器
#include "scx_common.bpf.h"

// BPF 调度器 = 实现 struct sched_ext_ops
struct sched_ext_ops my_ops = {
    .select_cpu    = (void *)my_select_cpu,
    .enqueue       = (void *)my_enqueue,
    .dispatch      = (void *)my_dispatch,
    .tick          = (void *)my_tick,
    .init_task     = (void *)my_init_task,
    .flags         = SCX_OPS_KEEP_BUILTIN_IDLE,
};

// 用 SEC 宏标记为 struct_ops
SEC(".struct_ops")
struct sched_ext_ops my_ops;

加载调度器：

# 编译 BPF 调度器
clang -O2 -g -target bpf -c my_scheduler.bpf.c -o my_scheduler.bpf.o

# 加载运行
bpftool struct_ops register my_scheduler.bpf.o

# 查看当前调度器
bpftool struct_ops dump

# 卸载（自动恢复为 CFS）
bpftool struct_ops unregister my_ops

BPF 程序可调用的帮助函数：

帮助函数说明
scx_bpf_dispatch(task, dsq_id, slice, flags)将任务派发到指定 DSQ
scx_bpf_consume(dsq_id)从 DSQ 消费任务到本地 CPU
scx_bpf_kick_cpu(cpu, flags)唤醒 CPU 执行调度
scx_bpf_dsq_nr_queued(dsq_id)查询 DSQ 中排队的任务数
scx_bpf_task_running(cpu, task)检查任务是否在运行
scx_bpf_cpuperf_cap(cpu)读取 CPU 性能上限

07 SCX_OPS 标志

// kernel/sched/ext.h
enum scx_ops_flags {
    // 保持内置空闲策略
    // 不覆盖 CPU 空闲处理，轻量级调度器使用
    SCX_OPS_KEEP_BUILTIN_IDLE  = 1 << 0,

    // 启用 QoS 频率调整
    SCX_OPS_ENABLING_QOS       = 1 << 1,

    // ops 正在初始化
    SCX_OPS_ENABLING           = 1 << 2,

    // ops 正在切换（卸载/加载）
    SCX_OPS_SWITCHING          = 1 << 3,

    // ops 已激活
    SCX_OPS_ENABLED            = 1 << 4,
};

开发者应当在 struct sched_ext_ops.flags 中设置 SCX_OPS_KEEP_BUILTIN_IDLE 除非该 BPF 调度器需要完全控制空闲 CPU 选择。

09 与 CFS/EEVDF 对比

特性CFS / EEVDFsched_ext
调度策略固定在内核中BPF 可编程
修改成本改内核 → 重启加载 BPF 程序
调试printk / ftracebpftool + trace + bpf_printk
部署发布内核更新动态加载/卸载
安全性内核代码 ReviewBPF Verifier 静态验证
性能高度优化略低于 CFS（BPF 开销）
灵活性通用场景优化可为特定负载定制
复杂度调度器 ~3000 行BPF 调度器 ~100-500 行
适用场景:
CFS/EEVDF — 通用服务器、桌面、笔记本（默认就好）。
sched_ext — HPC（自定义负载感知）、实时（自定义抢占策略）、
游戏/媒体（低延迟定制）、调度器研究（快速原型）。
