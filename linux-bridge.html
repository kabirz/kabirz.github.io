01 概述

Linux Bridge 是一个虚拟交换机，将多个网络设备（物理网卡、虚拟网卡、veth、tun/tap）连接在一起，工作原理与物理交换机类似——根据 MAC 地址转发数据帧。它是虚拟机和容器网络的核心基础设施。

核心特性
  • 多端口：连接任意数量的网络设备
  • MAC 学习：自动学习源 MAC 地址
  • 帧转发：根据 MAC 表单播或泛洪
  • STP：生成树协议防环路
  • VLAN：支持 802.1Q VLAN
  • 组播：IGMP snooping
Bridge vs 物理网卡
特性物理网卡Bridge
端口数两端任意多个
转发协议栈处理MAC 地址转发
数据路径协议栈 ↔ 外部端口 ↔ 端口

02 核心数据结构

// net_bridge — Bridge 核心结构
struct net_bridge {
    struct net_device     *dev;          // bridge 虚拟网卡（可配 IP）
    struct list_head     port_list;     // 所有端口链表
    struct rhashtable     fdb;           // MAC 转发表（RHASH）
    unsigned long      flags;         // BR_LEARNING / BR_FLOOD 等
    u32                ageing_time;   // MAC 老化时间
    u8                 stp_state;     // STP 状态
};

// net_bridge_port — Bridge 端口
struct net_bridge_port {
    struct net_device  *dev;       // 端口网络设备
    struct net_bridge  *br;        // 所属 bridge
    u32              flags;       // BR_FLOOD / BR_LEARNING 等
    u16              port_no;     // 端口号
};

// net_bridge_fdb_entry — MAC 转发表条目
struct net_bridge_fdb_entry {
    struct net_bridge_port *dst;   // 目标端口
    u8            addr[ETH_ALEN];  // MAC 地址
    unsigned long updated;     // 老化时间戳
    bool          is_local;    // 本端口学到的
};

03 工作原理

数据帧转发流程

数据帧到达端口 P（eth0 / veth0 / tap0）
        │
        ▼
  提取源 MAC 地址 S ──→ 学习 FDB：(S, P)
        │
        ▼
  提取目的 MAC 地址 D ──→ 查 FDB
        │
        ├── D 已知 ──────────→ 单播转发到目标端口
        │
        ├── D 未知 ──────────→ 泛洪到所有其他端口
        │
        └── 广播/组播 ────────→ 泛洪到所有端口

端口行为详解

当一个网络设备（如 eth0）加入 bridge 后，关键变化：

  • 数据拦截：从外面网络收到的数据包会无条件转发给 bridge，不再直接到协议栈
  • 双向通道：bridge 和端口之间建立双向通道
  • 协议栈隔离：端口的协议栈连接被"劫持"，数据不再直接到协议栈
  • 无差别对待：bridge 不区分物理设备和虚拟设备

04 MAC 学习

// MAC 地址学习算法
// 1. 数据帧从端口 P 进入 bridge
// 2. 提取源 MAC 地址 S
// 3. 在 FDB 中查找 S：
//    - 不存在 → 创建 (S, P, aging_time)
//    - 存在但端口不同 → 更新为 P
//    - 存在且端口相同 → 刷新 aging
// 4. 根据目的 MAC D 转发：
//    - 已知 → 单播到目标端口
//    - 未知 → 泛洪到所有端口
操作函数说明
查找br_fdb_find()按 MAC 查找 FDB 条目
更新br_fdb_update()更新/创建 FDB 条目
删除br_fdb_delete()删除 FDB 条目
老化br_fdb_flush()清除过期条目

05 实战搭建

步骤一：创建 Bridge

$ sudo ip link add name br0 type bridge
$ sudo ip link set br0 up
$ bridge link    # 查看当前状态（空）

步骤二：创建 veth 并连接

$ sudo ip link add veth0 type veth peer name veth1
$ sudo ip addr add 192.168.3.101/24 dev veth0
$ sudo ip addr add 192.168.3.102/24 dev veth1
$ sudo ip link set veth0 up
$ sudo ip link set veth1 up

$ sudo ip link set dev veth0 master br0
$ bridge link
6: veth0 state UP master br0 state forwarding

步骤三：配置 IP

$ sudo ip addr del 192.168.3.101/24 dev veth0
$ sudo ip addr add 192.168.3.101/24 dev br0

$ ping -c 1 -I br0 192.168.3.102
64 bytes from 192.168.3.102: icmp_seq=1 ttl=64 time=0.121 ms  # 成功

步骤四：连接物理网卡

$ sudo ip link set dev eth0 master br0
$ sudo ip addr del 192.168.3.21/24 dev eth0
$ sudo ip route add default via 192.168.3.1

$ ping -c 1 -I br0 192.168.3.1
64 bytes from 192.168.3.1: icmp_seq=1 ttl=64 time=3.91 ms  # 成功

06 ARP 问题

双 ARP 应答

当 br0 和 veth1 都配置了同一网段 IP，外面的 ARP 请求会得到两个 ARP 应答，导致外部设备的 ARP 表不稳定。

# 抓包可见两个应答：
# Reply 192.168.3.102 is-at xx:xx:xx:xx:xx:xx (br0)
# Reply 192.168.3.102 is-at yy:yy:yy:yy:yy:yy (veth1)

解决方案

问题原因解决
veth0 ping veth1 失败veth0 应答被 bridge 拦截给 br0 配 IP，不给 veth0
br0 ping 网关失败br0 没有默认路由删除 eth0 IP，添加默认网关
双 ARP 应答br0 和 veth1 都有 IP只给 br0 配 IP
无线网络不通MAC 地址转换失败统一所有端口 MAC

VM vs Docker 对比

特性虚拟机Docker
连接方式tun/tap → br0veth → br0
数据路径VM → tun/tap → br0（不经 Host）Container → veth → br0 → Host → NAT
性能高（零拷贝）中（经过协议栈+NAT）
MAC 处理VM 独立 MAC统一通过 Host 出去
无线兼容需要 MAC 转换兼容性好

核心区别

虚拟机：VM → tun/tap → br0 → eth0，数据包不经过 Host 协议栈，零拷贝效率高。

Docker：Container → veth → br0 → Host 协议栈 → NAT → eth0，经过协议栈 + NAT，但 MAC 地址问题少。

09 配置命令

# 创建/删除 bridge
$ sudo ip link add name br0 type bridge
$ sudo ip link set br0 up
$ sudo ip link del br0

# 添加/删除端口
$ sudo ip link set dev eth0 master br0
$ sudo ip link set dev eth0 nomaster

# 配置属性
$ sudo ip link set dev br0 type bridge stp_state 1

# 查看信息
$ bridge link
$ brctl show
$ bridge fdb show
