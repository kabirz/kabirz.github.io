01 概述

块设备子系统管理所有按块（sector）寻址的存储设备：硬盘、SSD、NVMe、U盘、虚拟磁盘（loop、ramdisk）。

关键演进：

阶段结构说明
2.4 时代单队列 (legacy)全局锁，多核扩展差
2.6 时代legacy + I/O schedulercfq、deadline、anticipatory
4.x (2014)blk-mq 引入多硬件队列，每 CPU software queue
5.xblk-mq 唯一移除 legacy，全面 blk-mq

IO 路径概览：

┌─────────────────────────────────────────┐
│   用户态：read/write/fsync                │
│   direct I/O / buffered I/O              │
└────────────────┬────────────────────────┘
                 │ VFS
┌────────────────▼────────────────────────┐
│   文件系统层 (ext4/xfs/btrfs)             │
│   - 聚合 page cache → multi-page bio    │
└────────────────┬────────────────────────┘
                 │ submit_bio()
┌────────────────▼────────────────────────┐
│   块层 (Block Layer)                     │
│   ┌─────────────────────────────────┐   │
│   │ bio → blk-mq plugging            │   │
│   │ → request → software queue       │   │
│   │ → hardware queue → driver        │   │
│   └─────────────────────────────────┘   │
└────────────────┬────────────────────────┘
                 │ queue_rq
┌────────────────▼────────────────────────┐
│   设备驱动 (SCSI/NVMe/MMC)               │
│   - DMA 传输                            │
│   - IRQ 完成                            │
└─────────────────────────────────────────┘

// 2. submit_bio 是块层核心入口
// block/blk-core.c
void submit_bio(struct bio *bio)
{
    // 设置 bio 时间戳（统计）
    // 检查 bio 合法性
    if (bio_check_eod(bio))
        goto end_io;
    
    // 分发到 blk-mq
    if (!bio->bi_bdev->bd_disk->fops->submit_bio)
        __submit_bio(bio);                    // 通用路径
    else
        bio->bi_bdev->bd_disk->fops->submit_bio(bio);  // 驱动自管
}
EXPORT_SYMBOL(submit_bio);

// 3. blk-mq 接管
static void __submit_bio(struct bio *bio)
{
    // 找到当前 CPU 的软件队列上下文
    struct blk_plug *plug = current->blk_plug;
    struct request_queue *q = bdev_get_queue(bio->bi_bdev);
    
    // 如果当前任务有 plug（批量缓冲）
    if (plug) {
        // 暂存到 plug list，等 flush
        blk_add_bio_to_plug(plug, bio, ...);
    } else {
        // 直接提交到软件队列
        __submit_bio_noacct_mq(q, bio);
    }
}

// 4. blk-mq 提交
static void __submit_bio_noacct_mq(struct request_queue *q, struct bio *bio)
{
    struct blk_mq_alloc_data data = { .bio = bio, .q = q, ... };
    
    // 分配/复用 request
    rq = blk_mq_get_request(q, bio, &data);
    
    // 把 bio 数据合并到 rq
    blk_mq_bio_to_request(rq, bio, nr_segs);
    
    // 提交到 hctx（硬件队列上下文）
    blk_mq_request_issue_directly(rq, false);
}

05 blk-mq 多队列

blk-mq (Block Multi-Queue) 是现代块层核心。设计目标：多核扩展性，让多 CPU 不再争抢一个全局队列。

// 三层队列：
//   software queue (每 CPU 一个) 
//   → hardware context (hctx)
//   → hardware queue (hw queue, NVMe SQ)

// 三种主流调度器：

// 1. mq-deadline (block/mq-deadline.c)
//   - 两个队列：read / write
//   - 按 sector 排序
//   - dispatch 时按 deadline 选择
//   - 默认调度器（适合 HDD）

// 2. BFQ (Block Flexible Queuing, block/bfq*.c)
//   - 按 task 分队列
//   - 权重分配（cgroup 集成）
//   - 适合桌面系统（音频/视频流畅）

// 3. Kyber (block/kyber-iosched.c)
//   - 自适应节流
//   - read/write 分别限速
//   - 适合 NVMe（防止 read starving write）

调度器的关键决策：

决策说明
合并 (merge)相同扇区的 bio 合并到同一 request
插入 (insert)新 request 加入调度器队列
分发 (dispatch)从调度器队列取出交给驱动
排序 (sort)按扇区排序，减少寻道（HDD）
公平 (fair)各 task/cgroup 轮流获得 IO
选用建议:
mq-deadline：HDD（机械硬盘）默认。简单、按 sector 排序最大化吞吐。
BFQ：桌面环境、低延迟需求。可保证响应性但开销大。
Kyber：NVMe / 高并发场景。限制 in-flight，防止 read starvation。
none：完全不调度（直通 blk-mq）。NVMe/SSD 上常用。
