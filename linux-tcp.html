01 概述

Linux TCP 协议栈是世界上使用最广泛的 TCP 实现之一。内核 net/ipv4/ 目录下包含超过 22602 行代码，实现了 TCP 协议的所有核心功能：可靠传输、流量控制、拥塞控制、重传、零拷贝等。

  • BBR：Google 开发，基于带宽-延迟模型
  • Reno：经典 AIMD（加性增/乘性减）

数据发送流程

  应用调用 send()
      │
      ↓
  tcp_sendmsg() — tcp.c
      │
      ├── 1. 将用户数据拷贝到 skb（发送缓冲区）
      │      如果缓冲区满 → 等待（sk_stream_wait_memory）
      │
      ├── 2. 调用 tcp_write_xmit()
      │      │
      │      ├── 3. 拥塞窗口检查：snd_cwnd ≥ snd_nxt - snd_una ?
      │      │      不满足 → 停止发送（拥塞控制）
      │      │
      │      ├── 4. 滑动窗口检查：接收窗口允许？
      │      │      不满足 → 停止发送（流量控制）
      │      │
      │      ├── 5. tcp_transmit_skb() — 构造 TCP 头
      │      │      ├── 设置序号、确认号、窗口大小
      │      │      ├── 计算校验和
      │      │      └── 添加选项（MSS、SACK、TS、窗口缩放）
      │      │
      │      └── 6. ip_queue_xmit() → IP 层发送
      │
      ↓
  数据包进入 IP 层 → 网卡 → 网线

03 TCP 状态机

11 种 TCP 状态

include/net/tcp_states.h — 11 个状态枚举：

TCP_ESTABLISHED  = 1    连接已建立
TCP_SYN_SENT     = 2    已发送 SYN（主动连接方）
TCP_SYN_RECV     = 3    已收到 SYN（被动连接方）
TCP_FIN_WAIT1    = 4    已发送 FIN（主动关闭方）
TCP_FIN_WAIT2    = 5    已收到 ACK，等待对方 FIN
TCP_TIME_WAIT    = 6    等待 2MSL（主动关闭方最终状态）
TCP_CLOSE        = 7    关闭
TCP_CLOSE_WAIT   = 8    收到 FIN，等待应用关闭（被动关闭方）
TCP_LAST_ACK     = 9    已发送 FIN，等待最后的 ACK
TCP_LISTEN       = 10   监听状态（服务端）
TCP_CLOSING      = 11   双方同时关闭
TCP_NEW_SYN_RECV = 12   新 SYN 到达 LISTEN socket

04 三次握手 / 四次挥手

三次握手

   客户端                              服务端
     │                                   │
     │  ──── SYN (seq=x) ─────────→     │   ① 客户端 → 服务端
     │                                   │      进入 SYN_SENT
     │                                   │      服务端收到 → tcp_v4_recv()
     │                                   │      创建 request_sock → SYN_RECV
     │  ←── SYN+ACK (seq=y, ack=x+1) ── │   ② 服务端 → 客户端
     │                                   │      进入 SYN_RECV
     │  进入 ESTABLISHED                  │
     │  ──── ACK (ack=y+1) ────────→    │   ③ 客户端 → 服务端
     │                                   │      服务端收到 → tcp_v4_reqsk_done()
     │                                   │      创建子 socket → ESTABLISHED
     │       数据传输开始                  │

四次挥手

   主动关闭方                          被动关闭方
     │                                   │
     │  ──── FIN (seq=u) ─────────→     │   ① 主动方发送 FIN
     │  进入 FIN_WAIT1                   │      进入 CLOSE_WAIT
     │                                   │      被动方应用收到 EOF
     │  ←── ACK (ack=u+1) ────────────  │   ② 被动方回复 ACK
     │  进入 FIN_WAIT2                   │      等待应用关闭
     │                                   │
     │      （被动方可能继续发数据）        │
     │                                   │
     │  ←── FIN (seq=w) ──────────────  │   ③ 被动方发送 FIN
     │  进入 TIME_WAIT                   │      进入 LAST_ACK
     │  ──── ACK (ack=w+1) ────────→    │   ④ 主动方回复 ACK
     │  等待 2MSL（60秒）                 │      进入 CLOSED
     │  最终进入 CLOSED                   │

TIME_WAIT 的作用

为什么需要 TIME_WAIT（2MSL）？
① 确保最后一个 ACK 能到达：如果 ACK 丢失，被动方会重发 FIN，主动方需要能回复
② 防止旧连接的延迟包影响新连接：等待 2MSL 确保网络中该连接的所有数据包都已消亡
③ 确保被动方正确关闭：如果 ACK 丢失，被动方不会收到 ACK 而卡在 LAST_ACK

CUBIC vs BBR

CUBIC（基于丢包的拥塞控制）：

  拥塞窗口
  (cwnd)
    │
    │          ╱‾‾‾‾‾‾‾‾‾╲           ← 三次函数增长
    │        ╱              ╲
    │      ╱                  ╲        ← 丢包！cwnd 减半
    │    ╱                      ╲
    │  ╱                          ╲
    │╱                              ╲
    └────────────────────────────────── 时间

  核心：丢包 = 网络拥塞 → 降低发送速率
  缺点：需要丢包才能感知拥塞，高带宽延迟积(BDP)网络效率低


BBR（基于带宽-延迟模型的拥塞控制）：

  发送速率
    │
    │    ╱‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾  ← 达到链路容量
    │  ╱
    │╱                               ← 探测带宽
    └────────────────────────────────── 时间

  核心：测量 RTprop（最小延迟）和 BtlBw（最大带宽）
       发送速率 = min(BtlBw, cwnd/RTT)
  优势：不依赖丢包感知拥塞，高 BDP 网络表现优异
特性CUBICBBR
触发机制丢包带宽 + 延迟测量
cwnd 增长三次函数探测最大带宽
丢包处理cwnd 减半降低发送速率到 BtlBw
高 BDP 网络慢启动阶段浪费带宽快速收敛到最优速率
公平性与 Reno/CUBIC 公平可能抢占其他流带宽

07 重传机制

四种重传方式

机制触发条件说明
超时重传RTO 定时器到期最基本的重传，等待确认超时后重发最早的未确认段
快速重传收到 3 个重复 ACK不等超时，立即重传丢失的段
SACK 重传收到 SACK 信息根据 SACK 确定哪些段已到达，只重传真正丢失的
TLPTail Loss Probe在最后一个包丢失时，避免等待 RTO，主动发送探测包

重传流程

超时重传：

  发送 seq=1,2,3,4,5
       │
       ▼
  RTO 定时器到期（仍有序号未确认）
       │
       ▼
  tcp_retransmit_skb() → 重传最早的未确认段
       │
       ▼
  调整 ssthresh = cwnd / 2
  重新进入拥塞避免

快速重传：

  发送 seq=1,2,3,4,5
  seq=3 丢失
       │
       ▼
  收到重复 ACK（ACK=3）× 3 次
       │
       ▼
  tcp_fastretrans_alert() → 立即重传 seq=3
       │
       ▼
  进入 Recovery 状态
  不需要等待 RTO

RTO 计算

// RTO = SRTT + 4 × RTTVAR
// SRTT  = 平滑 RTT（指数加权移动平均）
// RTTVAR = RTT 方差
//
// SRTT 新值 = (1 - α) × SRTT旧 + α × 测量RTT
//   α = 1/8
//
// RTTVAR 新值 = (1 - β) × RTTVAR旧 + β × |SRTT - 测量RTT|
//   β = 1/4
//
// RTO 下限: 200ms（min_rto）
// RTO 上限: 120s（TCP_RTO_MAX）

08 零拷贝发送

TCP 零拷贝的层次

层次机制说明
MSG_ZEROCOPYsendmsg() 标志用户态内存零拷贝到内核，通过 CQ 通知完成
SF_NOCACHEsendfile() 优化文件→网卡零拷贝（splice/sendfile）
GSO/GRO通用分段/聚合减少协议栈处理次数
传统发送（有拷贝）：

  用户态 buffer  ─── memcpy ──→  内核发送缓冲区  ───→  网卡
                        ↑ 一次拷贝

MSG_ZEROCOPY（零拷贝）：

  用户态 buffer  ─── 引用计数 ──→  内核引用用户态内存  ───→  网卡
                   （无 memcpy）
                         ↑ DMA scatter-gather 直接读用户态页面

  完成通知：通过 errqueue 返回 ZEROCOPY_DONE 消息
