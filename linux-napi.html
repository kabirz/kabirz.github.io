01 概述

NAPI（New API）是 Linux 内核中高性能网络数据包接收的核心机制。它将中断驱动和轮询模式结合，在高流量时自动从"中断模式"切换到"轮询模式"，避免 CPU 被中断淹没。所有现代 Linux 网卡驱动都使用 NAPI。

NAPI 核心思想
  • 低流量：中断驱动（省 CPU）
  • 高流量：轮询模式（高吞吐）
  • 自动切换：基于 NAPI_WEIGHT 自适应
  • 批量处理：一次 poll 处理多个包
  • 与 GRO 集成：合并小包减少协议栈开销

02 传统 vs NAPI

传统模式（每个包一次中断）：

  包 1 到达 → IRQ → softIRQ → 处理包 1
  包 2 到达 → IRQ → softIRQ → 处理包 2
  包 3 到达 → IRQ → softIRQ → 处理包 3
  ...

  问题：高流量时 CPU 忙于中断处理，无法做有用工作
  PPS 100 万 → 100 万次 IRQ/softIRQ → CPU 100% 忙碌

NAPI 模式（中断 + 轮询混合）：

  包 1 到达 → IRQ → 禁用 IRQ + 加入 poll_list
                  → 触发 softIRQ（NET_RX_SOFTIRQ）
                  → net_rx_action()
                  → 轮询处理所有包（weight=64 个）
                  → 处理完 → 重新启用 IRQ
  包 2..64 到达 → DMA 到内存，无中断
                  → net_rx_action 继续轮询
                  → 全部处理完

  优势：64 个包只需 1 次 IRQ + 1 次 softIRQ → CPU 利用率低
特性传统模式NAPI
中断频率每包一次仅首包触发一次
处理方式中断上下文softirq 轮询
批量处理不支持一次 poll 处理多个包
高流量时CPU 被中断淹没自动切换轮询
低流量时响应快自动恢复中断

03 核心数据结构

// include/linux/netdevice.h:381 — NAPI 结构
struct napi_struct {
    unsigned long         state;         // 状态位图（SCHED/MISSED/DISABLE）
    struct list_head      poll_list;     // 等待轮询的 NAPI 列表
    int                   weight;        // ★ 轮询权重（默认 64）
    int                 (*poll)(struct napi_struct *, int); // ★ 驱动的轮询函数
    struct net_device    *dev;          // 关联的网络设备
    struct sk_buff       *skb;          // 当前正在处理的 skb
    struct gro_node       gro;           // GRO 聚合节点
    struct hrtimer        timer;         // 定时器（busy polling）
    struct task_struct    *thread;       // Threaded NAPI 线程
    u32                   napi_id;       // NAPI 唯一 ID
    int                   irq;           // 关联的 IRQ 号
};
// include/linux/netdevice.h:3516 — 每 CPU 软中断数据
struct softnet_data {
    struct list_head      poll_list;      // 本 CPU 的 NAPI 轮询列表
    struct sk_buff_head   process_queue;  // 待处理的 backlog 队列
    unsigned int          processed;      // 已处理包数
    unsigned int          time_squeeze;   // 因时间不足未处理完的次数
    struct Qdisc         *output_queue;  // 发送队列（TC）
    struct sk_buff       *completion_queue; // 完成队列
};

数据结构关系

  每个 CPU 有独立的 softnet_data：

  CPU 0                          CPU 1
  ┌──────────────────┐          ┌──────────────────┐
  │  softnet_data    │          │  softnet_data    │
  │  poll_list:      │          │  poll_list:      │
  │   ├─ napi_struct │(eth0)    │   ├─ napi_struct │(eth1)
  │   └─ napi_struct │(eth2)    │   └─ napi_struct │(eth3)
  │                  │          │                  │
  │  process_queue:  │          │  process_queue:  │
  │   └─ sk_buff...  │          │   └─ sk_buff...  │
  └──────────────────┘          └──────────────────┘

  napi_struct 由驱动创建，绑定到 net_device
  softirq 调度时，遍历本 CPU 的 poll_list 逐个 NAPI poll

04 NAPI 工作流程

1. 网卡收到数据包，触发 IRQ

  NIC → DMA 到内存 → raise IRQ → CPU
        │
        ↓
  napi_schedule() — 驱动调用
        │
        ├── 设置 NAPI_STATE_SCHED（原子位操作）
        ├── 禁用网卡 IRQ：disable_irq(napi->irq)
        ├── napi 加入 per-CPU poll_list
        └── raise NET_RX_SOFTIRQ（触发软中断）

2. 软中断处理：net_rx_action()

  NET_RX_SOFTIRQ → net_rx_action() — net/core/dev.c
        │
        ├── 遍历 softnet_data->poll_list
        │
        ├── 对每个 napi_struct 调用：
        │       work = napi->poll(napi, budget)
        │       │
        │       ├── 驱动从 DMA 缓冲区取出 sk_buff
        │       ├── 调用 napi_gro_receive()（GRO 聚合）
        │       └── 返回已处理的包数
        │
        ├── budget 限制：每个 NAPI 最多处理 budget 个包
        │       总 budget = 300（所有 NAPI 共享）
        │
        ├── 如果 work == 0（处理完毕）
        │       → 从 poll_list 移除
        │       → napi_complete(napi)
        │       → 启用网卡 IRQ：enable_irq(napi->irq)
        │
        └── 如果 work > 0（还有包）
                → 重新调度（reschedule）
                → 继续下一轮 poll

3. 数据包进入协议栈

  napi_gro_receive(napi, skb)
        │
        ├── GRO 聚合：合并小包
        │
        └── __netif_receive_skb(skb)
                → 进入协议栈（IP/TCP/UDP）

05 Threaded NAPI

Threaded NAPI 是 Linux 5.14+ 引入的新特性。NAPI poll 在专用内核线程中执行，而非 softirq 上下文。

特性传统 NAPI（softirq）Threaded NAPI
执行上下文softirq（不可睡眠）内核线程（可睡眠）
优先级高（所有 softirq 共享）SCHED_FIFO 实时优先级
CPU 亲和性运行在触发 IRQ 的 CPU可绑定到任意 CPU
适用场景通用高吞吐 + 低延迟
启用 Threaded NAPI
echo 1 > /sys/class/net/eth0/threaded
ip link set eth0 threaded on

06 GRO 聚合

GRO（Generic Receive Offload）在 NAPI poll 中执行：

  无 GRO：
  包 1 → skb → 协议栈处理
  包 2 → skb → 协议栈处理
  包 3 → skb → 协议栈处理
  → 3 次协议栈开销

  有 GRO：
  包 1 → skb ─┐
  包 2 → skb ─┤→ GRO 合并 → 大 skb → 协议栈处理
  包 3 → skb ─┘
  → 1 次协议栈开销（减少 66%）

  GRO 合并条件：
  - 同一 (src, dst, port) 四元组
  - 连续的序号（TCP）
  - 相同的协议类型
  - 不超过 MSS（最大段大小）
GRO vs GSO
GRO（接收）：合并小包 → 减少协议栈处理次数 → 提升接收性能
GSO（发送）：延迟分段 → 减少协议栈处理次数 → 提升发送性能
两者对称：GRO 在接收路径合并，GSO 在发送路径分段。

07 RPS/RFS

RPS（Receive Packet Steering）— 软件层面的接收多队列：

  单队列网卡 + RPS：
  ┌──────────┐
  │   NIC    │ → 单队列 → CPU 0（传统：所有包在 CPU 0 处理）
  └──────────┘
  ┌──────────┐
  │   NIC    │ → RPS → 按流哈希分发 → CPU 0 / CPU 1 / CPU 2 / CPU 3
  └──────────┘
  优势：即使网卡只有单队列，也能多 CPU 并行处理

RFS（Receive Flow Steering）— 将包送到处理该流的 CPU：

  传统 RPS：按哈希分发，可能送到错误 CPU（导致 cache miss）
  RFS：     追踪应用在哪个 CPU 处理，将包送到同一 CPU
            → 应用 cache 命中率提升 → 性能更好

# 配置 RPS（为 eth0 启用 4 个 CPU）
$ echo "f" > /sys/class/net/eth0/queues/rx-0/rps_cpus
# f = 0xf = 二进制 1111 = CPU 0-3

# 配置 RFS（全局）
$ sysctl -w net.core.rps_sock_flow_entries=32768
$ echo 32768 > /sys/class/net/eth0/queues/rx-0/rps_flow_cnt
