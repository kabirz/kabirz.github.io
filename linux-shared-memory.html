01 概述

共享内存是 Linux 中性能最高的 IPC 机制。多个进程将同一块物理内存映射到各自的虚拟地址空间，读写时无需内核介入（无 memcpy、无上下文切换）。数据库（PostgreSQL、Redis）和高性能应用大量使用共享内存。

共享内存核心特性
  • 零拷贝：进程直接读写共享页面
  • 最高性能：无系统调用开销（写入后）
  • 多进程共享：任意数量进程映射
  • 需要同步：无内建同步，需配合信号量/锁
  • 按需分配：通过 page fault 懒分配

02 三种共享内存

1. POSIX 共享内存（推荐）：

  fd = shm_open("/myshm", O_CREAT|O_RDWR, 0666);
  ftruncate(fd, SIZE);
  ptr = mmap(NULL, SIZE, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
  // 多进程 shm_open 同一名称 → 共享同一页面

  优势：接口简洁、文件系统可见、可持久化

2. SysV 共享内存（传统）：

  key = ftok("/tmp", 'a');
  shmid = shmget(key, SIZE, IPC_CREAT|0666);
  ptr = shmat(shmid, NULL, 0);
  // 多进程通过 shmid 映射

  优势：兼容旧代码
  劣势：接口复杂、需要手动清理

3. mmap + MAP_SHARED + 匿名映射：

  ptr = mmap(NULL, SIZE, PROT_READ|PROT_WRITE,
             MAP_SHARED|MAP_ANONYMOUS, -1, 0);
  fork() → 子进程继承映射 → 共享内存

  优势：最简单、无需文件系统
  劣势：仅限父子进程

03 核心数据结构

// ipc/shm.c — SysV 共享内存段
struct shmid_kernel {
    struct  kern_ipc_perm  shm_perm;
    struct file          *shm_file;    // ★ 关联的 tmpfs 文件
    unsigned long        shm_nattch;   // 当前附着的进程数
    unsigned long        shm_segsz;    // 段大小
    time64_t             shm_atim;     // 最后 attach 时间
    time64_t             shm_dtim;     // 最后 detach 时间
    time64_t             shm_ctim;     // 创建时间
    struct pid           *shm_cprid;   // 创建者 PID
    struct pid           *shm_lprid;   // 最后操作者 PID
};

底层机制

  POSIX shm 和 SysV shm 底层都使用 shmem（tmpfs）：

  进程 A                        进程 B
  mmap(fd, MAP_SHARED)          mmap(fd, MAP_SHARED)
       │                            │
       ↓                            ↓
  ┌──────────┐                ┌──────────┐
  │ VMA      │                │ VMA      │
  │ vm_file ─┼──→ 同一个 tmpfs inode ←──┼── vm_file │
  └────┬─────┘                └────┬─────┘
       │                           │
       ↓                           ↓
  ┌──────────────────────────────────────────┐
  │     共享物理页面（tmpfs / shmem）          │
  │     address_space.i_pages → page 数组     │
  └──────────────────────────────────────────┘

  tmpfs 是基于内存的文件系统：
  - 不写磁盘（除非显式 swap）
  - 页面按需分配（首次访问时 fault）
  - 进程退出后自动释放
  - /dev/shm 是 tmpfs 的挂载点

04 创建与映射流程

POSIX 共享内存完整流程：

  进程 A（服务端）：
  1. shm_open("/myshm", O_CREAT|O_RDWR, 0666)
     → 创建 /dev/shm/myshm（tmpfs 文件）
     → 返回 fd
  2. ftruncate(fd, 4096)
     → 设置文件大小
  3. ptr = mmap(NULL, 4096, PROT_READ|PROT_WRITE,
               MAP_SHARED, fd, 0)
     → 建立虚拟地址到 tmpfs 页面的映射
  4. 初始化共享数据结构（锁、队列等）
  5. fork() 或通知其他进程

  进程 B（客户端）：
  1. shm_open("/myshm", O_RDWR, 0)
     → 打开已存在的共享内存
     → 返回 fd
  2. fstat(fd, &st)
     → 获取大小
  3. ptr = mmap(NULL, st.st_size, PROT_READ|PROT_WRITE,
               MAP_SHARED, fd, 0)
     → 映射到自己的地址空间
  4. 访问共享数据（直接读写内存，无系统调用！）

  进程 A：
  6. munmap(ptr, 4096)
  7. shm_unlink("/myshm")  // 删除共享内存对象
  8. close(fd)

// 代码示例
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>

// 服务端
int fd = shm_open("/myshm", O_CREAT|O_RDWR, 0666);
ftruncate(fd, sizeof(struct shared_data));
struct shared_data *data = mmap(NULL, sizeof(*data),
    PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
pthread_mutex_init(&data->mutex, NULL);  // 初始化锁

// 客户端
int fd = shm_open("/myshm", O_RDWR, 0);
struct shared_data *data = mmap(NULL, sizeof(*data),
    PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
pthread_mutex_lock(&data->mutex);   // 加锁访问
data->counter++;                      // 直接操作共享内存
pthread_mutex_unlock(&data->mutex);

05 同步机制

共享内存无内建同步
两个进程同时写入同一地址会导致数据竞争。必须使用外部同步机制：
同步方式开销适用场景
pthread_mutex低简单互斥访问
POSIX 信号量低生产者-消费者
spinlock极低极短临界区
无锁队列最低高性能队列（需 CAS）
futex低混合模式（用户态快路径+内核态慢路径）
读写锁低读多写少
典型共享内存 + 信号量模式（生产者-消费者）：

  生产者进程                          消费者进程
  │                                    │
  ├── sem_wait(&empty)                │  // 等待空槽位
  ├── sem_wait(&mutex)                │  // 加锁
  ├── ring[write_idx] = data          │  // 写入共享内存
  ├── write_idx = (write_idx+1) % N   │
  ├── sem_post(&mutex)                │  // 解锁
  ├── sem_post(&full)                 │  // 通知消费者
  │                                    │
  │                                    ├── sem_wait(&full)    // 等待数据
  │                                    ├── sem_wait(&mutex)   // 加锁
  │                                    ├── data = ring[read_idx] // 读共享内存
  │                                    ├── read_idx = (read_idx+1) % N
  │                                    ├── sem_post(&mutex)   // 解锁
  │                                    ├── sem_post(&empty)   // 通知生产者

06 IPC 性能对比

IPC 机制带宽延迟数据拷贝适用场景
共享内存极高极低0 次数据库、高频交易
UNIX Domain Socket高低2 次Docker、API
Pipe高低2 次shell 管道
消息队列中中2 次异步任务
Socket (TCP)高高3+ 次跨主机
Signal无数据中0 次异步通知
为什么共享内存最快？
① 写入后无需系统调用——直接修改内存
② 读取方无需 read()——直接访问映射的虚拟地址
③ 唯一开销是 page fault（首次访问时）和同步原语
