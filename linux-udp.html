01 概述

UDP（User Datagram Protocol）是 Linux 中最简单的传输层协议。与 TCP 不同，UDP 是无连接、不可靠的——没有握手、没有重传、没有拥塞控制。它的优势在于低延迟、低开销，适合实时应用和高性能场景。

UDP 特性
  • 无连接：不需要建立/释放连接
  • 不可靠：不保证交付、不保证顺序
  • 无拥塞控制：发送速率不受限
  • 头部小：仅 8 字节（TCP 最少 20 字节）
  • 支持广播/多播：TCP 只支持单播
  • 零拷贝支持：MSG_ZEROCOPY / GSO

03 核心数据结构

// include/linux/udp.h:53 — UDP socket 结构
struct udp_sock {
    // 继承自 inet_sock → sock
    struct inet_sock  inet;

    unsigned long    udp_flags;
    int              pending;       // 待发送帧数（UDP CORK）
    __u8             encap_type;    // UDP 封装类型（UDP tunnel）
    __u16            len;           // 待发送帧总长度
    __u16            gso_size;      // GSO 段大小

    // UDP tunnel 封装回调（VXLAN/GRE 等）
    int (*encap_rcv)(struct sock *sk, struct sk_buff *skb);

    // GRO 回调（接收路径聚合）
    struct sk_buff *(*gro_receive)(...);
    int (*gro_complete)(...);

    // 接收队列（高性能优化）
    struct sk_buff_head reader_queue;
};
// include/net/udp.h:97 — UDP 端口哈希表
struct udp_table {
    struct udp_hslot       *hash;    // 一级哈希：按本地端口
    struct udp_hslot_main  *hash2;   // 二级哈希：(端口, 地址)
    struct udp_hslot       *hash4;   // 三级哈希：(端口, 地址, 对端端口, 对端地址)
    unsigned int           mask;    // 哈希表大小 - 1
};

// 三级哈希查找策略：
// hash  → 最快：仅按端口查找（通配地址的 socket）
// hash2 → 精确：(端口 + 本地地址) → 支持多网卡绑定
// hash4 → 精确：(四元组) → 已连接 UDP（connect() 后）

继承层次

04 发送流程

应用调用 sendto(fd, buf, len, 0, dest_addr, addrlen)
        │
        ↓
  udp_sendmsg() — net/ipv4/udp.c
        │
        ├── 1. 路由查找
        │       ip_route_output_flow() → 确定出接口和源 IP
        │       如果 socket 已 connect()，使用缓存的路由
        │
        ├── 2. 构造 skb
        │       __ip_append_data() → 分配 skb + UDP 头
        │       如果 UDP_CORK 或 MSG_MORE → 缓存不立即发送
        │
        ├── 3. 填充 UDP 头
        │       源端口、目的端口、长度、校验和
        │
        ├── 4. GSO 分段检查
        │       如果启用 GSO → 构造大段（延迟到驱动层分段）
        │       udp_send_skb() → ip_send_skb()
        │
        ↓
  进入 IP 层 → netdevice → 网卡

UDP_CORK 与 MSG_MORE

UDP_CORK
// 设置后，数据不会立即发送
// 而是缓存在发送队列中
setsockopt(fd, SOL_UDP, UDP_CORK, &val, 1);
send(fd, buf1, len1, 0);  // 不发送
send(fd, buf2, len2, 0);  // 不发送
setsockopt(fd, SOL_UDP, UDP_CORK, &val, 0);
// uncork → 合并为一个包发送
MSG_MORE
// 每次 send 都带 MSG_MORE 标志
// 效果等同于 UDP_CORK
send(fd, buf1, len1, MSG_MORE);  // 不发送
send(fd, buf2, len2, MSG_MORE);  // 不发送
send(fd, buf3, len3, 0);         // 发送（合并）

05 接收流程

  网卡收到 UDP 数据包
        │
        ↓
  IP 层 → ip_rcv() → routing → NF_INET_LOCAL_IN
        │
        ↓
  udp_rcv() — net/ipv4/udp.c
        │
        ├── 1. 校验和验证
        │       如果校验和错误 → 丢弃
        │
        ├── 2. 查找匹配的 socket
        │       __udp4_lib_lookup_skb()
        │       → __udp4_lib_lookup() → udp_lib_lookup()
        │
        │       查找顺序（三级哈希）：
        │       hash4 → 已连接 socket（四元组精确匹配）
        │       hash2 → (端口 + 地址) 匹配
        │       hash  → 仅端口匹配（通配地址）
        │
        ├── 3. 找不到 socket → ICMP port unreachable
        │
        ├── 4. 入队
        │       udp_queue_rcv_skb()
        │       → __udp_enqueue_schedule_skb()
        │       → __skb_queue_tail(&sk->sk_receive_queue)
        │
        ├── 5. 通知应用
        │       sk->sk_data_ready(sk) → 唤醒等待的 recvfrom()
        │
        ↓
  应用 recvfrom() 读取数据
UDP 接收缓冲区满
UDP 没有流量控制。当 socket 接收缓冲区满时，新到达的包直接丢弃。不会通知发送方（不像 TCP 的零窗口通知）。
netstat -su 中的 RcvbufErrors 就是缓冲区满导致的丢包计数。

06 端口查找与哈希表

三级哈希表设计

hash — 一级：仅端口

  通配地址的 socket（bind 0.0.0.0:53）
  → 仅按端口号哈希
  → 最快，适用于大多数 server socket

hash2 — 二级：(端口 + 本地地址)

  绑定特定地址的 socket（bind 192.168.1.1:53）
  → 按 (端口, 地址) 哈希
  → 支持多网卡绑定

hash4 — 三级：(端口, 地址, 对端端口, 对端地址)

  已 connect() 的 UDP socket
  → 四元组精确匹配
  → 支持已连接 UDP 的高效查找

查找策略：
  1. 先查 hash4（最快，精确匹配）
  2. 再查 hash2（次快，端口+地址）
  3. 最后查 hash（最慢，仅端口）
为什么需要三级哈希？
① 性能：已连接 UDP 用 hash4 O(1) 精确匹配，避免遍历
② 兼容：通配地址的 socket 用 hash 仍然高效
③ 多宿主：同一端口绑定不同地址时用 hash2 区分

07 GSO/GRO/硬件卸载

发送：GSO（Generic Segmentation Offload）

  应用发送 64KB 数据
        │
        ↓
  UDP 层构造一个超大 skb（64KB，gso_size=1472）
        │
        ↓
  IP 层传递给网卡驱动
        │
        ↓
  网卡驱动检查：支持 GSO？
    ├── 是 → 网卡硬件分段为多个 MTU 大小的包
    └── 否 → __udp_gso_segment() 内核软件分段

接收：GRO（Generic Receive Offload）

  网卡收到多个小 UDP 包
        │
        ↓
  NAPI poll → napi_gro_receive()
        │
        ↓
  GRO 尝试合并：同一 (src, dst, port) 的连续 UDP 包
        │
        ├── 合并成功 → 一个大 skb 交给协议栈
        └── 合并失败 → 逐个处理
机制方向作用性能提升
GSO发送延迟分段，减少协议栈处理~30%
GRO接收合并小包，减少协议栈处理~40%
TSO发送网卡硬件分段（TCP 专用）~50%
LRO接收网卡硬件合并（已废弃）-

08 应用场景

DNS 查询

DNS 使用 UDP 53 端口。请求小（几十字节），响应也小，延迟敏感。UDP 的无连接特性完美匹配——不需要 TCP 的三次握手开销。

视频流 / 音频流

实时音视频（WebRTC、RTP）使用 UDP。丢失几个包不影响播放，但延迟是致命的。TCP 的重传机制反而会造成卡顿。

游戏服务器

在线游戏使用 UDP 传输位置/状态更新。每秒几十个包，丢失就丢弃（用新值覆盖），不需要重传。

VXLAN 隧道

容器 overlay 网络使用 VXLAN（UDP 4789）。每个容器数据包封装在 UDP 中跨主机传输。udp_sock 的 encap_rcv 回调处理解封装。
