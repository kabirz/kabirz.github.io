01 概述

Win32 API 是微软 Windows 平台最底层的用户态编程接口，自 Windows NT 时代沿用至今。它定义了窗口、消息、GDI 绘图、进程线程、文件 I/O、注册表等一切服务的 C 语言调用约定。所有上层 GUI 框架（MFC、WinForms、WPF、WinUI 3、Qt/Win32 移植）最终都在底层调用 Win32——它是 Windows GUI 栈的基石。

Win32 GUI 的本质是事件驱动的消息模型：操作系统捕获用户输入（鼠标、键盘）与系统事件（重绘、定时器），把它们封装成消息（MSG）投递到线程的消息队列；应用程序在消息循环中取出消息，分发给对应窗口的窗口过程（WndProc）处理。这种"操作系统 → 队列 → 窗口过程"的链路是 Win32 GUI 的灵魂。

核心特性
  • 纯 C 接口：无运行时、无 GC，可被任意语言调用
  • 句柄语义：HWND/HDC/HINSTANCE 等内核态对象引用
  • 消息驱动：事件→消息→队列→窗口过程
  • GDI 绘图：设备上下文 + 笔/刷/字体对象
  • 资源系统：.rc 脚本编译进 exe 二进制
  • Unicode 优先：Windows NT 内核原生 UTF-16

句柄（Handle）的本质：句柄是进程内稳定的内核对象索引，不是指针。HWND、HDC、HINSTANCE、HMENU、HGDIOBJ 都是不同的句柄类型，背后对应内核或子系统对象。同一个值在不同进程间无意义，跨进程访问需用 HWND_BROADCAST 或复制句柄机制。

02 核心概念

Win32 GUI 的四个基石：窗口类定义窗口的"模板"，窗口是类的实例，消息驱动一切行为，句柄是引用对象的统一方式。

窗口类 WNDCLASSEX

窗口类是创建窗口的蓝图，描述了该类窗口共有的属性：图标、光标、背景画刷、窗口过程地址等。注册后所有据此创建的窗口共享同一 WndProc。

typedef struct tagWNDCLASSEXW {
    UINT        cbSize;           // sizeof(WNDCLASSEX)，必填
    UINT        style;            // 类风格：CS_HREDRAW | CS_VREDRAW
    WNDPROC     lpfnWndProc;      // ★窗口过程函数指针（核心）
    int         cbClsExtra;       // 类附加内存（通常 0）
    int         cbWndExtra;       // 每个窗口附加内存（存自定义数据）
    HINSTANCE   hInstance;        // ★注册该类的模块实例
    HICON       hIcon;            // 大图标（Alt+Tab）
    HCURSOR     hCursor;          // 鼠标光标
    HBRUSH      hbrBackground;    // ★窗口背景画刷
    LPCWSTR     lpszMenuName;     // 默认菜单资源名
    LPCWSTR     lpszClassName;    // ★类名（创建窗口时用）
    HICON       hIconSm;          // 小图标（任务栏）
} WNDCLASSEXW, *PWNDCLASSEXW;

窗口句柄 HWND

HWND 是窗口在用户态（User32 子系统）的唯一标识。每个可见/不可见的窗口、控件、对话框都是 HWND。常用操作：ShowWindow 显示、SetWindowText 改标题、GetClientRect 取客户区、DestroyWindow 销毁。

消息结构 MSG

typedef struct tagMSG {
    HWND    hwnd;      // 目标窗口
    UINT    message;   // ★消息号：WM_PAINT / WM_COMMAND 等
    WPARAM  wParam;    // 参数1：消息相关（如按键虚拟码）
    LPARAM  lParam;    // 参数2：消息相关（如鼠标坐标 packed）
    DWORD   time;      // 投递时刻
    POINT   pt;        // 投递时鼠标屏幕坐标
} MSG, *PMSG;

实例句柄 HINSTANCE

HINSTANCE 标识加载到进程中的模块（exe 自身或 DLL）。在 32/64 位系统中，HINSTANCE 与 HMODULE 实为同义。注册窗口类、加载资源（LoadIcon、LoadMenu）都需要本模块实例作为参数，以让系统知道从哪个模块的资源段查找。

易混点：HINSTANCE 是模块级、HWND 是窗口级。一个进程通常只有一个 HINSTANCE（exe 自身），但可能有成百上千个 HWND。资源加载时务必传入资源所在模块的 HINSTANCE——比如 DLL 中加载自己资源时不能用 exe 的实例。

03 程序骨架

Win32 GUI 程序的入口是 WinMain（而非 main），它由 C 运行时启动代码调用。一个完整的 GUI 程序只需五步：注册窗口类 → 创建窗口 → 显示 → 消息循环 → 窗口过程处理消息。

#include <windows.h>

// 窗口过程：处理发往该窗口的所有消息
LRESULT CALLBACK WndProc(HWND hWnd, UINT msg,
                          WPARAM wParam, LPARAM lParam)
{
    switch (msg) {
    case WM_CREATE:
        // 窗口创建时一次性初始化
        return 0;

    case WM_PAINT: {
        PAINTSTRUCT ps;
        HDC hdc = BeginPaint(hWnd, &ps);
        TextOut(hdc, 10, 10, L"Hello Win32", 11);
        EndPaint(hWnd, &ps);
        return 0;
    }

    case WM_DESTROY:
        PostQuitMessage(0);     // 投递 WM_QUIT，让 GetMessage 返回 0
        return 0;

    default:
        return DefWindowProc(hWnd, msg, wParam, lParam);
    }
}

int WINAPI wWinMain(HINSTANCE hInst, HINSTANCE hPrev,
                      LPWSTR lpCmdLine, int nCmdShow)
{
    // 1. 注册窗口类
    WNDCLASSEX wc = { sizeof(wc) };
    wc.lpfnWndProc    = WndProc;
    wc.hInstance      = hInst;
    wc.lpszClassName  = L"MyAppClass";
    wc.hCursor        = LoadCursor(NULL, IDC_ARROW);
    wc.hbrBackground  = (HBRUSH)(COLOR_WINDOW + 1);
    wc.hIcon          = LoadIcon(NULL, IDI_APPLICATION);
    RegisterClassEx(&wc);

    // 2. 创建窗口
    HWND hWnd = CreateWindowEx(
        0, wc.lpszClassName, L"My First Win32 App",
        WS_OVERLAPPEDWINDOW,
        CW_USEDEFAULT, CW_USEDEFAULT, 800, 600,
        NULL, NULL, hInst, NULL);

    // 3. 显示并刷新
    ShowWindow(hWnd, nCmdShow);
    UpdateWindow(hWnd);

    // 4. 消息循环
    MSG msg;
    while (GetMessage(&msg, NULL, 0, 0) > 0) {
        TranslateMessage(&msg);   // 把按键消息翻译为 WM_CHAR
        DispatchMessage(&msg);    // ★调用目标窗口的 WndProc
    }
    return (int) msg.wParam;       // 退出码
}
WinMain 四参数
  • hInstance：exe 模块实例
  • hPrevInstance：Win16 遗留，恒为 NULL
  • lpCmdLine：命令行（不含程序名）
  • nCmdShow：建议的显示方式
CreateWindowEx 关键参数
  • dwExStyle：扩展样式（WS_EX_APPWINDOW 等）
  • lpClassName：已注册的类名
  • dwStyle：WS_OVERLAPPEDWINDOW 等
  • lpParam：传给 WM_CREATE 的 LPARAM
GetMessage 的三种返回值：> 0 表示取到正常消息；0 表示取到 WM_QUIT（退出循环）；-1 表示出错（如窗口已销毁但队列仍被查询）。生产代码应显式判断 -1。

04 消息机制详解

消息是 Win32 GUI 的血液。系统产生消息、消息队列暂存它们、消息循环取出并分发、窗口过程消费它们——这条链路串起整个 GUI 的运行时。

常见消息类型

消息触发时机wParam / lParam 含义
WM_CREATE窗口创建时（CreateWindow 内部）lParam 指向 CREATESTRUCT
WM_DESTROY窗口销毁前无
WM_PAINT窗口客户区需要重绘无（用 BeginPaint 取无效区）
WM_COMMAND菜单/快捷键/控件通知LOWORD=ID，HIWORD=通知码
WM_LBUTTONDOWN鼠标左键按下wParam=按键状态，lParam=坐标
WM_KEYDOWN非系统键按下wParam=虚拟键码
WM_CHARTranslateMessage 翻译后wParam=字符（WCHAR）
WM_SIZE窗口尺寸变化wParam=尺寸类型，lParam=新客户区
WM_QUITPostQuitMessage 投递wParam=退出码
WM_TIMERSetTimer 定时到期wParam=定时器 ID

PostMessage vs SendMessage

这是理解 Win32 消息系统的关键分水岭。两者看似都是"发消息"，行为却截然不同。

维度PostMessage（投递）SendMessage（发送）
语义异步同步
路径进队列，由 GetMessage 取出直接调用目标 WndProc
返回时机投递完成立即返回（不等处理）等 WndProc 返回后才返回
返回值BOOL（是否投递成功）LRESULT（WndProc 的返回）
跨线程不阻塞，安全可能阻塞，有重入风险
典型用例后台线程通知 UI查询控件状态（如获取文本）

消息流向

// 一次鼠标点击的完整链路：
用户鼠标点击
   ↓
输入子系统（驱动 + 系统服务）
   ↓
User32 把事件转为 WM_LBUTTONDOWN
   ↓
PostMessage 投递到拥有该窗口的线程队列
   ↓
线程消息循环 GetMessage 取出
   ↓
TranslateMessage（键盘消息才会产生 WM_CHAR）
   ↓
DispatchMessage 查表找到目标窗口的 WndProc
   ↓
你的 WndProc switch 处理，不处理的交给 DefWindowProc

WM_PAINT 处理

WM_PAINT 是最特殊的消息——它只在窗口存在"无效区域"时由系统投递，且合并多个无效区以减少重绘。处理它必须用 BeginPaint/EndPaint 对，这对函数会清除无效区，否则会陷入死循环。

case WM_PAINT: {
    PAINTSTRUCT ps;
    HDC hdc = BeginPaint(hWnd, &ps);   // 标记无效区为有效
    // 只画 ps.rcPaint 范围即可（裁剪已由系统设置）
    Rectangle(hdc, 0, 0, 100, 100);
    EndPaint(hWnd, &ps);                  // 释放 DC
    return 0;
}

// 主动触发重绘（不要直接发 WM_PAINT）：
InvalidateRect(hWnd, NULL, TRUE);   // 整客户区无效，背景擦除
UpdateWindow(hWnd);               // 立刻发 WM_PAINT（同步）

WM_COMMAND 命令来源

三种来源
  • 菜单：用户点击菜单项
  • 快捷键：按 TranslateAccelerator 匹配的键
  • 控件：子控件通知父窗口（按钮点击等）
参数解码
  • LOWORD(wParam)：菜单/控件 ID
  • HIWORD(wParam)：通知码（控件用）
  • lParam：控件 HWND（菜单为 NULL）

05 GDI 绘图

GDI（Graphics Device Interface）是 Win32 的2D 图形子系统。它以设备上下文（DC）为中心——DC 抽象了绘制目标（屏幕、打印机、内存位图），让相同的绘图代码不修改即可输出到不同介质。

设备上下文 HDC

HDC 封装了绘制目标 + 当前绘图属性（笔、刷、字体、映射模式、裁剪区、文本颜色等）。所有绘图原语都接受 HDC 参数。获取 HDC 的三种方式各有用途：

API场景特点
BeginPaint响应 WM_PAINT仅画无效区，自动清无效标记
GetDC / ReleaseDC响应非 WM_PAINT 的绘制整个客户区，需手动释放
CreateCompatibleDC双缓冲、内存位图内存 DC，需选入位图才能画

绘图原语

// 文本
TextOut(hdc, 10, 10, L"Hello", 5);
DrawText(hdc, L"多行\n文本", -1, &rc, DT_CENTER | DT_WORDBREAK);

// 填充图形
Rectangle(hdc, 10, 10, 200, 120);   // 边框=当前画笔，填充=当前画刷
Ellipse(hdc, 10, 10, 200, 120);
RoundRect(hdc, 10, 10, 200, 120, 20, 20);

// 线段
MoveToEx(hdc, 10, 10, NULL);          // 移动画笔位置
LineTo(hdc, 200, 120);             // 画到目标点

// 像素（慢，慎用）
SetPixel(hdc, 50, 50, RGB(255, 0, 0));

GDI 对象与选入流程

GDI 笔、刷、字体、位图、区域都是GDI 对象，有自己的句柄（HPEN/HBRUSH/HFONT/HBITMAP/HRGN，统称 HGDIOBJ）。把对象"画到 DC"的流程是四步黄金法则：

// 1. 创建对象
HPEN hPen = CreatePen(PS_SOLID, 2, RGB(255, 0, 0));
HBRUSH hBrush = CreateSolidBrush(RGB(0, 200, 0));

// 2. 选入 DC，保存旧对象（必须！返回值是被替换的旧对象）
HPEN   hOldPen   = (HPEN)SelectObject(hdc, hPen);
HBRUSH hOldBrush = (HBRUSH)SelectObject(hdc, hBrush);

// 3. 绘图（使用新对象）
Rectangle(hdc, 10, 10, 200, 120);

// 4. 恢复旧对象，再删除新对象（顺序不可反）
SelectObject(hdc, hOldPen);
SelectObject(hdc, hOldBrush);
DeleteObject(hPen);
DeleteObject(hBrush);
资源泄漏陷阱：DeleteObject 无法删除仍被 DC 选中的对象（会失败但返回 TRUE 误导）。每次新选入必须配对一次恢复旧对象——否则在长时间运行的程序中 GDI 句柄会迅速耗尽（每进程默认 10000 个上限）。

双缓冲避免闪烁

直接画到屏幕 DC 时，每条绘图指令都会立即可见，复杂的绘制会出现明显的闪烁。解决方案：在内存 DC 上画完整帧，再一次性 BitBlt 到屏幕。

case WM_PAINT: {
    PAINTSTRUCT ps;
    HDC hdc = BeginPaint(hWnd, &ps);

    RECT rc; GetClientRect(hWnd, &rc);
    int w = rc.right, h = rc.bottom;

    // 1. 创建与屏幕兼容的内存 DC 与位图
    HDC hdcMem = CreateCompatibleDC(hdc);
    HBITMAP hBmp = CreateCompatibleBitmap(hdc, w, h);
    HBITMAP hOld = (HBITMAP)SelectObject(hdcMem, hBmp);

    // 2. 在内存 DC 上完整绘制（这里随便画点东西）
    HBRUSH hb = CreateSolidBrush(RGB(255, 255, 255));
    FillRect(hdcMem, &rc, hb);
    DeleteObject(hb);
    Ellipse(hdcMem, 0, 0, w, h);

    // 3. 一次性拷贝到屏幕
    BitBlt(hdc, 0, 0, w, h, hdcMem, 0, 0, SRCCOPY);

    // 4. 清理
    SelectObject(hdcMem, hOld);
    DeleteObject(hBmp);
    DeleteDC(hdcMem);

    EndPaint(hWnd, &ps);
    return 0;
}

06 标准控件

Windows 公共控件（Common Controls）由 ComCtl32.dll 提供，包括按钮、编辑框、列表、树、进度条、状态栏等。它们本质上也是窗口——创建控件就是创建子窗口，只是窗口类是系统预定义的（如 "button"、"edit"、"static"）。

初始化控件

#include <commctrl.h>
#pragma comment(lib, "comctl32.lib")

// 在 WinMain 开头显式初始化（注册公共控件类）
INITCOMMONCONTROLSEX icc = { sizeof(icc) };
icc.dwICC = ICC_WIN95_CLASSES | ICC_BAR_CLASSES;
InitCommonControlsEx(&icc);

创建控件

// 创建一个按钮（子窗口）
HWND hBtn = CreateWindow(
    L"button",                        // ★系统预定义类名
    L"Click Me",
    WS_CHILD | WS_VISIBLE | BS_PUSHBUTTON,
    10, 10, 120, 32,
    hWndParent,                       // ★父窗口
    (HMENU)IDC_MY_BUTTON,           // ★用 HMENU 槽位传控件 ID
    hInstance, NULL);

// 创建编辑框
HWND hEdit = CreateWindow(L"edit", L"",
    WS_CHILD | WS_VISIBLE | WS_BORDER | ES_AUTOHSCROLL,
    10, 50, 200, 24,
    hWndParent, (HMENU)IDC_MY_EDIT, hInstance, NULL);

// 创建静态文本（标签）
HWND hLabel = CreateWindow(L"static", L"姓名：",
    WS_CHILD | WS_VISIBLE, 10, 80, 80, 20,
    hWndParent, (HMENU)IDC_STATIC, hInstance, NULL);

控件通知父窗口

控件事件通过 WM_COMMAND（旧式）或 WM_NOTIFY（新式，带结构体指针）通知父窗口。父窗口在 WndProc 中解码 wParam/lParam 路由到对应处理。

case WM_COMMAND: {
    WORD id     = LOWORD(wParam);   // 控件 ID
    WORD code   = HIWORD(wParam);   // 通知码
    HWND hCtrl  = (HWND) lParam;      // 控件 HWND

    switch (id) {
    case IDC_MY_BUTTON:
        if (code == BN_CLICKED) {
            MessageBox(hWnd, L"按钮被点击！", L"提示", MB_OK);
        }
        break;
    case IDC_MY_EDIT:
        if (code == EN_CHANGE) {
            // 用户输入了文字，内容已变化
        }
        break;
    }
    return 0;
}

按钮buttonBN_CLICKED、BN_DOUBLECLICKED
编辑框editEN_CHANGE、EN_UPDATE、EN_MAXTEXT
静态staticSTN_CLICKED（少用）
列表框listboxLBN_SELCHANGE、LBN_DBLCLK
组合框comboboxCBN_SELCHANGE、CBN_EDITCHANGE
滚动条scrollbarSB_LINEUP、SB_THUMBTRACK

控件消息（向控件发命令）

// 取编辑框文本
wchar_t buf[256];

// 或用通用 SendMessage

HMENU 复用：子窗口本身不需要菜单，所以 CreateWindow 的 HMENU 参数被复用为控件 ID。窗口内可通过 GetDlgCtrlID(hwnd) 取出该 ID。

07 对话框

对话框是从资源模板创建的窗口——它的子控件布局由 .rc 文件描述，无需手动一个个 CreateWindow。对话框分模态（阻塞父窗口，必须关闭才能继续）与非模态（与父窗口并存）两种。

模态 DialogBox
  • DialogBox / DialogBoxParam
  • 内部有自己的消息循环
  • 直到 EndDialog 才返回
  • 返回值是 EndDialog 的参数
  • 典型：关于框、设置页
非模态 CreateDialog
  • CreateDialog / CreateDialogParam
  • 立即返回 HWND
  • 共享主消息循环
  • 关闭用 DestroyWindow（不是 EndDialog）
  • 典型：查找框、工具面板

对话框过程 DlgProc

DlgProc 与 WndProc 形似而神不同：返回 INT_PTR，处理消息后返回 TRUE 表示"已处理"，返回 FALSE 让系统做默认处理——这与 WndProc 的 LRESULT 返回约定完全相反。

INT_PTR CALLBACK MyDlgProc(HWND hDlg, UINT msg,
                                WPARAM wParam, LPARAM lParam)
{
    switch (msg) {
    case WM_INITDIALOG:
        // 控件初始化（相当于 WM_CREATE）
        CheckDlgButton(hDlg, IDC_CHECK1, BST_CHECKED);
        return TRUE;            // ★TRUE = 让系统设焦点

    case WM_COMMAND:
        if (LOWORD(wParam) == IDOK ||
            LOWORD(wParam) == IDCANCEL) {
            EndDialog(hDlg, LOWORD(wParam));
            return TRUE;
        }
        break;
    }
    return FALSE;                   // ★FALSE = 默认处理
}

// 调用方：
INT_PTR ret = DialogBoxParam(
    hInstance, MAKEINTRESOURCE(IDD_MY_DIALOG),
    hWndParent, MyDlgProc, (LPARAM)pContext);

// 复选框状态
UINT checked = IsDlgButtonChecked(hDlg, IDC_CHECK1);
关闭方式不能混：模态对话框只能用 EndDialog 关闭，非模态只能用 DestroyWindow。混用会导致对话框无法关闭或资源泄漏。

10 高级主题

子类化（Subclassing）

子类化是替换窗口的窗口过程，从而改变其行为。这是 Win32 中"派生一个新控件类"的等价做法。通过 SetWindowLongPtr(GWLP_WNDPROC) 实现（64 位必须用 LongPtr 版本，不能用旧的 SetWindowLong）。

WNDPROC g_pOldProc;   // 保存原始 WndProc

LRESULT CALLBACK MyEditProc(HWND hWnd, UINT msg,
                                 WPARAM wp, LPARAM lp) {
    if (msg == WM_CHAR && wp == VK_RETURN) {
        // 拦截回车，做特殊处理
        MessageBox(hWnd, L"回车！", L"", MB_OK);
        return 0;
    }
    // 其余消息交给原 WndProc
    return CallWindowProc(g_pOldProc, hWnd, msg, wp, lp);
}

// 安装子类化
g_pOldProc = (WNDPROC)SetWindowLongPtr(
    hEdit, GWLP_WNDPROC, (LONG_PTR)MyEditProc);

// 卸载（恢复原 WndProc）
SetWindowLongPtr(hEdit, GWLP_WNDPROC, (LONG_PTR)g_pOldProc);

超类化（Superclassing）

超类化是复制一个已存在的窗口类，修改其 WndProc，注册成新类。与子类化的区别：子类化改的是单个窗口实例，超类化改的是类模板——之后所有该新类的窗口都走新行为。

DPI 感知

现代 Windows 必做：在高 DPI 显示器上，未声明 DPI 感知的程序会被系统位图缩放，导致文字模糊、布局错乱。两种声明方式：

方式一：API 运行时声明
SetProcessDpiAwarenessContext(DPI_AWARENESS_CONTEXT_PER_MONITOR_AWARE_V2);（在 WinMain 开头调用，必须在创建任何窗口前）。

方式二：manifest 声明（推荐）
在 exe 的 manifest 中加入 <DPIAwareness>PerMonitorV2</DPIAwareness>，系统加载时即生效，更可靠。

声明感知后还需响应 WM_DPICHANGED 消息重新调整布局，并用 GetDpiForWindow 取 DPI 缩放比例。

Unicode 与 TCHAR

Windows NT 内核原生 UTF-16，所有 USER/GDI 函数都有 W（Wide）与 A（ANSI）两个版本（如 CreateWindowW 与 CreateWindowA）。宏 CreateWindow 在 UNICODE 定义时展开为 W 版，否则为 A 版。

// TCHAR/_T 宏让代码在 ANSI/Unicode 间切换
TCHAR buf[256];
_tcscpy(buf, _T("Hello"));    // 编译期决定是 char 还是 wchar_t

// 现代实践：直接用 W 版 + wchar_t + L"" 字面量
wchar_t name[] = L"应用名";
SetWindowTextW(hWnd, name);

// 与 UTF-8 转换（与跨平台库、网络交互时）
int len = MultiByteToWideChar(CP_UTF8, 0, utf8, -1, NULL, 0);
MultiByteToWideChar(CP_UTF8, 0, utf8, -1, buf, len);

COM 初始化

许多 Windows API（Shell、Windows Ribbon、DirectX、现代对话框）基于 COM。使用前必须初始化 COM 公寓。

// 在 WinMain 开头（在创建窗口前）

// COINIT_APARTMENTTHREADED (STA) — GUI 线程推荐
// COINIT_MULTITHREADED     (MTA) — 后台/服务线程

11 完整示例

下面是一个完整可编译的 Win32 程序：带菜单、可在客户区画矩形、关闭时弹出确认对话框。它综合运用了窗口类、消息循环、GDI 绘图、菜单命令、对话框与资源加载。

// main.cpp — 完整 Win32 GUI 程序
#include <windows.h>
#include "resource.h"

HINSTANCE g_hInst;
HACCEL    g_hAccel;
HWND      g_hMain;
POINT     g_pts[256];      // 用户点击的坐标列表
int       g_nPts = 0;

// 窗口过程
LRESULT CALLBACK WndProc(HWND hWnd, UINT msg,
                          WPARAM wp, LPARAM lp)
{
    switch (msg) {

    case WM_CREATE: {
        // 加载菜单
        HMENU hMenu = LoadMenu(g_hInst, MAKEINTRESOURCE(IDR_MAINMENU));
        SetMenu(hWnd, hMenu);
        return 0;
    }

    case WM_LBUTTONDOWN: {
        int x = LOWORD(lp);
        int y = HIWORD(lp);
        if (g_nPts < 256) {
            g_pts[g_nPts].x = x; g_pts[g_nPts].y = y; g_nPts++;
            InvalidateRect(hWnd, NULL, FALSE);   // 触发重绘
        }
        return 0;
    }

    case WM_PAINT: {
        PAINTSTRUCT ps;
            HDC hdc = BeginPaint(hWnd, &ps);

            // 双缓冲
            RECT rc; GetClientRect(hWnd, &rc);
            HDC mem = CreateCompatibleDC(hdc);
            HBITMAP bmp = CreateCompatibleBitmap(hdc, rc.right, rc.bottom);
            HBITMAP old = (HBITMAP)SelectObject(mem, bmp);

            // 背景
            HBRUSH bg = CreateSolidBrush(RGB(255, 255, 255));
            FillRect(mem, &rc, bg);
            DeleteObject(bg);

            // 画所有点为红色小方块
            HBRUSH red = CreateSolidBrush(RGB(255, 0, 0));
            for (int i = 0; i < g_nPts; ++i) {
                RECT r = { g_pts[i].x - 5, g_pts[i].y - 5,
                          g_pts[i].x + 5, g_pts[i].y + 5 };
                FillRect(mem, &r, red);
            }
            DeleteObject(red);

            BitBlt(hdc, 0, 0, rc.right, rc.bottom, mem, 0, 0, SRCCOPY);
            SelectObject(mem, old);
            DeleteObject(bmp);
            DeleteDC(mem);

        EndPaint(hWnd, &ps);
        return 0;
    }

    case WM_COMMAND:
        switch (LOWORD(wp)) {
        case ID_FILE_CLEAR:
            g_nPts = 0;
            InvalidateRect(hWnd, NULL, TRUE);
            return 0;
        case ID_HELP_ABOUT:
            DialogBox(g_hInst, MAKEINTRESOURCE(IDD_ABOUT), hWnd, AboutProc);
            return 0;
        case ID_FILE_EXIT:
            SendMessage(hWnd, WM_CLOSE, 0, 0);
            return 0;
        }
        break;

    case WM_CLOSE:
        if (MessageBox(hWnd, L"确定退出？", L"确认",
                          MB_YESNO | MB_ICONQUESTION) == IDYES) {
            DestroyWindow(hWnd);
        }
        return 0;

    case WM_DESTROY:
        PostQuitMessage(0);
        return 0;

    default:
        return DefWindowProc(hWnd, msg, wp, lp);
    }
    return 0;
}

// "关于"对话框过程
INT_PTR CALLBACK AboutProc(HWND hDlg, UINT msg,
                                WPARAM wp, LPARAM lp) {
    switch (msg) {
    case WM_INITDIALOG:  return TRUE;
    case WM_COMMAND:
        if (LOWORD(wp) == IDOK || LOWORD(wp) == IDCANCEL) {
            EndDialog(hDlg, LOWORD(wp));
            return TRUE;
        }
        break;
    }
    return FALSE;
}

// 入口
int WINAPI wWinMain(HINSTANCE hInst, HINSTANCE,
                      LPWSTR, int nShow)
{
    SetProcessDpiAwarenessContext(DPI_AWARENESS_CONTEXT_PER_MONITOR_AWARE_V2);
    CoInitializeEx(NULL, COINIT_APARTMENTTHREADED);

    g_hInst  = hInst;
    g_hAccel = LoadAccelerators(hInst, MAKEINTRESOURCE(IDR_ACCEL));

    WNDCLASSEX wc = { sizeof(wc) };
    wc.lpfnWndProc   = WndProc;
    wc.hInstance      = hInst;
    wc.lpszClassName = L"DrawApp";
    wc.hCursor        = LoadCursor(NULL, IDC_CROSS);
    wc.hbrBackground = (HBRUSH)(COLOR_WINDOW + 1);
    RegisterClassEx(&wc);

    g_hMain = CreateWindow(wc.lpszClassName, L"画板",
        WS_OVERLAPPEDWINDOW, CW_USEDEFAULT, CW_USEDEFAULT,
        800, 600, NULL, NULL, hInst, NULL);

    ShowWindow(g_hMain, nShow);
    UpdateWindow(g_hMain);

    MSG msg;
    while (GetMessage(&msg, NULL, 0, 0) > 0) {
        // ★加速键必须在 TranslateMessage 之前翻译
        if (!TranslateAccelerator(g_hMain, g_hAccel, &msg)) {
            TranslateMessage(&msg);
            DispatchMessage(&msg);
        }
    }

    CoUninitialize();
    return (int) msg.wParam;
}
消息映射要点：菜单项 ID 与加速键 ID 共用同一个常量（如 ID_FILE_CLEAR），所以按 Ctrl+N 与点菜单 New 产生完全相同的 WM_COMMAND——无需为快捷键单独写处理。

Win32 之上有多个上层框架，每个都在不同维度做了抽象与提升。下表对比它们的核心维度，帮助你在新项目中做出选择。

何时仍选原生 Win32
  • 极致体积：KB 级可执行，无运行时依赖
  • 系统工具：任务管理器、注册表编辑器类工具
  • 驱动/服务配套 UI：与内核态交互紧密
  • 嵌入/资源受限：Windows Embedded 系统
  • 学习与教学：理解 GUI 工作原理
  • 互操作层：被其他语言通过 FFI 调用
何时避开 Win32
  • 需要复杂现代 UI（动画、矢量、特效）
  • 团队 C++ 经验不足
  • 需要快速迭代、可视化设计器
  • 跨平台需求
  • 需要数据绑定、MVVM 等现代模式
  • 全球化本地化工作量大（字符串表繁琐）
STA 单线程原则：Win32 GUI 是单线程公寓模型——一个窗口的所有操作必须在创建它的线程进行。后台线程要更新 UI 必须用 PostMessage（自定义消息）切回 UI 线程，绝不能直接调 SendMessage——这会导致死锁或断言。
