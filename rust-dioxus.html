01 概述

Dioxus 是一个用 Rust 编写的声明式、React 风格的 UI 框架。开发者用 RSX 宏（类 JSX 语法）描述界面，通过 Hooks 与 Signals 管理状态，同一份代码可编译到 Web、桌面、移动端、终端、服务端等多个渲染目标。它由 Jonathan Kelley 主导开发，是 Rust 生态中最接近“React for Rust”的方案。

核心特性
  • RSX 宏：类 JSX 的类型安全声明式 UI
  • 组件/Hooks：函数组件 + Hooks，React 开发者零门槛
  • Signals：细粒度响应式（0.5+），告别 diff 状态传递
  • 多平台：Web/Desktop/Mobile/TUI/SSR 一套代码
  • 编译期检查：Rust 类型系统保证 props/事件签名正确
  • WASM 优先：Web 端编译为 WebAssembly
渲染目标一览
平台后端产物
WebWASM + DOM.wasm + JS 胶水
Desktop系统 WebView跨平台桌面应用
Mobile原生渲染iOS/Android（实验）
TUI终端命令行界面
SSR服务端预渲染 HTML

02 架构与渲染管线

Dioxus 的核心思想是渲染器无关：上层是统一的组件/状态/RSX 抽象，下层通过 Renderer 适配不同平台。状态变化经响应式系统追踪，生成 VirtualDOM 差异（或细粒度订阅），再提交给渲染器更新真实 UI。

// 分层架构（简化）
┌─────────────────────────────────────────┐
│  你的代码：组件 + RSX + Hooks/Signals   │  ← 应用层（平台无关）
├─────────────────────────────────────────┤
│  Dioxus 核心：VirtualDOM + 响应式调度    │  ← 框架层
├─────────────────────────────────────────┤
│  Renderer 适配层（DomRenderer/          │
│    WebView/终端/...）                    │  ← 平台层
├─────────────────────────────────────────┤
│  平台：浏览器 DOM / 系统 WebView / 终端  │
└─────────────────────────────────────────┘

// 渲染流程
状态变更（Signal.write / set_state）
  → 响应式系统标记受影响组件为脏
  → 调度器在下一帧重新渲染脏组件（rsx! 重建 VNode）
  → Diff 生成 EditList（最小变更）
  → Renderer 提交到真实 UI（DOM patch / WebView / ...）
两种状态范式
范式粒度更新方式
VirtualDOM（0.4 时代）组件级整体 re-render + diff
Signals（0.5+）细粒度仅重算依赖该信号的值
入口函数 launch

0.5+ 统一用 launch(App) 启动：cargo 根据 feature 自动选择目标（web/desktop），无需手写平台入口。

03 RSX 宏

rsx! 是 Dioxus 的核心宏，把类 JSX 的语法在编译期转换为 VNode（虚拟节点）树。它类型安全——属性名、事件签名错误在编译期就会报错，而非运行时。

use dioxus::prelude::*;

fn main() {
    launch(App);
}

fn App() -> Element {
    let name = "Dioxus";
    rsx! {
        div { class: "card",
            h1 { "Hello, {name}!" }            // 字符串插值
            p { "声明式 UI in Rust" }
        }
    }
}

RSX 语法要素

rsx! {
    // 1. 元素 + 属性（属性名: 值,）
    a { href: "https://dioxuslabs.com", "官网" }

    // 2. 条件渲染（if 表达式）
    if show_detail {
        section { "详情..." }
    }

    // 3. 列表渲染（for 循环）
    ul {
        for item in items {
            li { "{item}" }
        }
    }

    // 4. 子组件（大括号传 props 与 children）
    Card { title: "标题",
        "这是 children"
    }

    // 5. 事件（on* 属性，闭包）
    button { onclick: move |_| do_something(), "点击" }

    // 6. 动态片段 / 裸表达式
    { computed_value }
}
字符串插值："{var}" 在 RSX 字符串内自动插值，底层生成 format_args!。任意 Display 类型可直接插入。复杂逻辑建议先在组件体内算好变量，RSX 只负责渲染。
与 JSX 的差异：RSX 用大括号块而非尖括号（div { ... } 而非 <div>...</div>）；属性用 key: value, 键值对；没有 className，统一用 class。

04 组件模型

Dioxus 组件是返回 Element（即 Option<VNode>）的普通函数。用 #[component] 宏标注后，框架为其生成可被 RSX 调用的组件包装。Props 是一个派生了 Props 的结构体。

// 1. 定义 Props 结构体
#[derive(Props, Clone, PartialEq)]
struct CardProps {
    title: String,
    #[props(default)]
    color: String,             // 带默认值的可选 prop
    children: Element,          // 接收 RSX 子内容
}

// 2. 组件函数：首参为 Props，返回 Element
#[component]
fn Card(props: CardProps) -> Element {
    rsx! {
        div { class: "card", style: "color: {props.color}",
            h2 { "{props.title}" }
            { props.children }       // 渲染 children
        }
    }
}

// 3. 使用（children 写在组件块内）
fn App() -> Element {
    rsx! {
        Card { title: "欢迎", color: "blue",
            "这是卡片正文内容"
        }
    }
}
组件规则
  • 纯函数：相同 props → 相同输出（除副作用 Hook）
  • 首字母大写：组件名用 PascalCase 区分元素
  • Props 不可变：通过新 props 触发重渲染
  • 可嵌套：组件组合构建复杂 UI
Props 特性
特性说明
#[props(default)]可选 prop 默认值
#[props(into)]自动类型转换（&str→String）
children: Element接收子节点
Clone, PartialEqprops 记忆化比较

05 Hooks 模式

Hooks 让函数组件拥有状态与副作用。Dioxus 的 Hooks 与 React 几乎同名同义，但有 Rust 的所有权与生命周期保障。Hooks 必须在组件顶层、按固定顺序调用（同 React 规则）。

fn Counter() -> Element {
    // use_signal：响应式可变状态（0.5+ 推荐）
    let mut count = use_signal(|| 0);

    // use_hook：一次性初始化（返回任意值，常用于桥接外部库）
    let handle = use_hook(|| create_external_handle());

    // use_effect：状态变化后的副作用（订阅/日志）
    use_effect(move || {
        println!("count 变为 {}", count());
    });

    rsx! {
        button { onclick: move |_| count += 1, "点击 {count}" }
    }
}
Hook用途返回
use_signal响应式可变状态Signal<T>
use_resource异步任务（Future）Resource<T>
use_effect副作用（依赖 Signal 变化）()
use_memo记忆化派生值Memo<T>
use_context读取上下文注入T
use_hook一次性初始化T
use_callback稳定回调（不随渲染重建）闭包
Hooks 规则：只能在组件/其他 Hook 内调用，且必须按固定顺序、无条件调用——不能放在 if/for/closure 内。违反会导致状态错位（同 React）。需要条件化时，把分支放在 Hook 闭包内部。

06 Signals 响应式

Signals 是 Dioxus 0.5 引入的革命性特性，灵感来自 SolidJS 的细粒度响应式。它取代了旧的 use_state + props 逐层传递的模式：状态直接原地可变，读取处自动建立订阅，修改时只重算依赖该信号的部分，无需组件整体 diff。

fn App() -> Element {
    let mut count = use_signal(|| 0);

    rsx! {
        button { onclick: move |_| count += 1, "+" }
        button { onclick: move |_| count -= 1, "-" }
        Display {}      // 子组件无需 props 也能读到最新值
    }
}

fn Display() -> Element {
    // 直接读取全局/上层 Signal —— 自动订阅
    let count = use_signal(|| 0);   // 演示：实际用 Context 共享同一 Signal
    rsx! { p { "count = {count}" } }
}

读写与运算符

let mut n = use_signal(|| 10);

n.read();               // 读：建立订阅，返回引用
n();                    // 读的简写（Signal 实现 Fn）
n.write();              // 写：返回可变引用，自动通知
n.set(5);               // 整体赋值
n += 1;                 // 运算符重载（+= -= *= ...）

// 派生信号：仅当 count 变化时重算
let doubled = use_memo(move || count() * 2);
Signals vs use_state
维度use_state（旧）Signal（新）
更新粒度组件整体细粒度订阅
状态传递逐层 props/callback原地共享
性能diff 成本随树增长仅重算依赖
心智不可变 + 重渲染可变 + 响应式
细粒度响应式
  • 读取即订阅（read()/()）
  • 写入即通知所有订阅者
  • Memo 缓存派生计算
  • 无需手动管理依赖数组
核心价值：Signals 让状态可以“原地修改、自动追踪”，配合 RSX 字符串插值 "{count}"，几乎消除了状态提升与回调钻取的样板代码。这是 Dioxus 区别于早期版本、对标 SolidJS/Leptos 的关键能力。

07 状态与上下文

跨组件共享状态用Context（依赖注入）。父组件 use_context_provider 注入，任意后代 use_context 取用——无需逐层 props 传递。配合 Signal 即构成全局响应式状态。

use dioxus::prelude::*;

// 用 Copy 类型作为 Context 键（通常用 Signal 包装的共享状态）
#[derive(Clone, Copy)]
struct AppState { count: Signal<i32> }

fn App() -> Element {
    // 顶层注入：use_context_provider 创建并下发
    use_context_provider(|| AppState {
        count: use_signal(|| 0),
    });

    rsx! { Child {} }
}

fn Child() -> Element {
    // 任意后代读取同一份状态
    let state = use_context::<AppState>();
    rsx! {
        button { onclick: move |_| state.count += 1, "{state.count}" }
    }
}
全局状态：需要在根组件之外访问（如事件循环、定时器）时，可用 Signal::global() 注册全局信号——但优先用 Context，保持可测试性与组件作用域。

08 事件处理

事件用 on* 属性绑定闭包。Dioxus 统一抽象了各平台事件（DOM/原生），提供类型安全的事件对象 Event<T>。

fn Form() -> Element {
    let mut text = use_signal(|| String::new());

    rsx! {
        input {
            value: "{text}",
            oninput: move |e: Event<InputData>| {
                // e.data().value() 拿到输入值
                text.set(e.data().value());
            }
        }
        button {
            onclick: move |e: Event<MouseData>| {
                // 阻止默认行为（如表单提交）
                e.prevent_default();
                println!("点击于 {}", e.data().client_coordinates());
            },
            "提交"
        }
    }
}
事件属性类型典型用途
onclickMouseData点击
oninputInputData输入框实时值
onkeydownKeyboardData按键（含 code/key）
onsubmitFormData表单提交
onmountedMountedData元素挂载（拿 DOM 句柄）
事件冒泡：事件默认冒泡，e.stop_propagation() 阻止向上传递。平台差异由 Event<T> 抹平——同一份事件代码在 Web/Desktop 行为一致。

09 异步与资源

use_resource 把一个 Future 绑定到组件生命周期，返回 Resource<T>。它会在依赖的 Signal 变化时自动重启，配合 Suspense 处理加载态——这是 Dioxus 处理网络请求、异步计算的标准方式。

fn UserList() -> Element {
    let mut query = use_signal(|| String::new());

    // query 变化时自动重新执行（闭包内读取了 query）
    let users = use_resource(move async move {
        if query().is_empty() { return Vec::new(); }
        fetch_users(&query()).await.unwrap_or_default()
    });

    rsx! {
        input { value: "{query}", oninput: move |e| query.set(e.data().value()) }

        // Suspense：等待异步就绪前显示 fallback
        SuspenseBoundary {
            fallback: |_| rsx!{ p{"加载中..."} },
            match &*users.read_unchecked() {
                None => rsx!{ p{"等待中"} },
                Some(list) => rsx!{
                    for u in list { div { "{u.name}" } }
                }
            }
        }
    }
}
能力说明
自动依赖追踪闭包内读取的 Signal 变化即重启
Resource::read读取最新值（阻塞渲染直到就绪）
SuspenseBoundary声明式加载态/错误态边界
use_future无返回值的副作用 Future
WASM 限制：Web 目标是单线程 WASM，.await 走异步运行时；阻塞调用会卡死 UI。网络请求用 reqwest + async，切勿在渲染路径做同步 IO。

10 路由

dioxus-router 提供类型安全的声明式路由。用 #[derive(Routable)] 定义一个 路由枚举，每个变体对应一个路径与组件——编译器保证所有路由都有归属，避免运行时 404。

use dioxus::prelude::*;
use dioxus_router::prelude::*;

// 1. 路由枚举：变体 = 路由
#[derive(Routable, Clone, PartialEq)]
#[rustfmt::skip]
enum Route {
    #[route("/")]
    Home {},
    #[route("/user/:id")]
    User { id: u64 },        // 动态参数自动解析
    #[route("/404")]
    NotFound {},
}

// 2. 每个变体对应一个同名组件
#[component]
fn User(id: u64) -> Element {
    rsx! { h1{"用户 #{id}"} }
}

// 3. Router 组件挂载，Outlet 渲染匹配的子路由
fn App() -> Element {
    rsx! {
        Router<Route> {}
        // 内部自动用 Outlet 渲染当前 Route 对应组件
    }
}

// 4. 导航：类型安全的链接与跳转
let navigator = navigator();
navigator.push(Route::User { id: 42 });

rsx! { Link { to: Route::Home {}, "首页" } }
类型安全：路由参数（如 :id）由 derive 宏解析为枚举字段，类型错误（传字符串到 u64）编译期即失败。导航也是 Route::User{id:42} 而非裸字符串。

11 多平台渲染

同一份组件/状态代码，通过 cargo 的 feature flag 切换渲染目标。各平台能力略有差异，但核心 API 一致。

目标Feature渲染后端特性
WebwebWASM → DOM快、小，部署静态托管
Desktopdesktop系统 WebView跨 Win/macOS/Linux，体积小
Mobilemobile原生渲染iOS/Android（实验阶段）
TUItui终端 cells命令行富 UI
SSRssr服务端预渲染 HTML / 全栈
// Cargo.toml：用 feature 选择目标
[dependencies]
dioxus = { version = "0.6", features = ["web"] }   // 改 "desktop" 即桌面应用

// main.rs 完全不变：launch 自动适配目标
fn main() { launch(App); }

// 平台特定代码：用 cfg 区分
#[cfg(feature = "desktop")]
fn title() -> String { dioxus::desktop::window().title() }
Web 目标
  • 编译为 WASM，JS 仅做胶水
  • dx serve 热重载开发
  • dx build --release 产出 dist/
  • 体积优化：tree-shaking + wasm-opt
Desktop 目标
  • 复用系统 WebView（无捆绑 Chromium）
  • 原生菜单、托盘、窗口控制
  • 文件系统、IPC 等能力开放
  • 跨三平台单二进制

12 完整示例

下面是一个可运行的多功能 To-Do 应用：响应式状态 + 条件/列表渲染 + 路由 + 持久化副作用，体现 Dioxus 各能力协作。

use dioxus::prelude::*;
use dioxus_router::prelude::*;

#[derive(Clone, PartialEq)]
struct Todo { id: u32, text: String, done: bool }

fn main() { launch(App); }

fn App() -> Element {
    let mut todos = use_signal(|| Vec::<Todo>::new());
    let mut input = use_signal(|| String::new());

    // 副作用：todos 变化时打印（演示 use_effect + Signal 联动）
    use_effect(move || {
        println!("当前 {} 项待办", todos.read().len());
    });

    rsx! {
        h1 { "Dioxus Todo" }
        div {
            input {
                value: "{input}",
                oninput: move |e| input.set(e.data().value()),
                onkeydown: move |e| if e.data().code() == Code::Enter {
                    if !input().is_empty() {
                        todos.write().push(Todo {
                            id: todos.read().len() as u32,
                            text: input.take(),
                            done: false,
                        });
                    }
                },
            }
        }
        ul {
            for todo in todos().iter() {
                TodoItem { key: "{todo.id}", id: todo.id }
            }
        }
        p { "剩余 {todos.iter().filter(|t| !t.done).count()} 项" }
    }
}

#[component]
fn TodoItem(id: u32) -> Element {
    let mut todos = use_signal::<Vec<Todo>>(|| default());
    let todo = todos.read().iter().find(|t| t.id == id).cloned();
    rsx! {
        if let Some(t) = todo {
            li {
                onclick: move |_| {
                    let mut w = todos.write();
                    if let Some(t) = w.iter_mut().find(|x| x.id == id) { t.done = !t.done; }
                },
                style: "text-decoration: {if t.done {'line-through'} else {'none'}}",
                "{t.text}"
            }
        }
    }
}
要点：单一 todos Signal 是状态真相源；输入框、列表项、计数都从它派生。点击切换 done 后，仅依赖该字段的 UI 重算——这是 Signals 细粒度响应式的直接收益。

13 生态与对比

Rust 生态有多个 GUI/UI 框架，定位各异。Dioxus 以“React 心智 + 多平台 + WASM 优先”为差异化优势。

框架范式渲染目标定位
DioxusRSX + Hooks/SignalsWeb/Desktop/Mobile/TUI多平台 React 风格
LeptosRSX + 细粒度响应式Web/SSRWeb 优先，极致响应式
Yewhtml! 宏 + Component traitWebWeb 专精（Elm/React 风）
TauriWeb 前端 + Rust 后端Desktop/Mobile用任意前端，Rust 做后端
Slint自有 DSL + RustDesktop/Mobile/嵌入式声明式 DSL，嵌入式友好
egui即时模式（immediate）Native/egui 渲染工具/调试 UI，简单快
IcedElm 架构（Model/Msg）Native（wgpu）纯原生 GPU 渲染
Dioxus 的优势
  • React 开发者迁移成本最低
  • 一套代码覆盖最多平台
  • Signals 兼顾性能与简洁
  • 编译期类型安全（路由/props/事件）
何时选 Dioxus
  • 团队熟悉 React，想用 Rust
  • 需要 Web + 桌面同源
  • 追求 WASM 体积与性能
  • 需要类型安全的大规模 UI
权衡：Dioxus Desktop 复用系统 WebView（依赖宿主环境），与 Iced/Slint 的原生 GPU 渲染相比，复杂自绘场景控制力较弱；Mobile 目标仍在成熟中。游戏/重度图形应考虑 wgpu 直绘或 egui。
