01 概述

本文以 UDP 数据包为例，完整介绍数据包如何从物理网卡一步一步传递到用户进程。整个过程分为四个阶段：网卡到内存、内核网络模块、IP/UDP 协议栈、应用层 socket。

说明：本文只讨论以太网物理网卡，不涉及虚拟设备。函数调用关系基于内核 7.0.12，与旧版有细微差别但原理相同。

02 网卡到内存

数据包进入主机后，经历以下 6 个步骤：

                            |           |          |

+-------->|  NIC   |--------+    DMA    |    DMA   |
|  网线   |        |----+    (步骤 2)   |          |
+---------+--------+  |                 |          |

                      |                 |          |
                      ↓                 |          |

                      |                 |          |
                      ↓ 5. Disable IRQ  |          |

                        |                          |
                        ↓ 8. net_rx_action         |
                        +--------------------------+
步骤操作说明
1数据包到达网卡如果目的 MAC 不是当前网卡且未开启混杂模式，包被丢弃
2DMA 写入内存网卡通过 DMA 将数据包写入驱动分配的内存缓冲区
3硬件中断（IRQ）网卡通过 IRQ 通知 CPU 有数据到达
4CPU 执行 IRQ HandlerCPU 根据中断向量表调用注册的中断处理函数（→ NIC Driver）
5禁用网卡中断驱动禁用 IRQ，后续数据包直接 DMA 到内存，不再中断 CPU
6触发软中断硬件中断处理结束，耗时操作移到软中断上下文处理
为什么需要软中断？
硬件中断处理程序执行过程中不能被中断。如果它执行时间过长，会导致 CPU 无法响应其他硬件中断。内核引入软中断（softirq），将耗时的数据包处理操作移到软中断上下文中执行。

步骤函数说明
7ksoftirqd内核软中断进程，调用 net_rx_action 处理网络数据
8net_rx_action调用网卡驱动的 poll 函数逐个处理数据包
9poll 函数驱动读取内存中的原始数据包
10skb 分配驱动将原始数据转换为内核 sk_buff 格式
11napi_gro_receive处理 GRO（合并小包），判断是否开启 RPS
12enqueue_to_backlogRPS 开启时，将 skb 放入 CPU 的 input_pkt_queue
13input_pkt_queueCPU 在软中断上下文中处理队列中的数据包
14__netif_receive_skb_coreRPS 关闭时直接调用；开启时从队列取出调用
15AF_PACKET检查是否有原始套接字（tcpdump 抓包点）
16协议栈分发根据 ethertype 交给 IP / ARP / IPv6 等协议层处理
17Enable IRQ所有数据包处理完成后，重新启用网卡硬中断

关键概念

NAPI

New API。传统模式下每个包都触发一次中断，高流量时 CPU 忙于中断处理。NAPI 在第一个包到达时触发中断，然后驱动禁用中断，改用轮询（poll）模式批量处理多个包，处理完后再启用中断。

GRO / RPS

GRO（Generic Receive Offload）：合并可聚合的小数据包，减少协议栈处理次数。
RPS（Receive Packet Steering）：将数据包分发到其他 CPU 的队列中处理，实现软件层面的接收多队列。

函数说明
ip_rcvIP 模块入口。丢弃垃圾数据包（混杂模式下收到的非本机包），然后调用 NF_INET_PRE_ROUTING
NF_INET_PRE_ROUTINGNetfilter 钩子，iptables PREROUTING 链的处理点。可修改或丢弃数据包
routing路由查找。目的 IP 非本地且未开 ip_forward → 丢弃；开了 → ip_forward 转发
ip_forward调用 NF_INET_FORWARD 钩子，未丢弃则调用 dst_output_sk 发送出去
ip_local_deliver目的 IP 是本地。调用 NF_INET_LOCAL_IN 钩子，通过后交给 UDP 层
NF_INET_LOCAL_INNetfilter 钩子，iptables INPUT 链的处理点
Netfilter 四个钩子位置
PREROUTING → routing → FORWARD（转发）/ LOCAL_IN（本机）→ POSTROUTING
在本例（UDP 包接收）中经过的钩子：PRE_ROUTING → LOCAL_IN

函数说明
udp_rcvUDP 模块入口。做必要检查
__udp4_lib_lookup_skb根据目的 IP + 端口查找匹配的 socket。找不到 → 丢弃

sk_filter每个 socket 可设置类似 tcpdump 的过滤规则，不满足条件 → 丢弃
__skb_queue_tail将 skb 放入 socket 的 receive_skb 队列末尾
sk_data_ready唤醒等待的进程（recvfrom 阻塞返回）或通知 epoll/select

06 应用层接收

应用层有两种接收数据的方式：

// 方式 1：阻塞式 recvfrom
nread = recvfrom(sockfd, buffer, MAXLINE, 0, NULL, NULL);
// 进程阻塞 → sk_data_ready 唤醒 → 从 receive_skb 队列读取数据

// 方式 2：epoll/select 监听
int nready = epoll_wait(epfd, events, MAX_EVENTS, -1);
// socket 可读事件触发 → 调用 recvfrom 读取队列数据

丢包点总结
整个接收路径中有多个可能丢包的位置：
• 网卡：混杂模式检查、MAC 过滤
• 驱动：内存不足时 DMA 缓冲区满
• enqueue_to_backlog：input_pkt_queue 满（可通过 net.core.netdev_max_backlog 调整）
• AF_PACKET：tcpdump 过滤
• ip_rcv：混杂模式 + PACKET_OTHERHOST
• routing：无路由且未开 ip_forward
• udp_rcv：找不到匹配的 socket
• sock_queue_rcv_skb：socket 接收缓冲区满（net.core.rmem_max）
• sk_filter：BPF 过滤器不匹配
