01 Goroutine

Goroutine 是 Go 并发模型的核心，是一种由 Go 运行时管理的轻量级线程。与操作系统线程相比，goroutine 的创建和切换成本极低（栈初始仅 2-4KB），使得 Go 程序可以轻松创建数十万甚至上百万个 goroutine。

go 语句

在函数调用前加上 go 关键字即可启动一个 goroutine：

// 基本用法
go doSomething()

// 匿名函数
go func() {
    fmt.Println("在 goroutine 中运行")
}()

// 带参数（值捕获）
for i := 0; i < 10; i++ {
    go func(n int) {
        fmt.Println(n)
    }(i)  // 传递 i 的副本
}

// 错误示范（循环变量捕获问题）
for i := 0; i < 10; i++ {
    go func() {
        fmt.Println(i)  // ❌ 所有 goroutine 看到同一个 i
    }()
}
// Go 1.22+ 已修复此问题，循环变量每次迭代创建新变量

goroutine 栈

Goroutine 使用动态可增长栈（初始约 2KB，最大可达 1GB），由运行时管理。栈扩展通过 runtime.morestack 实现：

// 查看当前 goroutine 数量
import "runtime"

func main() {
    fmt.Println(runtime.NumGoroutine())

    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            time.Sleep(time.Second)
        }()
    }
    wg.Wait()
    fmt.Println(runtime.NumGoroutine())  // 回到 1
}

GOMAXPROCS

GOMAXPROCS 控制 Go 程序可以使用的操作系统线程数，默认为 CPU 核心数。它不是 goroutine 数量的上限，而是并发执行 goroutine 的 worker 线程数。

import "runtime"

func main() {
    fmt.Println("默认 GOMAXPROCS:", runtime.GOMAXPROCS(0))
    // 设置
    runtime.GOMAXPROCS(8)  // 最多 8 个 OS 线程

    // 或者通过环境变量设置
    // GOMAXPROCS=4 go run main.go
}

关键理解：

  • GOMAXPROCS = 并行执行 goroutine 的 P（处理器）数量
  • goroutine 总数远大于 GOMAXPROCS → 调度器分时复用 OS 线程
  • I/O密集型 / CPU密集型任务可通过调节 GOMAXPROCS 优化
  • 容器环境下建议用 uber-go/automaxprocs 自动探测 cgroup 配额

02 Channel

Channel 是 Go 实现 CSP（Communicating Sequential Processes）模型的一等公民——"不要通过共享内存来通信，而应该通过通信来共享内存"。channel 本身是类型化的、并发安全的、基于 FIFO 的通信管道。

创建与基本操作

// 创建（make 函数，不是 new）
ch1 := make(chan int)          // 无缓冲 channel
ch2 := make(chan string, 10)   // 有缓冲 channel，容量 10

// 发送与接收
ch1 <- 42       // 发送（阻塞直到对端接收）
val := <-ch1    // 接收（阻塞直到有数据）
val, ok := <-ch1 // 带关闭检测的接收

// 关闭
close(ch)       // 关闭后不能再发送，但可以继续接收剩余内容

无缓冲 vs 有缓冲

这是 Go 并发设计中最关键的区别：

特性无缓冲（容量 0）有缓冲（容量 > 0）
发送阻塞直到接收方准备好阻塞直到有空间
接收阻塞直到发送方准备好阻塞直到有数据
行为同步——发送和接收配对异步——解耦生产和消费
典型用途信号通知、同步工作队列、限流
// 无缓冲 channel - 确保通知到达
func main() {
    done := make(chan struct{})
    go func() {
        fmt.Println("工作完成")
        done <- struct{}{}  // 发送通知
    }()
    <-done  // 等待通知（阻塞）
    fmt.Println("主 goroutine 继续")
}

// 有缓冲 channel - 工作队列
func worker(id int, jobs <-chan int) {
    for j := range jobs {
        fmt.Printf("worker %d 处理 job %d\n", id, j)
    }
}

func main() {
    jobs := make(chan int, 100)
    go worker(1, jobs)

    for i := 0; i < 50; i++ {
        jobs <- i
    }
    close(jobs)  // 通知 worker 没有更多任务
    time.Sleep(time.Second)
}

关闭与 range

使用 range 遍历 channel 直到关闭：

func producer(out chan<- int) {
    for i := 0; i < 10; i++ {
        out <- i * 2
    }
    close(out)  // 必须关闭，否则 range 死锁
}

func main() {
    ch := make(chan int)
    go producer(ch)

    // range 自动检测 close
    for val := range ch {
        fmt.Println(val)
    }
    // 等价于：
    // for {
    //     val, ok := <-ch
    //     if !ok { break }
    //     fmt.Println(val)
    // }
}

关闭 channel 的规则：

  • 发送方关闭 channel（接收方不应该关闭）
  • 重复关闭 close(ch) 会 panic
  • 向已关闭的 channel 发送会 panic
  • 从已关闭的 channel 接收立即返回零值 + false

Channel 方向

Go 类型系统支持 channel 方向约束，在编译期保证 channel 不被误用：

// 只读 channel（输入）
func consumer(ch <-chan int) {
    for v := range ch {
        fmt.Println(v)
    }
}

// 只写 channel（输出）
func producer(ch chan<- int) {
    for i := 0; i < 10; i++ {
        ch <- i
    }
    close(ch)
}

func main() {
    ch := make(chan int)  // 双向
    go producer(ch)       // 自动转换为 chan<-int
    consumer(ch)          // 自动转换为 <-chan int
}

03 Select 多路复用

select 是 Go 并发控制的核心，它允许一个 goroutine 等待多个 channel 操作，任意一个准备好时执行对应的 case。select 将 channel 和逻辑分支结合在一起，避免复杂的轮询和回调。

基本 select

func main() {
    ch1 := make(chan string)
    ch2 := make(chan string)

    go func() {
        time.Sleep(1 * time.Second)
        ch1 <- "来自 ch1"
    }()
    go func() {
        time.Sleep(2 * time.Second)
        ch2 <- "来自 ch2"
    }()

    select {
    case msg := <-ch1:
        fmt.Println(msg)
    case msg := <-ch2:
        fmt.Println(msg)
    case <-time.After(3 * time.Second):
        fmt.Println("超时")
    }
}

默认分支（非阻塞）

default 分支使 select 变为非阻塞：

// 非阻塞接收
select {
case msg := <-ch:
    fmt.Println("收到:", msg)
default:
    fmt.Println("没有数据，不阻塞")
}

// 非阻塞发送
select {
case ch <- value:
    fmt.Println("发送成功")
default:
    fmt.Println("channel 已满，跳过发送")
}

// 非阻塞检查 channel 状态
select {
case <-signal:
    fmt.Println("收到信号")
default:
    fmt.Println("信号未到，继续工作")
}

定时器与超时模式

// 超时控制
func fetchWithTimeout(url string, timeout time.Duration) (string, error) {
    result := make(chan string, 1)

    go func() {
        data, err := httpFetch(url)
        if err != nil {
            return
        }
        result <- data
    }()

    select {
    case data := <-result:
        return data, nil
    case <-time.After(timeout):
        return "", errors.New("请求超时")
    }
}

// 定期执行 + 可取消
func heartbeat(ctx context.Context, interval time.Duration) <-chan struct{} {
    out := make(chan struct{})
    go func() {
        ticker := time.NewTicker(interval)
        defer ticker.Stop()
        for {
            select {
            case <-ticker.C:
                out <- struct{}{}
            case <-ctx.Done():
                return  // 上下文取消时退出
            }
        }
    }()
    return out
}

select 与 for 循环

select 经常配合 for 循环使用以实现持续监听：

// 事件循环
func eventLoop(stop <-chan struct{}) {
    for {
        select {
        case <-stop:
            fmt.Println("收到停止信号，退出")
            return
        case task := <-taskCh:
            processTask(task)
        case <-ticker.C:
            doPeriodicWork()
        default:
            // 没有事件时做空闲工作
            idleWork()
        }
    }
}
阅读建议:
select 随机选择就绪的 case（不是顺序扫描），这保证了公平性。同时监听多个 channel 时，不要假设哪个会优先执行。如果需要优先级，可以用嵌套 select 或两个 select 配合 default 实现。

04 Sync.Mutex / RWMutex

虽然 Go 推荐通过 channel 通信来实现并发，但共享内存加互斥锁仍然是某些场景下的最优选择。Go 标准库提供了 sync.Mutex 和 sync.RWMutex，两者都是不可重入的（同一 goroutine 不能对同一个已锁定的 Mutex 再次加锁）。

Mutex 互斥锁

type SafeCounter struct {
    mu    sync.Mutex
    value int64
}

func (c *SafeCounter) Inc() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.value++
}

func (c *SafeCounter) Value() int64 {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.value
}

RWMutex 读写锁

RWMutex 适用于读多写少的场景。多个读操作可以同时进行，写操作是独占的：

type SafeCache struct {
    mu    sync.RWMutex
    data  map[string]any
}

func (c *SafeCache) Get(key string) (any, bool) {
    c.mu.RLock()           // 读锁
    defer c.mu.RUnlock()   // 注意是 RUnlock 不是 Unlock
    v, ok := c.data[key]
    return v, ok
}

func (c *SafeCache) Set(key string, value any) {
    c.mu.Lock()            // 写锁
    defer c.mu.Unlock()
    c.data[key] = value
}

// 读取性能对比（基准测试）：读锁比写锁快约 10 倍

零值锁

sync.Mutex 和 sync.RWMutex 的零值就是已解锁的锁，可以直接使用。不需要 new 或 make：

// 正确 — 零值可用
type Counter struct {
    sync.Mutex       // 可以直接内嵌
    Count int
}

func (c *Counter) Add(n int) {
    c.Lock()         // 直接使用内嵌的 Mutex 方法
    c.Count += n
    c.Unlock()
}

// 即使是指针类型，也建议用 new：
counter := new(Counter)  // Mutex 零值可用

// 注意：Mutex 不能被复制（复制后两个实例共享同一状态）
// func (c Counter) Bad() {  // ❌ 值接收者会复制 Mutex
//     c.Lock()
// }

锁的常见陷阱

// 陷阱 1: 忘记解锁（使用 defer 保证）
func (c *Counter) Bad() {
    c.Lock()
    val := c.Count
    // 如果这里有 return 或 panic...
    // c.Unlock()  // 可能不会执行
}

func (c *Counter) Good() {
    c.Lock()
    defer c.Unlock()  // 即使 return 或 panic 也执行
    return c.Count
}

// 陷阱 2: 锁不可重入
type Service struct {
    mu sync.Mutex
}

func (s *Service) Process() {
    s.mu.Lock()
    defer s.mu.Unlock()
    s.Validate()  // ❌ 如果 Validate 也加锁，死锁
}

// 陷阱 3: 复制锁
type SafeMap struct {
    mu sync.Mutex
    m  map[int]int
}

func copyStruct(s SafeMap) {  // ❌ 值传递会复制锁
    s.mu.Lock()
    // ...
    s.mu.Unlock()
}

05 Sync.WaitGroup / Once / Cond

除了互斥锁，sync 包还提供了多个高级同步原语，用于协调 goroutine 的执行流程。

WaitGroup — 等待一组 goroutine

func main() {
    var wg sync.WaitGroup

    for i := 0; i < 5; i++ {
        wg.Add(1)  // 计数器 +1
        go func(id int) {
            defer wg.Done()  // 计数器 -1
            fmt.Printf("worker %d 开始\n", id)
            time.Sleep(time.Duration(id) * 100 * time.Millisecond)
            fmt.Printf("worker %d 结束\n", id)
        }(i)
    }

    wg.Wait()  // 阻塞直到计数器为 0
    fmt.Println("所有 worker 完成")
}

// 常见错误：在 goroutine 内部调用 wg.Add
for i := 0; i < 5; i++ {
    go func() {
        wg.Add(1)  // ❌ 可能主 goroutine 已经 wg.Wait() 了
        defer wg.Done()
        // ...
    }()
}
wg.Wait()  // ❌ 可能在 Add 之前就执行了

Once — 只执行一次

type Singleton struct {
    // ...
}

var (
    instance *Singleton
    once     sync.Once
)

func GetInstance() *Singleton {
    once.Do(func() {
        instance = &Singleton{}
        fmt.Println("创建唯一实例")
    })
    return instance
}

// sync.Once 的 Do 函数保证：
// - 只执行一次
// - 即使并发调用也只会有一个 goroutine 执行
// - 内部 mutex + atomic 实现

Cond — 条件变量

Cond 用于 goroutine 等待特定条件变为 true。它比 channel 更灵活，也更复杂：

type Queue struct {
    items []int
    cond  *sync.Cond
}

func NewQueue() *Queue {
    return &Queue{
        cond: sync.NewCond(&sync.Mutex{}),
    }
}

func (q *Queue) Enqueue(item int) {
    q.cond.L.Lock()
    defer q.cond.L.Unlock()
    q.items = append(q.items, item)
    q.cond.Signal()  // 通知一个等待者
}

func (q *Queue) Dequeue() int {
    q.cond.L.Lock()
    defer q.cond.L.Unlock()
    for len(q.items) == 0 {
        q.cond.Wait()  // 等待（自动解锁，被唤醒后自动加锁）
    }
    item := q.items[0]
    q.items = q.items[1:]
    return item
}

Cond 的三个方法：

  • Wait() — 自动解锁 → 阻塞等待 → 被唤醒后加锁
  • Signal() — 唤醒一个等待者
  • Broadcast() — 唤醒所有等待者

Cond 很少直接使用，因为 channel 通常能更清晰地表达同样的语义。Cond 适用于多个等待者竞争同一个资源的场景。

06 Sync.Pool 对象池

sync.Pool 是一个临时对象缓存池，用于减少频繁分配和 GC 的压力。它的设计目标是"临时"——池中的对象可能在任意时刻被 GC 回收，所以 Pool 适合缓存分配开销大的临时对象。

type Pool struct {
    // New 在 Get 发现池为空时调用，创建新对象
    New func() any
}

// 创建一个 buffer 池
var bufferPool = sync.Pool{
    New: func() any {
        return new(bytes.Buffer)
    },
}

func processData(data []byte) string {
    buf := bufferPool.Get().(*bytes.Buffer)
    buf.Reset()              // 重置而不是新建
    defer bufferPool.Put(buf)  // 放回池中

    // 使用 buffer
    buf.Write(data)
    buf.WriteString(" processed")
    return buf.String()
}

// Benchmark 对比（000 次操作）:
// 无 Pool: 70ms, 大量 GC
// 有 Pool: 15ms, 几乎无 GC

Pool 的实现细节

// 运行时实现（简化）：
// 每个 P 有一个私有池和一个共享池链表
// Get: 先尝试私有的，再尝试共享的，最后调用 New
// Put: 放回当前 P 的共享池

// 关键特性：
// 1. 无锁（per-P 本地缓存）
// 2. GC 时会清空所有 Pool
// 3. 是"丢失缓存"不是"泄漏缓存"
// 4. 跨 goroutine 安全

// 适用场景：
// - 频繁分配/释放的小对象（如 buffer, 临时切片）
// - 高吞吐的并发处理
// - 减少 GC 压力的辅助手段

// 不适用场景：
// - 有状态的长期对象
// - 需要保证对象存活时间的场景

Pool 与 GC 的关系：每次 GC 时，Pool 中的所有对象都会被无声清空。这意味着 Pool 不能用于有持久状态的连接池（应当用 channel 实现连接池）。

07 Atomic 原子操作

Go 的 sync/atomic 包提供了底层原子内存操作，由 CPU 指令直接支持（如 x86 的 LOCK CMPXCHG），比互斥锁开销小得多。适用于计数器、标志位、状态转换等简单场景。

原子操作的基本函数

import "sync/atomic"

var counter atomic.Int64  // Go 1.19 引入的类型化 API

// 基本操作
counter.Store(100)        // 原子写入
v := counter.Load()       // 原子读取
old := counter.Add(5)     // 原子增减并返回旧值
swapped := counter.CompareAndSwap(100, 200)  // CAS

// 旧版函数式 API（仍可使用）
var counter2 int64
atomic.AddInt64(&counter2, 1)
atomic.StoreInt64(&counter2, 100)
v2 := atomic.LoadInt64(&counter2)

// Benchmark: atomic 比 Mutex 快约 5-10 倍

Atomic.Value

atomic.Value 用于原子地读写任意类型的值，适用于配置热更新、安全快照等场景：

type Config struct {
    Host    string
    Port    int
    Timeout time.Duration
}

type Server struct {
    config atomic.Value  // 存储 *Config
}

func (s *Server) LoadConfig() *Config {
    return s.config.Load().(*Config)
}

func (s *Server) UpdateConfig(cfg *Config) {
    s.config.Store(cfg)
}

// 使用场景：无需锁，实现无锁读取
func main() {
    srv := &Server{}
    srv.UpdateConfig(&Config{
        Host:    "0.0.0.0",
        Port:    8080,
        Timeout: 30 * time.Second,
    })

    // 并发安全读取（无锁）
    go func() {
        for {
            cfg := srv.LoadConfig()
            fmt.Printf("当前配置: %+v\n", cfg)
            time.Sleep(100 * time.Millisecond)
        }
    }()
}

原子操作与内存模型

Go 的 atomic 遵循 happens-before 语义：

  • atomic Store → 后续对其他 goroutine 的 Load 可见
  • atomic Load → 看到之前的 Store
  • atomic 操作构成同步点：Store 释放、Load 获取
  • 不使用 atomic 的共享变量访问是 data race
// 使用 atomic 实现无锁通知
var ready atomic.Bool

// goroutine 1
go func() {
    time.Sleep(time.Second)
    ready.Store(true)  // 设置标志
}()

// goroutine 2
for !ready.Load() {
    runtime.Gosched()  // 出让 CPU
}
fmt.Println("检测到 ready")

08 并发模式

Go 社区积累了大量实用的并发编程模式，这些模式将 channel、select、goroutine 组合成可复用的结构。

Worker Pool（工作者池）

func workerPool(numWorkers int, jobs <-chan int, results chan<- int) {
    var wg sync.WaitGroup
    for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for job := range jobs {
                results <- job * 2  // 模拟处理
            }
        }(i)
    }
    wg.Wait()
    close(results)
}

func main() {
    jobs := make(chan int, 100)
    results := make(chan int, 100)

    go workerPool(10, jobs, results)

    for i := 0; i < 50; i++ {
        jobs <- i
    }
    close(jobs)

    for r := range results {
        fmt.Println(r)
    }
}

Fan-Out / Fan-In

Fan-Out：将输入分发给多个 goroutine 并行处理。Fan-In：合并多个 channel 到同一个输出 channel。

// Fan-Out：分发到多个 goroutine
func fanOut(input <-chan int, workers int) []<-chan int {
    channels := make([]<-chan int, workers)
    for i := 0; i < workers; i++ {
        ch := make(chan int)
        channels[i] = ch
        go func(out chan<- int) {
            for v := range input {
                out <- v * 2
            }
            close(out)
        }(ch)
    }
    return channels
}

// Fan-In：合并多个 channel
func fanIn(channels ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c {
                out <- v
            }
        }(ch)
    }
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}

func main() {
    input := make(chan int)
    go func() {
        for i := 0; i < 20; i++ {
            input <- i
        }
        close(input)
    }()

    channels := fanOut(input, 4)
    merged := fanIn(channels...)

    for v := range merged {
        fmt.Println(v)
    }
}

流水线（Pipeline）

流水线模式将一个处理阶段拆分为多个步骤，每个步骤由一个 goroutine 执行，通过 channel 连接前后阶段：

// 阶段 1: 生成数字
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out)
    }()
    return out
}

// 阶段 2: 平方
func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * n
        }
        close(out)
    }()
    return out
}

// 阶段 3: 乘以 2
func multiply(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * 2
        }
        close(out)
    }()
    return out
}

func main() {
    // 流水线：generate → square → multiply
    for v := range multiply(square(generate(1, 2, 3, 4, 5))) {
        fmt.Println(v)  // 输出: 2, 8, 18, 32, 50
    }
}

扇出（Fan-Out Stage）

// 在流水线中，如果一个阶段性能不够，可以扇出多个相同阶段的 goroutine：
func fanOutStage(in <-chan int, workers int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for n := range in {
                // 模拟耗时计算
                out <- n * n
            }
        }()
    }
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}

超时控制模式

// Or-Channel：只要任意一个 channel 完成就返回
func or(channels ...<-chan struct{}) <-chan struct{} {
    switch len(channels) {
    case 0:
        return nil
    case 1:
        return channels[0]
    }

    done := make(chan struct{})
    go func() {
        defer close(done)
        select {
        case <-channels[0]:
        case <-channels[1]:
        case <-or(channels[2:]...):  // 递归
        }
    }()
    return done
}

上下文传递模式

// 通过 context 传递取消信号和超时
func withTimeout(ctx context.Context, duration time.Duration) {
    ctx, cancel := context.WithTimeout(ctx, duration)
    defer cancel()

    result := make(chan string, 1)
    go func() {
        result <- expensiveOperation()
    }()

    select {
    case v := <-result:
        fmt.Println("结果:", v)
    case <-ctx.Done():
        fmt.Println("操作超时或被取消")
    }
}
并发编程总结:
Go 的并发原语设计精简但功能强大。理解 goroutine 的轻量特性、channel 的同步语义以及 select 的多路复用，配合 sync 包的工具，可以构建出安全、高效、可维护的并发程序。核心原则：通过通信来共享内存，而不是通过共享内存来通信。
